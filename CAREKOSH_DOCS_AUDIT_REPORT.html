Bottom line

No deployed backend/mobile code issue was found during this documentation audit, and the macOS dev environment is structurally compatible with Apple Silicon and Intel. However, the docs are not yet fully final/push-ready as a documentation set: two current guides would mislead a developer if followed literally (docs/NEW_DEVELOPER_QUICKSTART.md and vitaltrack-backend/API_TESTING_GUIDE.md), and the setup/env cluster still contains stale vitaltrack.app and Windows-only assumptions. Treat this report as the exact fix plan before declaring docs fully synced.

§2 Prioritized action list

Do first — misleading to a developer today

  • high NEW_DEVELOPER_QUICKSTART.md — the prescribed Docker flow uses REQUIRE_EMAIL_VERIFICATION=true (from docker-compose.dev.yml), so "you're taken straight to the dashboard" is false; also the dev entrypoint is Uvicorn (Dockerfile.dev), not Gunicorn. Fix the doc or flip the compose flag to false.
  • high vitaltrack-backend/API_TESTING_GUIDE.md — current curl examples contain stale staging/prod base URLs, wrong item filter query names, stock update bodies missing version, stale item stats JSON, and an incomplete order-status flow. Fix before using it as the QA/operator API script source.
  • med setup-local-dev.sh / .bat — stop writing MAIL_FROM=noreply@vitaltrack.app (→ carekosh.com); the generated FRONTEND_URL points at the Metro port :8081 instead of the backend :8000/api/v1/auth.
  • med docker-compose.dev.yml — hardcodes a Windows-only LAN IP 172.25.32.1 in CORS_ORIGINS + FRONTEND_URL; breaks for a macOS dev. Use localhost or a documented placeholder.
  • med vitaltrack-mobile/.env.example — production example still says api.vitaltrack.app; should be api.carekosh.com (+ staging).
  • med CAREKOSH_E2E_VERIFICATION_GUIDE.html — several wrong response bodies/fields developers would copy verbatim (see §3).

Then — accuracy polish (one pass across the themed docs)

  • Apply the /live vs /health, order-status enum, audit_log naming, and profile-UI corrections everywhere they appear.
  • Re-frame LOCAL_TESTING_INTERNALS.md and local_testing_field_manual.html for macOS.
  • Correct vitaltrack-backend/README.md and vitaltrack-backend/DOCKER_GUIDE.md so Docker dev commands consistently use docker compose -f docker-compose.dev.yml ..., and so the automatic migration entrypoint is described accurately.
  • Complete vitaltrack-backend/.env.example with the email/verification keys; add optional structured env_file loading to docker-compose.yml (or document that it's ignored).
  • Update CAREKOSH_ROADMAP.md to mark the custom domain + email as shipped (PRs #49/#50/#51) — only the hosted privacy policy remains open.

Optional — historical records (safe to leave)

  • The historical docs are mostly era-correct. Only fix one if you want it to double as current reference. Exception worth doing anyway: remove the non-existent MAIL_FROM_NAME row from PROJECT_LEARNINGS_AND_JOURNEY.md so nobody configures a setting that doesn't exist.

§6 macOS dev-environment readiness

Verdict: macOS-compatible ✓

A new developer on a MacBook (Apple Silicon arm64 and Intel) can set the project up end to end. The shell scripts use macOS-correct primitives (ipconfig getifaddr en0/en1, an OSTYPE==darwin* branch, no GNU-only sed -i/readlink/realpath), and both Docker images (python:3.12-slim, postgres:16-alpine) are multi-arch with no linux/amd64 pin — they build natively on arm64. The fixes below are stale values and consistency gaps, not blockers.
