B

Play Console one-time setup (all policy forms)

In Play Console: Create app → name CareKosh, type App, Free, accept declarations. Then complete every section below — Google will not let you release until all required ones are green.

B1 · Privacy policy URL required · not hosted

  • Host a publicly resolvable policy (GitHub Pages, public Notion page, any static host).
  • It must cover what CareKosh stores/transmits: name, email, username, optional supplier/contact phone, inventory, orders, categories, activity, selected photos, generated PDFs, clipboard export, auth tokens (SecureStore only), and processors (Render, Neon, Brevo, Google Play).
  • Enter it in App content → Privacy policy and link it from the in-app About screen.

B2 · Account deletion required · web URL not hosted

  • In-app path already ships: Profile → Delete Account → email confirmation; server deletion completes only after the user confirms by email (DELETE /auth/me, PR #13).
  • Google also requires a public web “request account deletion” URL. Host it with the privacy policy and enter it under App content → Data deletion. Missing this = instant rejection for new apps.

B3 · Data safety form required · not submitted

  • Fill it from the full inventory below (reproduced so you don't have to flip docs).
  • Declare data encrypted in transit (preview & production are HTTPS-only; dev HTTP is never shipped) and that users can request account deletion.
  • Before submitting, re-verify against the deployed backend schema + current SDK list. Also declare the email-verification, password-reset, and account-deletion token columns + expiry timestamps as PII storage touchpoints (Goal 9 re-audit).

B4 · Content rating required

Complete the questionnaire. Category: Medical / health-adjacent utility. No violence, sexual content, gambling, ads, or user-to-user communication/UGC.

B5 · Target audience & content required

Target audience 13+, not directed to children (so the Families policy does not apply).

B6 · App access — reviewer sign-in required · use real creds

CareKosh is login-gated, so Google reviewers need a working account. Provide a real reviewer email + password (not a placeholder) plus the cold-start note — use the template below. Create it as a dedicated production smoke account (Phase G) and rotate the password after review.

H

Closed testing — the mandatory 12 / 14-day gate

Applies to you

The publisher is a personal developer account created after Nov 13, 2023. (Organization accounts, or personal accounts created before that date, are exempt — if that's you, skip to Phase J.)

The rule (verified with Google Play support docs)

  • Run a Closed test (not internal) with ≥ 12 testers who opt in and stay opted in for 14 continuous days.
  • The 12-tester minimum was reduced from 20 on Dec 11, 2024; the 14-day duration is unchanged.
  • Only after meeting this can you apply for production access.

How to run it

  1. Play Console → Testing → Closed testing → create a track (e.g. “Alpha”) and an email list of testers.
  2. Recruit ≥ 12 real testers. Each must accept the opt-in link, install, and remain opted in for the full 14 days.
  3. Promote your build to the closed track (promote the internal release, or eas submit then promote in Console).
  4. Keep the build live and installed across the 14 continuous days — don't remove testers mid-window or the clock can reset.
  5. Encourage real usage, collect feedback, ship fixes (each eas build auto-bumps the version code).
Tip

Line up your 12 testers before you start the window so day 1 actually has 12 opted-in installs. The clock is unforgiving.

L · Rollback & incident response

Roll back the backend if

/live fails > 2 consecutive external checks after deploy · /health fails and Neon/Render logs show a deploy-caused DB readiness failure · smoke-account login or inventory list fails due to server errors.

Do NOT roll back for

A known free-tier cold start that recovers · a single credential typo · a Play review/opt-in delay.

Backend rollback

  1. Render service → Events → select last known-good deploy → Rollback.
  2. Render disables auto-deploys on rollback (prevents the bad commit redeploying) — keep off until the fix lands.
  3. Verify /live, /health, smoke login, inventory list.
  4. Render rollback does NOT roll back Neon data. If a data mutation caused it, use the backup/restore runbook in LAUNCH_READINESS_RUNBOOK_GOAL_10.md and get explicit approval before any production restore.

Mobile bad-release recovery

There is no true Play rollback for already-installed users:

  1. Pause/halt the staged rollout in Play Console.
  2. Build a fixed AAB with a higher version code (eas build auto-increments).
  3. Upload to internal → smoke on a real device → promote only after app access, Data Safety, login, inventory, and cold-start are verified.
  4. Keep the backend compatible with the bad version while users may still have it installed.
