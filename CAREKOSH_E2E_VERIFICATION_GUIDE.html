End-to-End Verification Guide

Every curl, every expected response, every gate. Run this after a deploy; if a step fails, stop and fix — do not proceed.

CareKosh · Home ICU Inventory · Staging + Production · April 2026

Environments under test

Staging

vitaltrack-api-staging.onrender.com
Neon branch: vitaltrack_staging · Test data only · Safe to wipe

Production

vitaltrack-api.onrender.com
Neon branch: neondb · Real user data · Never wipe
Ground rules

1. Run the full flow on staging first. 2. Only hit production with read-only + idempotent requests unless you are intentionally cleaning up your own test account. 3. Do not create real-looking data on production.

Phase 1 Pre-flight

Step 1

Both services healthy

curl https://vitaltrack-api-staging.onrender.com/health
curl https://vitaltrack-api.onrender.com/health
Expected {"status":"healthy","environment":"staging"} (or "production")
Cold start

Free Render tier sleeps after 15 min idle. The first request takes 30–60 s to wake the container; subsequent calls are fast. If you get a 502 then a 200, that's cold start — not a bug.

Step 2

Latest deploy is live

  1. Open dashboard.render.com
  2. Click vitaltrack-api-staging → Events tab → top entry should say "Deploy live"
  3. Repeat for vitaltrack-api (production)
  4. Check commit hash matches the last merge on main
Step 3

Alembic head matches code

In Render shell for each service:

alembic current
Expected Latest revision id, matching the filename of the newest file in vitaltrack-backend/alembic/versions/. As of PR #13: 20260419_add_account_deletion_token_fields.

Phase 2 Auth flow — register through refresh

Step 4

Register a test user

BASE=https://vitaltrack-api-staging.onrender.com

curl -X POST $BASE/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "E2E Test",
    "email": "e2e+'$(date +%s)'@example.com",
    "username": "e2etest",
    "password": "TestPass123!"
  }'
Expected (201) {"access_token":"...","refresh_token":"...","token_type":"bearer","expires_in":1800,"user":{...}}
PR #12

email is required (was optional pre-PR-#12). Registering without it returns 422.

Step 5

Verify-email gate (if REQUIRE_EMAIL_VERIFICATION=true)

Without clicking the verification link in the email first, login should fail:

curl -X POST $BASE/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"identifier":"e2e+...@example.com","password":"TestPass123!"}'
Expected (403) {"detail":"Email not verified"}

Click the verification link in the email sandbox (Mailtrap dev, Brevo staging/prod), then retry login — it should now succeed.

Step 6

Login + extract token

LOGIN=$(curl -s -X POST $BASE/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"identifier":"e2e+...@example.com","password":"TestPass123!"}')
ACCESS=$(echo $LOGIN | jq -r '.access_token')
REFRESH=$(echo $LOGIN | jq -r '.refresh_token')
echo "ACCESS: ${ACCESS:0:20}..."
echo "REFRESH: ${REFRESH:0:20}..."
Step 7

Refresh with rotation

curl -X POST $BASE/api/v1/auth/refresh \
  -H "Content-Type: application/json" \
  -d "{\"refresh_token\":\"$REFRESH\"}"
Expected New access + refresh tokens. Old $REFRESH is now revoked.

Prove the rotation worked — try the old refresh again:

curl -X POST $BASE/api/v1/auth/refresh \
  -H "Content-Type: application/json" \
  -d "{\"refresh_token\":\"$REFRESH\"}"
Expected (401) {"detail":"Invalid refresh token"}

Phase 3 CRUD + OCC

Step 8

Create category → create item

AUTH="Authorization: Bearer $ACCESS"

CAT=$(curl -s -X POST $BASE/api/v1/categories \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d '{"name":"Oxygen","icon":"o2","color":"#1d6b52"}' | jq -r '.id')

ITEM=$(curl -s -X POST $BASE/api/v1/items \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d "{
    \"category_id\":\"$CAT\",
    \"name\":\"Oxygen Mask (Adult)\",
    \"quantity\":10,
    \"minimum_quantity\":5,
    \"unit\":\"pieces\",
    \"is_critical\":true
  }" | jq -r '.id')

echo "CAT=$CAT ITEM=$ITEM"
Step 9

Read back, confirm version=1

curl -s $BASE/api/v1/items/$ITEM -H "$AUTH" | jq '{name,quantity,version}'
Expected {"name":"Oxygen Mask (Adult)","quantity":10,"version":1}
Step 10

Update with correct version — version increments

curl -s -X PUT $BASE/api/v1/items/$ITEM \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d '{"name":"Oxygen Mask (Adult)","quantity":12,"minimum_quantity":5,"version":1}' \
  | jq '{name,quantity,version}'
Expected {"name":"Oxygen Mask (Adult)","quantity":12,"version":2}
Step 11

Update with stale version — 409 Conflict (OCC)

curl -s -w "\nHTTP %{http_code}\n" -X PUT $BASE/api/v1/items/$ITEM \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d '{"name":"X","quantity":99,"minimum_quantity":5,"version":1}'
Expected (409) {"detail":"Version conflict","server_version":2,"server_quantity":12}
Why this matters

This is the single reason the server-first migration was worth doing. Two caregivers editing the same item at the same second no longer silently overwrite each other.

Step 12

Quick stock update via PATCH

curl -s -X PATCH $BASE/api/v1/items/$ITEM/stock \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d '{"quantity":8,"version":2}' | jq '{quantity,version}'
Expected {"quantity":8,"version":3}

Phase 4 Orders + apply-stock

Step 13

Create order, mark received, apply to stock

# Create order
ORDER=$(curl -s -X POST $BASE/api/v1/orders \
  -H "$AUTH" -H "Content-Type: application/json" \
  -d "{\"title\":\"Restock\",\"items\":[{\"item_id\":\"$ITEM\",\"quantity\":20}]}" \
  | jq -r '.id')

# Transition: pending → ordered → received
curl -s -X PATCH $BASE/api/v1/orders/$ORDER/status \
  -H "$AUTH" -H "Content-Type: application/json" -d '{"status":"ordered"}' > /dev/null
curl -s -X PATCH $BASE/api/v1/orders/$ORDER/status \
  -H "$AUTH" -H "Content-Type: application/json" -d '{"status":"received"}' > /dev/null

# Apply to stock
curl -s -X POST $BASE/api/v1/orders/$ORDER/apply -H "$AUTH" | jq '.status'

# Verify item quantity increased by 20
curl -s $BASE/api/v1/items/$ITEM -H "$AUTH" | jq '{quantity,version}'
Expected Order status → "stock_updated". Item quantity was 8, now 28. Version increments on every update.
Step 14

Activity log records every write

curl -s "$BASE/api/v1/activities?limit=10" -H "$AUTH" | jq '.[] | {action,item_name,created_at}'
Expected Entries for: category create, item create, item update (×N), order create, order status × 2, apply-stock. Each with created_at.

Phase 5 Account deletion (PR #13)

Step 15

Request deletion — email sent

curl -s -X DELETE $BASE/api/v1/auth/me -H "$AUTH"
Expected {"message":"Deletion confirmation email sent"}. In the mail sandbox, a confirmation email with a link to GET /api/v1/auth/confirm-delete/{token} (token is raw; DB stores the SHA-256 hash with a 24 h TTL).
Step 16

Cancel pending deletion

curl -s -X POST $BASE/api/v1/auth/cancel-delete -H "$AUTH"
Expected {"message":"Account deletion cancelled"}. Trying to use the email link after cancel should return an HTML error (hash cleared).
Step 17

Confirm deletion (testing only — irreversible)

Request deletion again (Step 15), open the email, click the link:

curl -s https://vitaltrack-api-staging.onrender.com/api/v1/auth/confirm-delete/RAW_TOKEN
Expected HTML success page. User row is deleted; cascade unwinds categories, items, orders, order_items, activity_logs, refresh_tokens, audit_logs.

Prove cascade worked — any authenticated call with a saved access token should now 401:

curl -s -w "HTTP %{http_code}\n" $BASE/api/v1/items -H "$AUTH"
Expected HTTP 401. Access token decodes but the user no longer exists.

Phase 6 Database-side checks

Step 18

Row counts + version spot check

Connect to Neon via Neon Console SQL editor (or psql with the connection string) on the vitaltrack_staging branch:

SELECT count(*) FROM users;
SELECT count(*) FROM items;
SELECT count(*) FROM activity_logs;

-- OCC evidence
SELECT id, name, quantity, version FROM items ORDER BY updated_at DESC LIMIT 5;

-- Audit log
SELECT action, entity_type, created_at
FROM audit_logs
ORDER BY created_at DESC
LIMIT 10;
Expected Item rows have version >= 1. audit_logs has entries with JSONB before_snapshot / after_snapshot for every recent write.
Step 19

Cascade check (post-deletion)

-- After Step 17, should all return 0
SELECT count(*) FROM categories WHERE user_id = 'DELETED_USER_ID';
SELECT count(*) FROM items      WHERE user_id = 'DELETED_USER_ID';
SELECT count(*) FROM orders     WHERE user_id = 'DELETED_USER_ID';
SELECT count(*) FROM refresh_tokens WHERE user_id = 'DELETED_USER_ID';

Phase 8 Loading UX — skeletons, cache persistence, cold-start retry (PRs #15, #16)

Step 22

Skeleton loading verification

  • Clear app data (adb shell pm clear com.vitaltrack.mobile) or first install → login
  • Dashboard tab → animated skeleton placeholders visible during initial fetch
  • Inventory tab → skeleton placeholders (list-shaped)
  • Orders tab → skeleton placeholders (card-shaped)
  • After data loads, skeletons replaced by real content (header/SafeAreaView stays stable — no layout jump)
Step 23

Cache persistence verification

  • Open app with logged-in session, browse dashboard/inventory/orders (populates cache)
  • Kill app from recents; wait 1 minute; re-open → cached data visible instantly, then refreshes within 30 s (staleTime)
  • Toggle airplane mode on; re-open app → cached data still visible (read-only snapshot)
  • Logout → login as a different user → previous user's inventory must NOT appear (cache cleared on both sides for shared-device privacy)
  • Optional — flip ENABLE_CACHE_PERSISTENCE to false in providers/QueryProvider.tsx, restart Metro → app behaves as before PR #16 (no persistence, no instant load)
Step 24

Cold-start auto-retry verification

  • Wait 15+ minutes with no staging activity so Render sleeps the service
  • Open app, go to login, enter valid credentials, tap Sign In
  • "CareKosh server is waking up..." card appears with spinner + "Attempt X of 12" + Cancel button
  • After ~4 attempts, copy changes to "Still connecting... This can take up to 60 seconds after inactivity."
  • When server wakes, auto-login completes and dashboard loads — no manual re-tap required
  • Repeat with a deliberately wrong password → immediate "Incorrect password or account not found" error; no retry loop (auth errors must never trigger retry)
  • Tap Cancel during a retry → interval stops, form re-enables, state resets cleanly
