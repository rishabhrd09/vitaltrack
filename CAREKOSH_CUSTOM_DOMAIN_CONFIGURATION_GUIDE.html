Quick Verdict

The custom-domain migration is in the right shape. The infrastructure side now points both public API hostnames at the existing Render services, and the branch updates the build-time/mobile/ops references so future preview APKs and production AABs use the CareKosh-owned domains instead of direct onrender.com hostnames.

What Was Done

1. Domain ownership

The domain carekosh.com was registered and managed in Cloudflare. There is no platform-specific domain requirement. Portability comes from owning the domain, controlling DNS, and having the app call your domain rather than a hosting-provider URL.

2. DNS records

Cloudflare DNS now routes API traffic through two CNAME records:

  • api.carekosh.com -> vitaltrack-api.onrender.com
  • staging-api.carekosh.com -> vitaltrack-api-staging.onrender.com

The records were kept DNS only while Render issued certificates.

3. Render custom domains

Render custom domains were added to the two existing web services:

  • Production service vitaltrack-api: api.carekosh.com
  • Staging service vitaltrack-api-staging: staging-api.carekosh.com

Render verified DNS and issued TLS certificates for these domains.

4. Render environment variables

The backend email-link base URL was changed through Render environment variables:

  • Production FRONTEND_URL: https://api.carekosh.com/api/v1/auth
  • Staging FRONTEND_URL: https://staging-api.carekosh.com/api/v1/auth

The value should be the URL only. Do not type FRONTEND_URL=https://... inside the value field.

Architecture Picture

How Mobile Knows Which Backend To Call

The mobile app does not ask Render which backend to use at runtime. The backend URL is injected during development or EAS build through EXPO_PUBLIC_API_URL. Expo inlines that value into the JS bundle, and the app's API client reads it.

File Role Domain-specific behavior
vitaltrack-mobile/eas.json Defines EAS build profiles. preview bakes https://staging-api.carekosh.com. production bakes https://api.carekosh.com.
vitaltrack-mobile/app.config.js Expo config and release guard. Throws if a preview build is pointed at anything except staging, or if a production build is pointed at anything except production.
vitaltrack-mobile/services/api.ts Shared mobile API client. Reads process.env.EXPO_PUBLIC_API_URL into API_BASE_URL, then appends /api/v1 for API routes.
vitaltrack-mobile/package.json Local Metro start helpers. npm run start:staging and npm run start:prod now point local development sessions at the custom domains.

How Backend Knows Its Public URL

The backend does not need to know which mobile app called it for normal API requests. It responds to whatever HTTPS host reaches the Render service. The important backend-side public URL is FRONTEND_URL, because CareKosh sends verification, password reset, and account-deletion emails with links back into backend-hosted auth pages.

File or dashboard Role Domain-specific behavior
Render service environment Runtime config source for deployed backend. Stores FRONTEND_URL, DATABASE_URL, ENVIRONMENT, SECRET_KEY, email credentials, and other environment-specific values.
vitaltrack-backend/app/core/config.py Loads and validates backend settings. Requires FRONTEND_URL in production and parses CORS_ORIGINS. It does not hardcode the domain.
vitaltrack-backend/app/utils/email.py Builds verification and password reset emails. Uses {FRONTEND_URL}/verify-email?token=... and {FRONTEND_URL}/reset-password?token=....
vitaltrack-backend/app/api/v1/auth.py Builds account deletion confirmation emails. Uses {FRONTEND_URL}/confirm-delete/{token}.
vitaltrack-backend/render.yaml Production Render blueprint/IaC reference. Now documents production FRONTEND_URL as https://api.carekosh.com/api/v1/auth.

CI/CD and Secrets Flow

GitHub Actions does not store Render environment variable values in code. It uses named secrets when it needs to talk to external services. The deployed backend itself gets runtime values from Render service settings.

Secret or key name Where it is set Used by Purpose
RENDER_DEPLOY_HOOK GitHub -> repository -> Settings -> Secrets and variables -> Actions. Because deploy-backend uses the production environment, it may also be stored under the GitHub production environment secrets. .github/workflows/ci.yml, deploy-backend job Private Render URL that triggers a production backend redeploy after merge to main.
EXPO_TOKEN GitHub Actions secrets. build-preview and disabled build-production jobs Allows GitHub Actions to ask EAS Cloud to build APK/AAB artifacts.
DATABASE_URL Render service environment variables. FastAPI app, Alembic migration entrypoint, health check Points each backend service at its own Neon database. Value must never be printed.
SECRET_KEY Render service environment variables. FastAPI auth/JWT code JWT signing secret. Rotation logs users out because existing tokens no longer validate.
MAIL_PASSWORD, MAIL_FROM Render service environment variables. Email sender Transactional email credentials and sender identity. Provider response bodies must stay out of logs.
Play service-account JSON Local/EAS credential management, not committed to the repo. eas submit if/when automated Used to submit Android builds to Play Console when that flow is enabled.
Render auto-deploy vs deploy hook: Render can deploy from its GitHub integration when the connected branch changes. The CI RENDER_DEPLOY_HOOK call is a backup/explicit trigger. Neither changes which backend the mobile app calls; mobile routing comes from EXPO_PUBLIC_API_URL inside the built artifact.

Release sequence after this branch merges

  1. Merge the domain-alignment PR only after CI is green.
  2. Build a new preview APK from the merged code.
  3. Install the preview APK and test staging register, login, email verification, forgot password, inventory, orders, and PDF/bill paths.
  4. After preview is clean, build a new production AAB. That AAB will bake in https://api.carekosh.com.
  5. Upload the AAB to Play internal testing first, then continue through closed/production tracks as required by the account.
  6. Keep Render subdomains enabled during transition. They are useful fallback and are explicitly treated as production by the smoke-script guard.
Do not skip the rebuild. DNS changes help future portability, but installed mobile binaries do not dynamically follow new API URLs unless they were already built to call the custom domain. The production Play Store artifact must be built after these config changes land.
