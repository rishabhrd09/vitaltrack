02 The Big Picture: What CareKosh Is

Before we dissect optimistic locking, JWT rotation, and connection-pool tuning, you need the aerial photograph: what this system is for, the shape of its runtime, and where every chapter that follows lives on the map. Read this once and the rest of the guide becomes navigable.

In one line CareKosh (codename VitalTrack) is an async FastAPI inventory API for home-ICU / medical supplies — a stateless app fronted by Gunicorn+Uvicorn on Render, talking to Postgres on Neon, serving a React Native mobile client.

The product in plain terms

CareKosh is the backend for a medical / home-ICU inventory app. The user is typically a caregiver managing critical supplies for a patient at home — oxygen consumables, dressings, medications, feeding-tube kits. The mobile app (React Native / Expo) lets them organize supplies into Categories, track Items with live stock levels and low-stock thresholds, raise Orders (a purchase workflow that moves through a status lifecycle and ultimately credits received quantities back into inventory), and review an Activity feed of everything that happened. The whole API is what powers those screens.

Two product realities shape almost every engineering decision you will see:

📵

Offline-first

The mobile client works without a network and syncs later, so every entity carries a client-generated local_id and writes are concurrency-safe by design.

🩺

Safety-critical data

Running out of a supply is a real-world harm, so stock mutations are atomic, every change is audited, and responses never cache.

Runtime topology

Here is the deployed system end to end. Trace the path of one request through it, then we'll narrate each hop.

  1. Mobile → HTTPS — the Expo app makes TLS requests to the public Render URL. Native HTTP clients are not bound by CORS, which is exactly why dev can run with a wildcard origin (more in the security chapter).
  2. Render edge → Gunicorn — Render terminates TLS and forwards to the container. The process manager is Gunicorn, configured with --worker-class uvicorn.workers.UvicornWorker --workers 2 (Dockerfile:80-82). Gunicorn owns the master process and supervises workers.
  3. Uvicorn (2 workers) → FastAPI — each worker is an async ASGI event loop running one copy of our app. Two workers means two independent event loops and two independent DB connection pools.
  4. FastAPI app → Neon Postgres — handlers run on the loop and await non-blocking SQL over asyncpg against the managed Postgres instance. TLS to the database is required and enforced in non-dev environments.
  5. Attached services — Brevo sends transactional email (verification, password reset); Sentry captures exceptions and a 10% trace sample. Both are optional and configured purely through environment variables.
  6. Probes — Render (and the container's own HEALTHCHECK) poll /live and /health to decide whether the instance is alive and ready (Dockerfile:73-74, render.yaml:20).
🔬 Under the Hood

Why Gunicorn and Uvicorn? Uvicorn is the ASGI server that actually speaks HTTP and drives the asyncio event loop. Gunicorn is a battle-tested process manager: it forks workers, restarts crashed ones, and handles graceful reloads on SIGTERM. Running Uvicorn as a Gunicorn worker class gives you Uvicorn's async performance with Gunicorn's supervision and multi-process fan-out — a standard production pattern. Two workers on a small Render plan is a deliberate concurrency-vs-memory trade-off.

Stateless app, attached backing services

The architecture is textbook Twelve-Factor. The application process holds no durable state of its own: no local files, no in-memory sessions, no sticky data. Everything stateful lives in an attached backing service reached purely by a URL from the environment — Postgres for data, Brevo for mail, Sentry for telemetry. Configuration arrives the same way, parsed once into a typed, validated settings object (Chapter on config). This is what makes the app horizontally scalable and disposable: any of the two workers — or a freshly booted replacement after a crash — is interchangeable, because the truth lives in the database, not the process.

💡 Interview Insight

When an interviewer asks "how would you scale this?", the answer is unlocked by statelessness. Because no request depends on process-local memory, you scale by adding workers/instances behind the load balancer; the only shared contention point is the database connection budget. The one honest caveat worth volunteering: slowapi's in-memory rate-limit fallback is per-process, so with two workers the effective limit can double — a real consequence of statelessness done with an in-memory store.

How the routes hang together

Routing is a composition hierarchy. Each feature module owns a small router with its own prefix and tag; those are mounted onto one versioned aggregate, which the app then includes wholesale.

Follow one request

03 End-to-End Request Lifecycle

We trace a single authenticated request — POST /api/v1/items with a bearer token and a JSON body — from the raw TCP socket all the way back to JSON on the wire. This is the spine of the whole guide: almost every other chapter is a zoom-in on one stage you meet here.

In one line One client request becomes one coroutine on the event loop that climbs the middleware onion, resolves dependencies (DB session + current user), runs the handler — which commits its writes explicitly before returning — then serializes the response model, while the get_db dependency makes the request atomic (rollback on any error); every await yields control back to the loop.

The journey, stage by stage

FastAPI is a thin, ergonomic layer over Starlette (the ASGI toolkit) and Pydantic (validation). The app object built by create_app() is, underneath, a callable implementing the ASGI contract async def __call__(scope, receive, send). Everything below is what happens between Uvicorn invoking that callable and the bytes leaving the socket.

  1. Uvicorn accepts the socket & builds the ASGI scope — Uvicorn (the ASGI server) accepts the TCP connection, parses the raw HTTP/1.1 bytes, and constructs an ASGI scope dict: type="http", method, path, headers, query string, client address. It then calls app(scope, receive, send) on the running event loop. receive is an awaitable that streams the request body in chunks; send pushes response events back out.
  2. The middleware onion (inbound) — the request climbs the wrapped layers. The crucial gotcha: add_middleware prepends, so registration order is the reverse of inbound execution order. Registered as CORS, then add_request_id, then add_security_headers app/main.py:106-149, the inbound order is security-headers → request-id → CORS → router. Each layer calls await call_next(request) to descend inward.
  3. Routing & path-operation match — Starlette's router matches POST /api/v1/items against the merged route table (the /api/v1 aggregate router plus the health routes) and selects the matching path operation function.
  4. Dependency resolution — FastAPI walks the dependency graph. get_db opens an AsyncSession and yields it app/core/database.py:106-123; get_current_user pulls the bearer credential, verifies the JWT, and loads the row with noload("*") app/api/deps.py:58-60.
  5. Pydantic validates the body — the JSON body is parsed into the handler's request schema. A validation failure never reaches your code: it raises RequestValidationError, caught by the global handler and reshaped into a structured {field, message, code} 422 app/main.py:173-195.
  6. The handler coroutine runs — the async path operation executes on the event loop, await-ing non-blocking asyncpg I/O over a pooled connection. Each await is a suspension point where the loop is free to service other requests. Like nearly every write endpoint here, the handler calls await db.commit() explicitly before returning — so it can db.refresh() and serialize the persisted row (e.g. register commits at app/api/v1/auth.py:157).
  7. Response serialization — when the handler returns, FastAPI serializes the result through the route's response_model (Pydantic), filtering and coercing fields to JSON-native types.
  8. Dependency teardown — the safety net — get_db's post-yield code runs: await session.commit() (a fallback — usually a no-op, because the handler already committed) and finally: await session.close() returns the connection to the pool app/core/database.py:118-123. Its real guarantee is the except: await session.rollback() branch — any exception rolls the whole request back.
  9. Headers stamped on the way out — the response descends the onion outward: CORS adds Access-Control-* headers, the request-id wrapper echoes X-Request-ID, and the security wrapper stamps nosniff, X-Frame-Options: DENY, Referrer-Policy, Cache-Control: no-store, and (prod only) HSTS app/main.py:142-148.
  10. JSON returned — Uvicorn serializes the ASGI send events back into HTTP bytes and writes them to the socket. The coroutine is done; the connection may be kept alive for the next request.
🔬 Under the Hood

ASGI = scope + receive + send. ASGI replaces WSGI's single synchronous (environ, start_response) call with three async primitives. scope is an immutable dict describing the connection (its type is "http", "websocket", or "lifespan"). receive is an awaitable you call to pull inbound events (body chunks, disconnects); send is an awaitable you call to push outbound events (http.response.start with status + headers, then http.response.body). Middleware is just a callable that wraps another ASGI app and can intercept any of these. Because every primitive is a coroutine, a single OS thread multiplexes thousands of concurrent connections — the loop parks a request at each await and runs another.

The middleware onion, concretely

The two custom middlewares are function middlewares (Starlette's BaseHTTPMiddleware), registered with the @app.middleware("http") decorator. They follow the chain-of-responsibility shape: do something, call inward, then mutate the response on the way back out.

app/main.pyPython
@app.middleware("http")
async def add_request_id(request: Request, call_next):
    request_id = request.headers.get("X-Request-ID") or uuid4().hex
    response = await call_next(request)   # <-- descend into inner layers + handler
    response.headers["X-Request-ID"] = request_id
    return response                        # <-- response now flows back outward

Notice the structure: code before await call_next runs on the way in; code after it runs on the way out, once the inner layers and the handler have produced the final response. That is why the request-id and security-header wrappers can stamp headers on any response — including error responses produced deep inside. It also explains why CORS is registered as a class-based middleware: it must short-circuit OPTIONS preflight requests before they reach a handler, so it sits inboard of the response-mutating wrappers.

The transaction boundary — when exactly does it commit?

get_db is an async-generator dependency, which FastAPI treats as setup/teardown around the handler. This is the Unit-of-Work pattern with the transaction scoped to the request:

app/core/database.pyPython
async with AsyncSessionLocal() as session:
    try:
        yield session            # <-- handler runs here, with the session
        await session.commit()   # <-- runs AFTER the handler returns
    except Exception:
        await session.rollback() # <-- any raise -> nothing is persisted
        raise
    finally:
        await session.close()    # <-- always returns the connection to the pool
💡 Interview Insight

Where is the transaction boundary? The get_db yield-dependency wraps each request: it rolls back on any exception and commits on success. In practice almost every write handler also calls await db.commit() explicitly before returning (so it can refresh() and return fresh data), which makes the dependency-level commit a redundant safety net. The guarantee that actually matters is the rollback: a handler that raises (including an IntegrityError bubbling to the global 409 handler) never partially persists, because the except branch rolls back first. So don't claim the only commit is at teardown — the handler usually commits itself; the dependency owns rollback-atomicity at the request boundary.

⚠️ Pitfall

Because handlers keep using ORM objects after they commit (they db.refresh() then serialize the row into the response model), the session factory sets expire_on_commit=False app/core/database.py:97. By default SQLAlchemy expires every ORM attribute on commit, so the next attribute read triggers a lazy reload — and in async SQLAlchemy an implicit lazy load raises MissingGreenlet. Keeping attributes un-expired makes post-commit access safe — the kind of subtle async detail interviewers love to probe.

📝 Note

One coroutine, many yield points

The whole chain — middleware, dependency resolution, handler, commit, serialization — executes as a single coroutine scheduled on the event loop. There is no thread per request. Every await (the DB round-trip in get_current_user, the inserts in the handler, the commit) is a suspension point: the loop parks this request and runs another while Postgres is busy. That is the entire reason an async stack can hold thousands of concurrent connections on a handful of OS threads.

✅ Pro Tip

The corollary an interviewer wants to hear: one synchronous blocking call poisons the whole loop. A naive time.sleep(), a sync requests.get(), or a CPU-bound hot loop inside a handler blocks the single thread and stalls every other in-flight request — not just the one that called it. In async FastAPI, "await everything I/O-bound, offload everything CPU-bound" is the load-bearing rule.

05 Async, the Event Loop & ASGI vs WSGI

This codebase is built on a single, deceptively simple promise: one Python worker can serve hundreds of concurrent requests because waiting on I/O is free. Understanding exactly why that is true — and the one thing that breaks it — is the difference between a candidate who has "used FastAPI" and one who can reason about a production system under load.

In one line FastAPI runs your async def handlers on a single-threaded event loop; an await on I/O yields the loop to other requests, but a blocking call (like a slow Argon2 hash) freezes every request on that worker — which is why this code pushes hashing onto a thread with anyio.to_thread.run_sync(...).

Concurrency from first principles

A coroutine is a function you can pause and resume. When you write async def, calling it does not run the body — it returns a coroutine object. The event loop is the scheduler that actually drives coroutines forward: it runs one until it hits an await on something not-yet-ready (a socket with no bytes, a DB query in flight), at which point the coroutine yields control back to the loop, which picks another ready coroutine to advance. This is cooperative multitasking: tasks voluntarily hand back the CPU at await points. Nothing preempts them.

The payoff is concurrency without threads. While one request waits 30 ms for Postgres to answer, the loop is not idle — it advances 50 other requests. A single OS thread, doing nothing during I/O, multiplexes thousands of in-flight operations. That is the entire reason an async web server scales the way it does.

🔬 Under the Hood

The event loop is essentially a while True loop wrapped around the OS epoll/kqueue syscall. It registers every pending socket with the kernel, then sleeps until the kernel reports "these file descriptors are ready." For each ready FD it resumes the coroutine that was awaiting it, runs it until the next await, and loops. "Async I/O" is not magic parallelism — it is one thread reacting to kernel readiness notifications, fast.

async vs threads vs processes — and the GIL

Why not just use threads? Because CPython has a Global Interpreter Lock: only one thread executes Python bytecode at a time. Threads help when work is I/O-bound (a blocked thread releases the GIL so another can run), but they pay context-switch and memory costs, and you get no true parallelism for Python-level CPU work. The async model sidesteps thread overhead entirely for I/O: one thread, no locks around shared state between tasks (a coroutine only ever pauses at explicit await points, so there are no surprise interleavings mid-statement). For CPU-bound work you still need real parallelism — which means processes (multiple workers, each with its own GIL) or, pragmatically, threads for code that releases the GIL in C.

ASGI vs WSGI — and why FastAPI needs ASGI

WSGI (Web Server Gateway Interface) is the old Python standard: a synchronous callable app(environ, start_response) that handles exactly one request, top to bottom, on the calling thread. It has no concept of await, no streaming lifecycle, and no WebSockets — it cannot express a coroutine that pauses mid-request. Flask and Django classically ran on WSGI servers (Gunicorn, uWSGI), achieving concurrency only by spawning more threads/processes.

ASGI (Asynchronous Server Gateway Interface) is the async successor. The app is an async callable: async def app(scope, receive, send). The scope describes the connection (an http request, a websocket, or a lifespan event); receive/send are awaitable channels for streaming messages in and out. Because the app is a coroutine, it can await the database and yield the loop — and because the protocol models persistent connections, it supports WebSockets and long-lived streams that WSGI structurally cannot. FastAPI is a Starlette subclass implementing exactly this interface; Uvicorn is the ASGI server that parses raw HTTP off the socket into a scope and calls your app on the loop. In this project, that callable is the module-level app object Uvicorn imports as app.main:app app/main.py:322, launched via uvicorn.run("app.main:app", ...) app/main.py:331-332.

The one pitfall that matters: blocking the event loop

Here is the trap that separates correct async code from a production incident. The loop is single-threaded. If an async def handler calls a function that runs CPU work or blocking I/O without awaiting — a tight loop, a synchronous HTTP client, a slow cryptographic hash — it never yields. The loop is stuck inside that one call. Every other in-flight request on that worker is frozen until it returns, because there is no other thread to run them. One slow operation in one request degrades latency for all concurrent requests.

⚠️ Pitfall

Calling a blocking library inside an async def route doesn't just slow that request — it stalls the entire worker. A single time.sleep(5), requests.get(...), or unthreaded password hash inside a coroutine makes your p99 latency for every concurrent caller spike by the same duration. Async failures are rarely loud; they show up as mysterious tail-latency under load.

The exact mitigation in this codebase

Password hashing is the canonical offender. This app uses Argon2 (an intentionally memory-hard, CPU-bound KDF tuned to take tens of milliseconds — that slowness is the security property). Running it directly in a handler would pin the loop for the whole hash. So every call site wraps the synchronous hash_password / verify_password in anyio.to_thread.run_sync(...), which dispatches the blocking function to a worker thread and awaits the result — freeing the loop to serve other requests while the hash runs.

💡 Interview Insight

"Why hash in a thread, but query the database on the loop?" is the question that proves you actually understand async. The answer: Argon2 is CPU-bound — it burns the CPU with no I/O wait, so there is no natural await point; left on the loop it blocks everything, so you offload it to a thread. A Postgres query is I/O-bound — the work happens in the database, and the async driver (asyncpg) yields the loop the instant the request is sent on the wire, resuming when the answer arrives. Threading the DB call would add overhead for zero benefit; threading the hash is mandatory. The discriminator is CPU-bound vs I/O-bound, not "slow vs fast."

🔬 Under the Hood — why a thread, not a process, for Argon2

The textbook rule, straight from Python's own asyncio docs, is CPU-bound → process pool, blocking I/O → thread pool — precisely because the GIL lets only one thread execute Python bytecode at a time, so threads give no real parallelism for Python-level CPU work. Argon2 looks like it should therefore need a process. It doesn't: argon2-cffi (and bcrypt) do the heavy computation inside a C extension that releases the GIL while hashing. So a worker thread genuinely runs the hash in parallel with the event loop — with none of a process pool's pickling, IPC, or cold-start cost. That is the exact reason the right tool here is anyio.to_thread.run_sync and not a ProcessPoolExecutor. If you're tempted to say "CPU-bound always means processes," this GIL-releasing C extension is the senior-level exception to name.

The two escapes from blocking

When you must run blocking code, you have two sanctioned escapes, and FastAPI gives you both:

  1. Offload to a thread yourself — anyio.to_thread.run_sync(fn, *args) (or the lower-level loop.run_in_executor). You stay in an async def handler and explicitly punt the blocking call to AnyIO's worker-thread pool. This is exactly what the auth routes do. AnyIO caps the pool (40 threads by default), giving natural backpressure rather than unbounded thread spawning.
  2. Write a plain def route — if you declare a path operation as def (not async def), FastAPI detects it and runs the entire handler in the threadpool automatically, so blocking inside it never touches the loop. The catch: a def route can't await the async DB session, so you'd lose the async ORM. The auth routes are async def precisely so they can await asyncpg while still threading just the hash.
✅ Pro Tip

Reach for a sync def route when the whole handler is blocking (e.g. a legacy sync SDK with no async equivalent). Reach for anyio.to_thread.run_sync when the handler is mostly async (awaiting the DB) and only one call is blocking — you want to keep the async benefits and thread out the single hot spot. This codebase is a textbook case of the second.

💡 Interview Insight — the hidden concurrency ceiling

"You have 200 concurrent requests hitting a sync def handler that does blocking work — what happens?" Strong answer: at most 40 run at once. FastAPI/Starlette offload sync handlers and sync dependencies to AnyIO's worker-thread pool, whose default CapacityLimiter is 40 tokens, shared process-wide; request 41 queues behind it. Crucially that 40-thread ceiling is independent of your SQLAlchemy pool_size — they are two separate bottlenecks that compose, and whichever is smaller bites first. Naming both ceilings — and that the thread pool is shared with sync dependencies — is exactly the kind of detail that signals you've run async in production.

Async drivers vs sync drivers — the silent trap

An async stack is only async end-to-end if every layer cooperates. This project pairs SQLAlchemy 2.0's async engine with asyncpg, the native async Postgres driver, configured at app/core/database.py:84-92. asyncpg performs non-blocking socket I/O, so await db.execute(...) genuinely yields the loop while Postgres works. The classic mistake is reaching for the synchronous driver (psycopg2) or calling a synchronous ORM method inside an async def handler — that call blocks the socket read on the loop's thread and silently re-introduces the very stall you were avoiding, with none of the obvious symptoms.

⚠️ Pitfall

Mixing a sync ORM/driver into async code is the most insidious blocking bug, because nothing crashes — it just doesn't scale. A handler using psycopg2 under an event loop will appear correct in a single-user test and collapse under concurrency, since each query blocks the loop for its full round-trip. If you're on the async engine, the entire data path (driver, session, queries) must be awaited; this is why the session factory is async_sessionmaker(class_=AsyncSession, ...) and the dependency yields an AsyncSession at app/core/database.py:94-100.

📝 Note

The async session factory sets expire_on_commit=False app/core/database.py:97 for a related async-specific reason: with expiry on, accessing an ORM attribute after commit triggers a lazy reload — implicit I/O that async SQLAlchemy forbids and that would raise during response serialization. It is a small but telling sign of how "everything must be awaited" reshapes ordinary ORM habits in an async world.

Putting it together: one worker, many requests

Trace a burst of logins hitting a single Uvicorn worker. Uvicorn parses each request into an ASGI scope and schedules the login coroutine on the loop. Each coroutine reaches await db.execute(...) — asyncpg sends the query and yields, so the loop immediately advances the next pending login. As Postgres answers come back, the kernel marks those sockets ready and the loop resumes each coroutine to the next step: the Argon2 verify. That verify is dispatched to the thread pool via anyio.to_thread.run_sync, so even the CPU-bound check doesn't monopolize the loop. The net effect is that a single thread keeps dozens of logins in flight at once — and not one of them can wedge the others, because the only CPU-heavy step has been moved off the loop's critical path. That is the whole design, in one request.

  • Path params — names that appear in the route template, like item_id in @router.get("/{item_id}") at app/api/v1/items.py:212-218. They are taken straight from the URL.
  • Query params — scalar-typed params with defaults (or wrapped in Query(...)), like page and search above. The ge=1, le=100 and max_length=100 constraints are enforced before your code runs.
  • Body params — params typed as a Pydantic BaseModel, like data: ItemCreate in app/api/v1/items.py:251-252. FastAPI reads the request body, parses JSON, and validates it against the model.
  • Dependencies — anything resolved via Depends(), here hidden behind the DB and CurrentUser aliases (more on those next).

Two more declarative knobs do real work. status_code=status.HTTP_201_CREATED on the create handler (app/api/v1/items.py:245-249) sets the success status without you ever touching a Response object. And response_model — ItemList on the list endpoint, ItemResponse on the single-item endpoint (app/api/v1/items.py:212-214) — declares the output contract, which is what powers serialization, filtering, and the OpenAPI schema.

🔬 Under the Hood

FastAPI calls inspect.signature() on every handler at startup, walks the parameters, and builds a Dependant object — a tree describing how to extract and validate each argument. At request time it executes that tree, not your raw function directly. The classification (path vs query vs body) is decided once at registration, so the per-request cost is just running the pre-built extractors. That is why the type hints must be accurate: they are compiled into the request contract.

2 · Dependency Injection — Depends() and the resolution graph

Dependency injection is FastAPI's most distinctive feature. A dependency is just a callable; Depends(callable) tells FastAPI "call this, validate its own parameters recursively, and pass the result in." CareKosh defines its building block — the per-request DB session — as a yield-dependency:

app/core/database.pyPython
async def get_db():
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()

This is the setup/teardown pattern. Everything before yield runs before the handler; the handler receives the yielded session; everything after yield runs once the handler returns or raises. That makes get_db a unit-of-work boundary: commit on success, rollback on any exception, always close. (The full transaction story lives in the persistence chapter — here the point is purely how FastAPI drives the generator.)

The Annotated type-alias pattern

Repeating db: AsyncSession = Depends(get_db) in forty handlers is noise. CareKosh collapses each dependency into a reusable type alias at the bottom of app/api/deps.py:173-179:

app/api/deps.pyPython
CurrentUser = Annotated[User, Depends(get_current_user)]
ActiveUser = Annotated[User, Depends(get_current_active_user)]
VerifiedUser = Annotated[User, Depends(get_current_verified_user)]
SuperUser = Annotated[User, Depends(get_current_superuser)]
OptionalUser = Annotated[Optional[User], Depends(get_optional_user)]
DB = Annotated[AsyncSession, Depends(get_db)]
Pagination = Annotated[PaginationParams, Depends()]

Annotated[T, Depends(...)] bundles the runtime type (T, used by your editor and by serialization) with the injection metadata (the Depends) in a single name. A handler then reads as plain English — db: DB, current_user: CurrentUser — and FastAPI unwraps the alias to find the dependency. It is DRY, it keeps type information for IDEs and type-checkers, and it centralizes the wiring: change how get_current_user works and every endpoint inherits it. Note Pagination = Annotated[PaginationParams, Depends()] at app/api/deps.py:179 — with no argument, Depends() uses the annotated class itself as the callable, instantiating PaginationParams (app/api/deps.py:156-167) from the request's page/page_size query params.

The auth dependency chain (sub-dependencies)

Dependencies can depend on other dependencies, forming a graph FastAPI resolves bottom-up. The verified-user check does not re-authenticate — it depends on get_current_user and only adds one rule on top:

async def get_current_verified_user(

So when a route asks for VerifiedUser, FastAPI resolves a chain: security (the HTTPBearer(auto_error=False) at app/api/deps.py:22) extracts the bearer token → get_db opens a session → get_current_user (app/api/deps.py:28-76) decodes the JWT, loads the user with noload("*") to suppress relationship fan-out (app/api/deps.py:58-59), and rejects inactive accounts with 403 → get_current_verified_user (app/api/deps.py:88-102) adds the verified check. get_current_superuser (app/api/deps.py:105-119) follows the same shape. Each layer is a small, testable, composable unit — the chain-of-responsibility pattern expressed as function signatures.

💡 Interview Insight

When an interviewer asks "what does Depends() actually return, and when is it cached?", they want two facts. (1) It returns the callable's return value — for Depends(get_current_user), the resolved User; for a yield-dependency, the yielded value. (2) Results are cached per request by default. Because get_db appears in both get_current_user and directly in the handler (via DB), FastAPI calls it once and shares the same AsyncSession across the whole request — which is exactly why the auth lookup and the handler's queries land in the same transaction. The cache key is the callable; pass use_cache=False to force re-execution.

  1. Build the graph — at startup FastAPI flattens the handler's Annotated[..., Depends(...)] params into a dependency tree, deduplicating shared sub-dependencies like get_db.
  2. Resolve leaves first — per request it executes from the leaves up: token extraction, then get_db, then get_current_user, then any wrapper like get_current_verified_user.
  3. Cache within the request — each callable runs at most once; later nodes reuse the cached value, so one DB session is shared.
  4. Inject & run — resolved values are passed as named arguments and the handler executes.
  5. Tear down in reverse — after the handler returns, yield-dependencies run their post-yield code (commit/rollback/close) in reverse resolution order.

The custom 422 handler

When request validation fails, FastAPI raises RequestValidationError. CareKosh intercepts it to return a cleaner, structured body instead of FastAPI's default error format:

            "field": ".".join(str(loc) for loc in error["loc"]),

This reshapes Pydantic's raw error list into a flat {field, message, code} array (app/main.py:173-195) — far friendlier for a mobile client to render inline next to the offending form field. 💡 Interview Insight

"Why two classes — why not reuse one model for request and response?" Because they have opposite trust and shape requirements. The request model is a gate: it rejects bad input, sanitizes strings, and forces version to be present. Sharing one model would mean a field that is optional-on-create becomes optional-on-read, secrets on the ORM object could leak, and you'd lose the 201-with-different-shape pattern. Separation is a security and contract decision, not boilerplate.

Settings done right

07 Configuration & the 12-Factor App

Configuration is where most production incidents quietly begin: a forgotten secret, a database URL that doesn't parse, a wildcard CORS rule shipped to prod. This chapter shows how CareKosh turns a pile of environment variables into a typed, validated, fail-fast object — and why that one design choice maps cleanly onto Factor III of the Twelve-Factor App.

In one line A single Settings(BaseSettings) class parses the environment once at startup, coerces every value to its declared type, runs validators that reject a weak prod secret or a malformed DB URL, hides secrets behind SecretStr, and is cached as a process-wide singleton via @lru_cache.

The shape of the configuration object

Everything lives in one class, app/core/config.py:14. Because it subclasses BaseSettings from pydantic-settings, each declared field is automatically sourced from an environment variable of the same name (and from a .env file in development), then coerced and validated against its Python type annotation. A string "8000" becomes an int; "true" becomes a bool. If coercion fails, construction raises — the app refuses to boot.

app/core/config.pyPython
class Settings(BaseSettings):

        env_file=".env",
        env_file_encoding="utf-8",

        extra="ignore",

    APP_NAME: str = "CareKosh API"
    DEBUG: bool = False
    ENVIRONMENT: str = "development"  # development, staging, production
    PORT: int = 8000

Three flags in model_config carry real weight. case_sensitive=False means database_url and DATABASE_URL both bind — convenient across shells and platforms. extra="ignore" means an unrecognised variable in the environment (say, a Render-injected PORT-adjacent var you don't model) is silently dropped instead of crashing startup. And env_file=".env" loads a local file in dev while production simply doesn't ship one — the environment is the config.

Factor III: config in the environment

The Twelve-Factor App's third factor draws a hard line between code and config: anything that varies between deploys (DB credentials, the signing secret, allowed origins) belongs in the environment, never in the codebase. The litmus test the factor proposes: could you open-source the repo this minute without leaking a single credential? Here, yes — every secret has a placeholder default and is read from the environment at runtime.

🔬 Under the Hood

The payoff of Factor III is one artifact, many environments. The same Docker image runs in dev, staging, and prod; only the injected env vars differ. CareKosh proved this concretely when it migrated hosting from Railway to Render — the platform changed, the database moved, but not a single line of Python was touched. The new DATABASE_URL and secrets were set in Render's dashboard, the image was redeployed, and the app booted. Config-as-environment is what makes that a five-minute operation instead of a code change.

Field validators: turning env strings into safe, correct values

Annotations handle type correctness; @field_validator functions handle semantic correctness. CareKosh defines four, and each one encodes a lesson learned in production.

1. Rewriting the database driver — ensure_asyncpg_driver

Managed Postgres providers hand you a URL in their format, not yours. Render gives postgresql://…; Neon sometimes gives postgres://… and appends ?sslmode=require&channel_binding=require. SQLAlchemy's async engine needs the postgresql+asyncpg:// scheme, and asyncpg chokes on sslmode as a URL parameter. This validator — declared mode="before" so it runs on the raw string prior to type validation — normalises all of that at app/core/config.py:43-69.

app/core/config.pyPython
@field_validator("DATABASE_URL", mode="before")
@classmethod
def ensure_asyncpg_driver(cls, v):
    if isinstance(v, str):
        # asyncpg can't read ?sslmode=… as a URL param — strip it.
        # SSL is set via connect_args in database.py instead.
        if "?" in v:
            v = v.split("?")[0]
        if v.startswith("postgresql://") and "+asyncpg" not in v:
            v = v.replace("postgresql://", "postgresql+asyncpg://", 1)
        elif v.startswith("postgres://"):
            v = v.replace("postgres://", "postgresql+asyncpg://", 1)
    return v
📝 Note

Because the ?sslmode=require query is stripped here, TLS can't be conveyed through the URL — so database.py re-adds it through connect_args={"ssl": True} only outside dev/test. The two pieces are deliberately coupled: this validator removes the param, the engine restores the behaviour the right way.

2 & 3. Fail-closed production gates

Two validators enforce invariants that only matter in production, reading the already-validated ENVIRONMENT from info.data. reject_weak_secret_in_production (app/core/config.py:76-88) refuses to boot if the shipped placeholder secret survives into prod, or if any secret is shorter than 32 characters. require_frontend_url_in_production (app/core/config.py:133-139) demands a real frontend URL for email links.

The @lru_cache singleton

Parsing the environment, reading .env, and running every validator is real work — you don't want it repeated each time a request resolves Depends(get_settings). Wrapping the factory in @lru_cache (app/core/config.py:169-172) memoises the result: the first call constructs and validates Settings; every subsequent call returns the same cached instance.

💡 Interview Insight

When an interviewer asks "how do you manage config in a 12-factor app?" they're probing whether you understand the trade-offs, not just the library. Why validate at startup instead of first use?

The deepest design decision here is when validation runs. Because settings = get_settings() executes at module import — before Uvicorn ever binds the socket — a misconfigured deploy crashes immediately and loudly, not three hours later when the first user hits the one endpoint that reads MAIL_PASSWORD.

🔬 Under the Hood

This is the fail-fast principle. A bad DATABASE_URL or a placeholder secret should never become a latent fault that surfaces under production traffic. Validating at boot means a broken config is caught by the deploy's health check, the rollout halts, and the previous good version keeps serving. Lazy "validate on first use" turns a deploy-time error into a runtime error with a much larger blast radius — and a far worse on-call story.

Talking to Postgres

08 The Database Layer: Async SQLAlchemy 2.0

Every request to CareKosh ultimately touches Postgres, and it does so over a single, carefully tuned async engine. This chapter dissects that layer end to end: how the connection pool is sized and self-heals, why a server-side statement_timeout is the keystone of resilience, how get_db turns each request into one atomic transaction, and the SQLAlchemy 2.0 idioms — Mapped, select(), noload — that keep an async ORM from blowing up in your face.

In one line One async engine with a 15-connection ceiling, an 8s server-side statement timeout, pre-ping recovery, and a yield-dependency that commits on success / rolls back on error — wrapping every request in a clean unit of work.

The async engine: asyncpg under the hood

The whole layer is built on create_async_engine driving the asyncpg driver. Unlike the classic synchronous psycopg2, asyncpg speaks the Postgres wire protocol on top of the asyncio event loop, so a coroutine that is waiting on I/O — a query result coming back over the socket — yields control instead of blocking the thread. That non-blocking property is the entire reason a single FastAPI worker can hold thousands of in-flight requests; the engine is the bridge between the event loop and the database.

📝 Three things people conflate — engine, sessionmaker, session

Keep the lifetimes straight and most beginner confusion evaporates. The engine is long-lived application infrastructure — built once at import, it owns the connection pool and lives for the whole process; it knows how to talk to Postgres but it is not a session. The async_sessionmaker (AsyncSessionLocal) is a factory — the stamp that mints sessions; it isn't a session either. An AsyncSession is one request's unit of work: open it, query / add / commit / refresh, close it. The engine is shared by the whole app; a fresh session is minted per request by get_db. In one line: the engine is not a session, and the sessionmaker is not a session — it's the thing that makes them.

app/core/database.pyPython
# Cap every statement at 8s so one hung query can't pin a pooled connection
# indefinitely and cascade into pool exhaustion (RESIL-1).
_server_settings = {"statement_timeout": "8000"}

# SSL required for Neon (staging + production) but not for local Docker or CI
if settings.ENVIRONMENT not in ("development", "testing"):
    _connect_args = {"ssl": True, "server_settings": _server_settings}
else:
    _connect_args = {"server_settings": _server_settings}

engine = create_async_engine(
    settings.DATABASE_URL,
    pool_size=settings.DATABASE_POOL_SIZE,
    max_overflow=settings.DATABASE_MAX_OVERFLOW,
    pool_timeout=settings.DATABASE_POOL_TIMEOUT,
    pool_pre_ping=True,  # Enable connection health check
    echo=settings.DEBUG,  # Log SQL statements in debug mode
    connect_args=_connect_args,
)

Notice the SSL toggle at app/core/database.py:79-82. Managed Postgres (Neon) mandates TLS, but local Docker and CI do not — and forcing it there would break the dev loop. SSL is passed through connect_args rather than baked into the URL on purpose: the config validator strips Neon's ?sslmode=require query parameter because asyncpg rejects sslmode as a URL param, so the TLS intent has to be re-expressed here as a driver argument.

The connection pool, demystified

Opening a Postgres connection is expensive — a TCP handshake, a TLS handshake, authentication, and session setup. A pool amortizes that cost by keeping a set of live connections open and lending them out. The engine here exposes three knobs that together define its concurrency envelope.

SettingValue (default)What it does
pool_size5Persistent connections kept open and reused for the life of the process.
max_overflow10Burst connections opened on demand above pool_size, then closed when returned. Effective ceiling = 5 + 10 = 15 concurrent connections.
pool_timeout30sHow long a coroutine waits for a free connection once all 15 are checked out before raising TimeoutError. This is backpressure, not an unbounded queue.
pool_pre_pingTrueIssues a cheap liveness check (effectively SELECT 1) when a connection is checked out, transparently discarding and replacing dead ones.
🔬 Under the Hood

The real maximum concurrent connections is 15, not 5. pool_size is the steady-state floor; max_overflow is the elastic ceiling. When demand exceeds 15, the 16th coroutine doesn't fail immediately — it waits up to pool_timeout for a connection to be returned, which is exactly the backpressure you want under load rather than spawning unbounded connections and crushing Postgres' own max_connections limit.

pool_pre_ping: defeating stale connections

A pooled connection is a long-lived socket, and long-lived sockets die silently. Neon (and any PgBouncer-style proxy) closes idle connections after a timeout; a network blip or a database restart can sever a connection while the pool still believes it's healthy. The next coroutine to borrow that connection would hit the dreaded "server closed the connection unexpectedly."

pool_pre_ping=True at app/core/database.py:89 closes that gap. On every checkout, the pool runs a tiny round-trip; if it fails, the connection is discarded and a fresh one is opened, all invisibly to your handler. The cost is one extra trivial query per checkout — a trade nearly always worth making against managed/proxied databases.

statement_timeout: the keystone of resilience

The single most interview-worthy line in this file is the one that looks the most innocuous: _server_settings = {"statement_timeout": "8000"} at app/core/database.py:76. It tells Postgres, at the session level, to force-abort any single SQL statement that runs longer than 8000ms. The comment labels it RESIL-1, and the reasoning is a small but vicious failure-cascade.

  1. A query hangs — a missing index, a lock wait, or a pathological plan makes one statement run effectively forever.
  2. It pins its connection — that connection is checked out of the pool and cannot be returned while the query is in flight.
  3. Repeat under load — a few such queries consume all 15 connections. Now every other request blocks waiting on the pool.
  4. Pool exhaustion → outage — after pool_timeout (30s) the waiters start raising TimeoutError; the service is effectively down even though the database itself is up.

The 8s server-enforced timeout breaks the chain at step 1: Postgres aborts the runaway statement, the connection is released back to the pool, and the blast radius is contained to the one offending request. Because it is enforced by the server, it works even if the application thread is wedged — a defensive guarantee a client-side timeout can't match.

💡 Interview Insight

When an interviewer asks "what's your maximum number of DB connections and what happens under load," they're probing whether you understand pool exhaustion as a cascading failure. The senior answer connects three facts: the ceiling is 15, a hung query pins a connection indefinitely, and a server-side statement_timeout is what prevents one slow query from deadlocking the whole pool. Naming that causal chain is what separates a staff-level answer from "we set a timeout somewhere."

The session factory: configured for async correctness

Sessions are minted from a single async_sessionmaker. Three flags here are deliberate, and one of them is critical specifically because this is an async app.

expire_on_commit=False
Keep ORM attributes loaded and usable after commit (the async landmine — see below).
autoflush=False
Pending changes are not implicitly flushed before queries; the service layer controls exactly when SQL is emitted.
autocommit=False
The session runs inside an explicit transaction; nothing persists until commit() is called.
⚠️ Pitfall

By default SQLAlchemy sets expire_on_commit=True: after commit() it marks every loaded attribute as expired, so the next attribute access triggers a fresh SELECT to reload. In a sync app that's just a lazy reload. In an async app it's a time bomb. get_db commits at the end of the request, and then FastAPI serializes the returned ORM object into the response. If those attributes are expired, that serialization triggers implicit I/O outside an awaited context, and SQLAlchemy raises MissingGreenlet (greenlet_spawn has not been called). Setting expire_on_commit=False keeps the already-loaded values valid post-commit, so serialization reads from memory and never surprises you with I/O. One nuance worth naming: expire_on_commit=False fixes only this trigger — the post-commit expired-column reload. A second, independent MissingGreenlet trigger remains: lazily touching an unloaded relationship (say user.items), which no commit flag fixes — that one is solved only by eager loading (next section). Conflating the two is a common mistake; a strong answer separates them.

⚠️ Pitfall — one AsyncSession, one task

An AsyncSession is a single in-progress transaction on a single connection: a mutable, stateful object with no internal locking. The SQLAlchemy docs say it outright — "a single instance of AsyncSession is not safe for use in multiple, concurrent tasks." The classic trap is fanning out queries with asyncio.gather(...) that all share the same db session — interleaved access corrupts its state. The mnemonic: "Session per thread (sync), AsyncSession per task (async)." CareKosh sidesteps this because every request gets its own session from get_db and handlers await their queries sequentially — but the moment you parallelize DB work, mint a fresh session per task.

get_db: a transaction per request

The data layer is injected into handlers through a single yield-based dependency. FastAPI treats a generator dependency as setup/teardown: the code before yield runs before the handler, the handler receives the yielded value, and the code after yield runs after the handler returns (or raises). That structure is precisely what you need to implement the Unit of Work pattern — one transaction that spans the whole request.

app/core/database.pyPython
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()

Read the control flow carefully — every line is load-bearing (app/core/database.py:115-123):

  • yield session hands the live session to the handler. Everything the handler does — reads, mutations, even an audit-log row added by a service helper — accumulates in this one session.
  • await session.commit() runs only if the handler returned cleanly. This is the transaction boundary: the business mutation and any audit row are flushed together, atomically.
  • except → rollback() → raise means any exception discards all uncommitted work and re-raises. A handler that throws never partially commits. The IntegrityError that later surfaces as a global 409 has already been rolled back here.
  • finally → close() always returns the connection to the pool, success or failure — this is what keeps the 15-connection ceiling honest.
🔬 Under the Hood

This is the textbook Unit of Work: changes are tracked in the session and committed as one atomic unit at the request boundary. The benefit is correctness for free — a handler that mutates three tables and then validation-fails on the fourth leaves the database untouched. It also means your service-layer code never sprinkles commit() calls around; it just mutates objects and trusts the boundary. The audit service leans on exactly this: it calls db.add(entry) but deliberately does not commit, so the audit record rides the same transaction as the change it describes.

There is a twin, get_db_context at app/core/database.py:126-143, with identical commit/rollback/close logic but exposed as a plain @asynccontextmanager. It exists for code that runs outside a request — the readiness probe and background jobs — where FastAPI's Depends machinery isn't available to drive the generator.

Handlers never call get_db directly. It is wrapped in an Annotated type alias so injection reads as a clean type hint:

The async lazy-loading hazard

The relationship above declares lazy="selectin", and the auth query above passes noload("*"). These two choices are both answers to the same async-specific danger.

SQLAlchemy keeps an identity map per session and, by default, lazy-loads relationships: accessing user.items for the first time fires a fresh query. In a sync app, fine. In an async app, that implicit query happens during plain attribute access — outside an await — and SQLAlchemy raises MissingGreenlet because it can't suspend to do I/O. So every relationship the model declares uses lazy="selectin", which eagerly batches all children with a single WHERE fk IN (...) query up front, killing the classic N+1 problem and making the loaded object graph fully serializable after the request commits.

But eager loading on every authenticated request is wasteful when the auth dependency only needs scalar fields like is_active. That's why get_current_user overrides the model default with noload("*") at app/api/deps.py:58-60 — it suppresses the five selectin relationships entirely (the DB-1 optimization), so logging in doesn't drag in every category, item, and order the user owns.

✅ Pro Tip

The trio lazy="selectin" + expire_on_commit=False + Pydantic from_attributes=True is the combination that makes "return the ORM object straight from the handler" safe in async. Selectin pre-loads the graph, no-expire keeps it valid past commit, and from_attributes serializes it without touching the DB again. Drop any one and you risk a MissingGreenlet during response serialization.

Deterministic constraint names for Alembic

The full request → DB journey

Putting it together, here is the path a single mutating request takes through this layer:

  1. Inject the session — FastAPI resolves db: DB, opening one AsyncSession from AsyncSessionLocal and yielding it (autoflush=False, expire_on_commit=False).
  2. Query & mutate — the handler builds select() statements, await db.execute(...)s them over a pooled asyncpg connection, and mutates ORM objects. Each statement is capped server-side at 8s.
  3. Return — control unwinds into get_db's post-yield block.
  4. Commit or roll back — clean return → commit() flushes everything atomically; any exception → rollback() + re-raise.
  5. Serialize — because attributes didn't expire and relationships were selectin-loaded, the ORM object serializes to JSON with no further I/O.
  6. Close — finally: close() returns the connection to the pool.
🛡️ Security / OWASP

The async ORM is also a defense. Building queries with select(User).where(User.id == user_id) uses parameter binding, so user_id is sent to Postgres as a bound value, never string-concatenated into SQL — structurally closing off SQL injection (OWASP A03). The same applies to every where() and filter in the app: as long as values flow through SQLAlchemy expressions rather than raw f-strings, injection isn't merely unlikely, it's not possible by construction.

💡 Interview Insight

When an interviewer sees lazy="selectin" they are really probing three things at once: do you understand the N+1 problem, do you know the trade-offs between lazy strategies, and do you know why async makes this non-optional? With the default lazy="select", touching user.items fires one query per parent (N+1). lazy="joined" avoids that but, with five collections on User, a single join produces a cartesian row explosion. selectin issues exactly one extra query per relationship via WHERE items.user_id IN (...) — O(1) extra queries, no row multiplication.

🔬 Under the Hood

The async angle is the kicker. SQLAlchemy's asyncio support forbids implicit IO during attribute access — a classic lazy load would need to await with no greenlet to suspend into, raising MissingGreenlet. selectin eager-loads the collection up front inside the awaited query, so later access (during Pydantic serialization) reads already-materialized data. That is why lazy="selectin" + expire_on_commit=False + from_attributes form the trio that makes ORM-object-to-response serialization safe after commit.

Standout decision #1 — the denormalized order snapshot

This is the most interview-worthy decision in the schema. An OrderItem does not hold a foreign key to items. Its item_id is an indexed String(36) — a soft reference — and the row then duplicates name, brand, unit, quantity, current_stock, and minimum_stock as a frozen copy.

💡 Interview Insight

"Why denormalize here when you spent the rest of the schema normalizing?" The answer they want is the distinction between a reference (always reflects current truth) and a snapshot (reflects truth at a moment). Orders, invoices, and audit rows are snapshots by nature. Bonus points for noting the soft item_id gives you no referential-integrity guarantee — orphaned references are possible by design.

Standout decision #2 — optimistic concurrency via version

Item carries an integer version with both a Python default=1 and a DDL server_default="1" app/models/item.py:113-118. The protocol is enforced by the schemas: ItemUpdate.version and StockUpdate.version are required ints that "must match server's current version" app/schemas/item.py:91.

app/models/item.pyPython

    Integer, default=1, server_default="1", nullable=False,

The client reads an item (gets version=N); any update must echo version=N. The service compares the supplied value to the row's current version and rejects a mismatch with 409 Conflict, incrementing on success. This is a compare-and-swap that prevents the lost-update anomaly — critical for an offline-first, multi-device app where two phones can edit the same item.

🔬 Under the Hood

This is application-managed optimistic locking, deliberately not SQLAlchemy's built-in __mapper_args__ = {"version_id_col": version}. The built-in auto-appends WHERE version = :old to every UPDATE and raises StaleDataError on a zero-row match — which surfaces as a generic 500 unless caught. Managing the check in the service layer gives full control over the HTTP status, the error body, and coexistence with custom sync/merge logic. As for the dual default: default=1 is applied by the ORM at insert, while server_default="1" writes DEFAULT '1' into the DDL so even raw-SQL or backfilled rows start at 1 — the comparison can never trip over a NULL version.

How the audit row stays atomic

The audit service deliberately db.add()s the entry but never commits app/services/audit.py:8-27.

The request's session is opened by the get_db dependency, which commits once at the end or rolls back on any exception. So the business mutation and its audit row land in a single transaction: either both persist or neither does.

⚠️ Pitfall

If log_audit committed internally you would split the work across two transactions and open three failure modes: (1) the audit commits, then the mutation rolls back — a phantom audit for a change that never happened; (2) the mutation commits but a later step fails — a real change with no audit (a silent gap); (3) the inner commit prematurely ends the unit of work and fights the outer commit. Atomic audit logging is the only way the ledger can be trusted to mirror persisted state exactly.

10 Authentication & Authorization

This is the deepest chapter in the guide — and the one interviewers mine hardest. We walk the full credential lifecycle of the CareKosh backend: how passwords are hashed with a memory-hard KDF off the event loop, how JWTs are minted and verified, how refresh tokens rotate atomically to defeat replay, how the dependency-injection ladder enforces authorization, and how email-driven flows resist enumeration and timing attacks. Almost every line maps onto an OWASP recommendation; we'll name them as we go.

In one line Passwords use Argon2 (offloaded to a worker thread), access tokens are short-lived and stateless, refresh tokens are tracked by jti and rotated with an atomic UPDATE…RETURNING compare-and-swap so a stolen token can be used at most once.

1 · Password security: Argon2, with a migration story

Credentials never touch the database in plaintext. Hashing is configured once, in a passlib CryptContext that lists two schemes in priority order:

app/core/security.pyPython
pwd_context = CryptContext(
    schemes=["argon2", "bcrypt"],
    deprecated="auto",
)

def hash_password(password: str) -> str:
    return pwd_context.hash(password)        # always hashes with argon2 (first scheme)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)

The first scheme, Argon2, is OWASP's #1 recommended password algorithm and the winner of the 2015 Password Hashing Competition. Its defining property is memory-hardness: each hash deliberately consumes a large block of RAM, so an attacker can't cheaply parallelize cracking across thousands of GPU/ASIC cores the way they can with a CPU-only function. bcrypt is only CPU-hard; Argon2 raises the cost on both axes. app/core/security.py:21-34

Two configuration details are worth saying out loud in an interview. First, the salt has no column. Argon2 emits a self-describing PHC string — $argon2id$v=19$m=…,t=…,p=…$salt$hash — that embeds the algorithm, version, memory/time/parallelism cost parameters, a per-hash random salt, and the digest. verify_password reads everything it needs out of that one string, so there is no separate salt field on the User model. Second, deprecated="auto" marks every scheme except the default — the first entry in the schemes list, here Argon2 — as deprecated, which means the bcrypt fallback is the deprecated one. passlib can still verify a legacy bcrypt hash, but it flags it via needs_update(), enabling a rehash-on-login migration: when a bcrypt user logs in, you re-hash their password with Argon2 transparently and the fleet drifts to the stronger algorithm over time.

🛡️ Security / OWASP

This maps to OWASP ASVS V2.4 (Credential Storage). The bearer secrets here — passwords — must use a deliberately slow, memory-hard KDF. Argon2id is the first-choice recommendation; bcrypt is the acceptable fallback. The two-scheme context gives you both compliance today and a clean upgrade path.

2 · Don't block the event loop — offload hashing

Argon2 is intentionally slow: tens to hundreds of milliseconds per call. FastAPI route handlers are coroutines multiplexed onto a single-threaded asyncio event loop, and a synchronous CPU-bound call inside a coroutine has no await point — it blocks the entire loop until it returns. Every other in-flight request (other logins, DB I/O completions, health checks) freezes behind it. The fix appears at every hashing site:

app/api/v1/auth.pyPython
# register: hash off the event loop while building the User
hashed_password=await anyio.to_thread.run_sync(hash_password, data.password),

# login: verify off the event loop
if not user or not await anyio.to_thread.run_sync(
    verify_password, data.password, user.hashed_password
):
    raise HTTPException(status_code=401, detail="Incorrect email/username or password")

anyio.to_thread.run_sync dispatches the blocking function to anyio's bounded worker-thread pool and awaits it, so the coroutine yields and the loop keeps servicing other requests. app/api/v1/auth.py:121 app/api/v1/auth.py:218-220

💡 Interview Insight

"Your async login calls Argon2 directly — what breaks under load?" This is the classic async-Python trap. The answer is event-loop starvation: throughput collapses as logins serialize behind each other's hashing. Two killer follow-ups: Why does a thread help despite the GIL? — Argon2's native C routine releases the GIL during computation, so you get true parallelism across worker threads. Why anyio and not asyncio.to_thread? — Starlette/FastAPI run on anyio, so you reuse its capacity-limited pool and the code stays portable across asyncio and trio.

3 · JWTs: structure, claims, and the access/refresh split

A JWT is three Base64url segments — header.payload.signature. The header names the algorithm; the payload carries claims; the signature is a MAC over the first two. CareKosh signs with HS256 (HMAC-SHA256), a symmetric scheme keyed by one shared secret. Token creation builds a typed claims dict:

}

🔬 Under the Hood

HMAC vs RSA signatures. HS256 is a symmetric MAC: the same secret both signs and verifies, so anyone who can verify can also forge. That's fine for a single backend that owns the secret. RS256 is asymmetric: a private key signs, and a freely-distributed public key verifies — so untrusted services can validate tokens without the power to mint them. The trade-off: HS256 is faster and simpler but couples all verifiers to one secret (and rotating it invalidates every live token at once); RS256 scales to multi-service / third-party verification at the cost of key management. CareKosh is one backend, so HS256 is the right call.

💡 Interview Insight

"JWTs are stateless — so how do you revoke one?" The honest answer is the architecture here: you don't revoke access tokens. You keep them short-lived (30 min) and accept that a stolen one is valid until exp. Revocability lives on the refresh side: every refresh token has a jti with a matching RefreshToken DB row, and "revoke" means flipping is_revoked=true on that row. The result is a hybrid — stateless access tokens on the hot path for speed, stateful refresh tokens where server-side control matters.

5 · Refresh-token rotation — the centerpiece

This is the question interviewers most love to ask: "A client retries /refresh twice in parallel, or an attacker replays a stolen refresh token — walk me through exactly what happens." The answer is a single SQL statement.

After JWT validation, the code does not read-then-write. It issues one atomic compare-and-swap:

app/api/v1/auth.pyPython
# Atomically claim (revoke) the presented refresh token. Only one concurrent
# request can flip is_revoked from false->true, so a single token can be
# rotated at most once even if it is submitted twice simultaneously.
claim_result = await db.execute(
    update(RefreshToken)
    .where(
        RefreshToken.jti == jti,
        RefreshToken.user_id == user_id,
        RefreshToken.is_revoked.is_(False),    # <-- the concurrency guard
    )
    .values(is_revoked=True)
    .returning(RefreshToken.id)
)
if claim_result.scalar_one_or_none() is None:
    raise HTTPException(status_code=401, detail="Invalid or expired refresh token")

The is_revoked == False predicate is the entire trick. Under the database's row-level locking, two concurrent transactions submitting the same token serialize: the first to acquire the row's lock sees false, flips it to true, and RETURNING yields the row's id. The second, once it can read the row, finds is_revoked already true, so its WHERE matches zero rows, RETURNING yields nothing, scalar_one_or_none() is None, and it gets a 401. Only the winner proceeds to mint a fresh jti + token pair and persist a new RefreshToken row. app/api/v1/auth.py:778-793 This is the SEC-3 fix.

  1. Validate the JWT — verify_refresh_token decodes the token, checks signature + exp, asserts type=="refresh", and returns (user_id, jti).
  2. Atomically claim the jti — the UPDATE…WHERE is_revoked=false…RETURNING flips exactly one row. No matching row ⇒ 401 (replayed, stolen, or already-rotated).
  3. Re-check the user — load the user, enforce is_active; a disabled account can't refresh.
  4. Rotate — mint a brand-new jti + access/refresh pair, persist a new RefreshToken row, single commit.
⚠️ Pitfall

The naive version — SELECT row; if not row.is_revoked: row.is_revoked = True — has a TOCTOU race. Two concurrent requests can both SELECT and read false before either writes true, so both pass the check and both rotate. Now there are two valid refresh tokens descended from one — a replayed stolen token succeeds. Pushing the check into the WHERE clause of a single UPDATE collapses the check-and-set into one indivisible operation and closes the window.

🛡️ Security / OWASP

This is the OAuth 2.0 refresh-token rotation + replay-detection pattern, and it satisfies OWASP ASVS V3 (Session Management): refresh tokens are single-use, so a leaked-and-replayed token is dead on arrival. A hardening extension is full reuse detection — when an already-revoked token is presented, revoke the whole token family / all sessions, since reuse strongly implies theft. CareKosh currently 401s the single request without nuking the family; worth naming as a follow-up.

7 · Email verification, password reset & deletion: SHA-256-hashed tokens

Verification, reset, and deletion links all carry a bearer secret — anyone holding it can act — so it must never sit in the DB in usable form. The generator returns a pair: a high-entropy random value (emailed to the user) and its SHA-256 hash (the only thing stored):

app/utils/email.pyPython
def generate_verification_token() -> tuple[str, str]:
    unhashed_token = secrets.token_urlsafe(32)               # 256 bits of CSPRNG randomness
    hashed_token = hashlib.sha256(unhashed_token.encode()).hexdigest()
    return unhashed_token, hashed_token                       # raw emailed, hash stored

def verify_token(unhashed_token: str, hashed_token: str) -> bool:
    computed_hash = hashlib.sha256(unhashed_token.encode()).hexdigest()
    return secrets.compare_digest(computed_hash, hashed_token)  # constant-time compare

If the database leaks, the attacker holds only hashes and cannot derive a valid link. app/utils/email.py:51-61 The comparison uses secrets.compare_digest — a constant-time compare that doesn't short-circuit on the first mismatched byte, closing the timing side-channel where early-exit comparison could leak how many leading bytes matched.

🔬 Under the Hood

Why SHA-256 here but Argon2 for passwords? Both are secrets-at-rest, but they differ in entropy. A reset token is secrets.token_urlsafe(32) — 256 bits of true randomness — so brute-forcing it is infeasible no matter how fast the hash is; a single cheap SHA-256 pass suffices, and you want it cheap because it runs on every link click. A password is low-entropy and human-chosen, so an offline attacker can run dictionary attacks against leaked hashes — which is exactly why you pay for a slow, memory-hard KDF there. Argon2 on a 256-bit random token would add cost for zero benefit; SHA-256 on a password would be a critical vulnerability.

Email-enumeration protection. forgot-password and resend-verification always return the same message whether or not the account exists, is inactive, or is already verified — so the endpoints can't be used as an oracle to enumerate registered accounts:

⚠️ Pitfall (residual side-channel)

Uniform responses aren't enough on their own — uniform timing matters too. The to_thread offload and per-IP rate limits blunt it; a fuller fix verifies against a dummy hash for missing users to equalize timing.

8 · Two-step, email-confirmed account deletion

11 Domain & CRUD Design

Four resources — Items, Categories, Orders, and a read-only Activity feed — carry the whole product. What makes this layer interview-worthy is not that it does CRUD, but how it does CRUD under concurrency: every mutation is an atomic compare-and-swap, every query is hard-scoped to the owner, and the order workflow is a guarded finite state machine.     return JSONResponse(
        status_code=409,
        content={
            "error": "Version conflict",
            "server_version": item.version,      # already-loaded, stale values
            "server_quantity": item.quantity,
        },
    )
🔬 Under the Hood

Why is this not the classic read-then-write race? Because the safety check is never evaluated in Python between the read and the write — it lives inside the UPDATE's WHERE clause, and the database evaluates the predicate and performs the write as one atomic statement under a row lock. If two requests both read version=5 and both attempt to write, the first matches and bumps to 6; the second now matches zero rows because version=6≠5. The intervening SELECT carries no correctness weight at all — it exists only to return a 404 and to have stale values for the conflict body. This is lock-free optimistic concurrency: no row is held between read and write, so it scales better than pessimistic SELECT … FOR UPDATE under low contention, at the cost of client-visible retries when contention is high.

One detail separates a senior answer from a junior one: on conflict the handler returns the values it already loaded rather than calling db.refresh(item). A concurrent DELETE could make the row vanish, and refreshing a deleted row raises — which would turn a clean 409 into a 500. Returning the stale snapshot keeps the conflict response well-formed no matter what the other writer did.

💡 Interview Insight

Expect: "Walk me through a lost-update scenario." "Optimistic vs pessimistic — trade-offs?" "Why put the version in the WHERE clause instead of comparing in Python?" The killer follow-up is "what happens to the loser?" — here the loser is told the current server_version/server_quantity so the client reconciles and retries; the server never silently overwrites.

Applying an order to stock: claim, then apply

The most delicate operation is apply_order_to_stock — it must add each line's quantity to inventory exactly once, even if a user double-clicks "Apply." The design is a two-phase "claim then apply":

  1. Pre-flight resolve — confirm every order line's item still exists and belongs to the user app/api/v1/orders.py:471-493, so a missing item aborts before any state change (no partial application).
  2. Claim the order — atomically flip RECEIVED → STOCK_UPDATED with a guarded UPDATE. This is the idempotency latch app/api/v1/orders.py:496-515.
  3. Apply the stock — only the request that won the claim loops over items, adding quantities with DB-side arithmetic app/api/v1/orders.py:520-531.
  4. Commit once — claim + every increment + audit logs are all-or-nothing in a single transaction app/api/v1/orders.py:570.

The ordering is the whole trick. Of two concurrent requests, only one can match status=RECEIVED — the DB serializes them via row lock, the first flips it to STOCK_UPDATED, the second sees zero rows and 400s. Because the claim happens before any stock mutation, the terminal status itself becomes the dedupe key. And the increment is quantity = quantity + delta evaluated by the database, not quantity = python_value — so even the stock change is atomic, never last-writer-wins. Flip the order — apply first, set status last — and a double-submit could double-credit before either set the status.

Categories, denormalized snapshots, and transaction boundaries

Categories add one notable rule: a is_default category cannot be deleted — the handler returns 409 before touching anything app/api/v1/categories.py:287-291, because defaults are seeded for every user and deleting one would cascade away its items. (That's also why apply needs its missing-items pre-check — the snapshot is a copy, not a live foreign-key reference.)

Transaction boundaries are explicit and uniform: one commit per request. The session is opened by the get_db dependency, the handler stages its mutation plus an ActivityLog plus an AuditLog, and a single await db.commit() persists all of them atomically app/api/v1/items.py:310-336. The flow is db.add → db.flush() (sends the INSERT and materializes the server-generated id without committing) → stage activity + audit → commit() → refresh(). Crucially the audit service only db.adds and never commits, deferring to the caller — so a stock change can never exist without its audit row, or vice versa. They share one transaction and roll back together.

Defense in depth

12 Security Hardening

Security in this backend is not one feature — it is a stack of independent layers, each of which assumes the layer above it might fail. Layer 1 — Rate limiting at the edge

The first ring throttles abuse before a request ever touches a handler's logic. The limiter is a single shared slowapi.Limiter built once at import time. Two design choices matter for interviews: the key function and the fail-open resilience flags.

app/utils/rate_limiter.pyPython
def get_real_client_ip(request: Request) -> str:
    # Cloudflare sets this to the actual client IP
    cf_ip = request.headers.get("CF-Connecting-IP")
    if cf_ip:
        return cf_ip
    # Standard proxy header (first IP is the real client)
    forwarded_for = request.headers.get("X-Forwarded-For")
    if forwarded_for:
        return forwarded_for.split(",")[0].strip()
    if request.client and request.client.host:
        return request.client.host
    return "unknown"

limiter = Limiter(
    key_func=get_real_client_ip,
    swallow_errors=True,
    in_memory_fallback_enabled=True,
)

Why the custom key_func? This service runs behind Render → Cloudflare, so request.client.host is the proxy's IP — using it would lump every user into one global bucket and throttle the whole world together. The function instead prefers CF-Connecting-IP, then the first entry of X-Forwarded-For (the originating client; later entries are intermediary proxies), and only then falls back to the socket peer (app/utils/rate_limiter.py:10-32).

The two flags trade enforcement for availability. swallow_errors=True means that if the rate-limit storage backend errors, the request is allowed rather than 500'd — a deliberate fail-open. in_memory_fallback_enabled=True keeps counting in per-process memory if the primary store is unreachable. The limits themselves are declared per-endpoint via decorators:

Layer 2 — OWASP security response headers

Every response — success or error — is stamped with a set of defensive headers by a middleware. This layer is pure browser hardening: it costs nothing per request and shrinks several attack classes at once.

app/main.pyPython
@app.middleware("http")
async def add_security_headers(request: Request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["X-XSS-Protection"] = "1; mode=block"
    response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
    response.headers["Cache-Control"] = "no-store"
    if settings.ENVIRONMENT == "production":
        response.headers["Strict-Transport-Security"] = "max-age=31536000; includeSubDomains"
    return response

Note the structure: the headers are applied after await call_next(request), so they wrap whatever the inner layers produced (app/main.py:139-149). Each one maps to a specific threat model:

X-Content-Type-Options: nosniff
Stops browsers from MIME-sniffing a response into an executable type — defeats drive-by content-type confusion.
X-Frame-Options: DENY
The page may not be framed at all — clickjacking protection.
X-XSS-Protection: 1; mode=block
Legacy browser XSS filter; largely superseded by CSP but harmless to keep.
Referrer-Policy: strict-origin-when-cross-origin
Limits how much of the URL (paths, query) leaks to third parties via the Referer header.
Cache-Control: no-store
Prevents browsers and proxies from caching potentially sensitive medical API responses.
Strict-Transport-Security
Forces HTTPS for a year — production only, because pinning HSTS on http://localhost would brick local dev.
🔬 Under the Hood

Why is HSTS gated on environment but the others are not? HSTS is a persistent browser commitment: once a browser sees max-age=31536000 it refuses plain-HTTP to that host for a year. On localhost that is catastrophic; on the others (nosniff, DENY) there is no downside in dev, so they ship unconditionally. Content-Security-Policy is deliberately absent — it is a markup-injection defense for served HTML, and this is a pure-JSON API.

📝 Note

This header layer is genuinely important to call out: it was undocumented in the project's own architecture docs before this guide. It is a real, shipping OWASP control living quietly in twelve lines of middleware.

Layer 3 — The CORS wildcard-vs-credentials invariant

You met the CORS branch in the request-lifecycle chapter; here is the security rule it encodes. In allow-all mode the app sets allow_origins=["*"] with allow_credentials=False; in production it uses specific origins with allow_credentials=True (app/main.py:102-124). This is not a stylistic choice — the Fetch/CORS spec forbids combining Access-Control-Allow-Origin: * with Access-Control-Allow-Credentials: true, and browsers reject any response that does. The reasoning: the wildcard says “any site may read this,” while credentials say “this response is authenticated” — allowing both would let evil.com read a logged-in user's data.

🛡️ Security / OWASP

CORS is browser-enforced, not a server-side authorization control. A native mobile app, curl, or a server-side attacker ignores CORS entirely. Real authorization here comes from JWT bearer auth plus ownership scoping — never from the CORS config. State this explicitly; interviewers love watching candidates mistake CORS for access control.

Layer 4 — Tokens hashed at rest, compared in constant time

Email-verification, password-reset, and account-deletion tokens are bearer secrets: anyone holding the raw value can act on it, exactly like a password. So the raw token is emailed to the user and only its SHA-256 hash is stored in the database. If the DB leaks, an attacker holds hashes that cannot be reversed into valid links.

app/utils/email.pyPython
def generate_verification_token() -> tuple[str, str]:
    unhashed_token = secrets.token_urlsafe(32)
    hashed_token = hashlib.sha256(unhashed_token.encode()).hexdigest()
    return unhashed_token, hashed_token

def verify_token(unhashed_token: str, hashed_token: str) -> bool:
    computed_hash = hashlib.sha256(unhashed_token.encode()).hexdigest()
    return secrets.compare_digest(computed_hash, hashed_token)

Two subtleties. First, why SHA-256 and not Argon2 like the password? Because the token is secrets.token_urlsafe(32) — 256 bits of cryptographically random data — so brute force is infeasible regardless of hash speed; a single fast pass is enough, and you want it fast because it runs on every link click. A user password is low-entropy and human-chosen, so it needs the deliberately slow, memory-hard Argon2 KDF (app/core/security.py:21-24). Second, secrets.compare_digest is a constant-time comparison that closes a timing side-channel — a naive == short-circuits on the first mismatched byte, leaking how many leading bytes matched.

🛡️ Security / OWASP

Reset/verification tokens are treated as secrets-at-rest with the same care as passwords: hashed, never logged, time-bounded, and on a successful password reset every refresh token for the user is revoked in the same transaction (app/api/v1/auth.py:724-731) — invalidating sessions on all other devices, the standard account-takeover defense.

The reset and resend-verification endpoints must never become an oracle that tells an attacker which emails are registered. So they return an identical response whether or not the account exists, is inactive, or is already verified:

Layer 6 — SQL injection is structurally impossible here

There is no string-built SQL anywhere in the request path. Every query goes through SQLAlchemy's expression language, which sends parameterized statements — values travel separately from the SQL text, so a user input can never be parsed as SQL.

This is the cheapest, most complete control in the table: injection is prevented by construction, not by escaping.

Two more rings round out the posture. Secrets are wrapped in pydantic's SecretStr so they render as ********** in logs, reprs, and tracebacks — the raw value is only ever pulled via a dedicated accessor. And in production a fail-closed validator rejects a weak or placeholder signing key outright:

Production readiness

13 Reliability & Observability

A backend that handles medical inventory cannot afford to confuse "my database is briefly unreachable" with "my process is dead," nor leak a SQL constraint name to an attacker on a 500. This chapter is about the unglamorous machinery that keeps the service up, lets you see inside it when something breaks, and fails in safe, boring ways: split health probes, server-side timeouts, graceful shutdown, structured logs, Sentry, correlation IDs, and a tier of global exception handlers that turn raw stack traces into clean, non-leaky JSON.

In one line /health is readiness (probes the DB under a 2s timeout, 503 if down), /live is liveness (process-only, never touches the DB) — and conflating them is how a transient DB blip becomes a cluster-wide restart storm.

Two probes, two different orchestrator actions

The single most important reliability decision in this codebase is that there are two health endpoints, not one, and they answer two orthogonal questions. /live answers "is this process alive and the event loop responsive?" /health answers "can this instance actually serve traffic right now — including its dependencies?" Kubernetes (and Render) attach a different recovery action to each.

🔬 Under the Hood

Why the asyncio.wait_for(..., timeout=2.0) wrapper? A health probe that can block indefinitely is worse than no probe. If a TCP connection to Postgres hangs (half-open socket, network partition), a bare await db.execute(...) never returns — the health endpoint itself hangs, the orchestrator's probe times out ambiguously, and the failure mode is undefined. wait_for imposes a hard ceiling: the probe coroutine is cancelled at 2s and the function deterministically returns False → 503. A health check must always answer quickly, even when the answer is "no."

⚠️ Pitfall

The server-side statement_timeout is 8000ms but the readiness probe caps at 2.0s. That gap is intentional, but reason about it together: a slow-but-alive DB responding between 2s and 8s will be marked unready (503) even though the query would eventually succeed. For a readiness check that's the correct call — you don't want to send user traffic to an instance whose DB is crawling — but it's the kind of subtlety an interviewer will poke at.

    pool_timeout=settings.DATABASE_POOL_TIMEOUT, # 30s backpressure
    pool_pre_ping=True,                          # discard dead connections
    connect_args=_connect_args,
)

The pool-exhaustion story. The hard ceiling is 15 concurrent connections (5 pooled + 10 overflow). Without statement_timeout, a single runaway query would hold its connection forever; under load, a handful of such queries drain all 15, every subsequent request then blocks for the full pool_timeout (30s) waiting for a checkout, and the service effectively deadlocks — one slow query metastasizes into a total outage. The Postgres-side 8s timeout force-aborts the offending statement server-side, returns its connection to the pool, and contains the blast radius app/core/database.py:74-92. pool_pre_ping=True issues a cheap liveness check when a connection is checked out, transparently replacing connections that Neon or a proxy silently killed during idle — so a request after an idle period gets a fresh connection instead of a "server closed the connection unexpectedly" 500.

Observability: logs, Sentry, correlation IDs

You cannot operate what you cannot see, and a backend that returns generic 500s (more on that below) is blind unless its telemetry is solid. Three layers cover that.

1 — Structured logging to stdout (12-factor). logging.basicConfig writes a timestamped, level-tagged line to stdout app/main.py:30-35. 3 — X-Request-ID correlation. A middleware honors an inbound X-Request-ID if an upstream gateway already set one, otherwise mints a fresh uuid4().hex (122 bits of randomness — effectively collision-free), and echoes it on the response so a client can quote it in a bug report.

    response = await call_next(request)

✅ Pro Tip

The win is end-to-end correlation: in a distributed system one user action traverses load balancer → API → DB → external services, and a stable ID threaded through every log line lets you reconstruct the causal chain with a single grep <request-id>. 14 Testing Strategy

A medical-inventory API that mis-applies stock or leaks one family's data to another is worse than useless — it is dangerous. VitalTrack's tests are written to make those failures impossible to ship unnoticed: real Postgres, the real ASGI app driven in-process, and a battery of ownership and concurrency assertions that encode the system's invariants as executable contracts.

In one line 132 async tests across 5 domain files run the real FastAPI app in-process over httpx ASGITransport against a disposable Postgres, with each test getting a freshly drop/created schema — and CI bolts on two cheap structural guards: an exact 39-route contract and a 70% per-file coverage floor on the riskiest modules.

Where these tests sit in the pyramid

The classic testing pyramid puts many fast, isolated unit tests at the bottom, fewer integration tests in the middle, and a thin layer of slow end-to-end tests on top. VitalTrack deliberately leans into the middle band. Almost every test is an integration-leaning API test: it sends a real HTTP request through the entire stack — routing, dependency injection, Pydantic validation, the security middleware, the SQLAlchemy session, and a live Postgres — and asserts on the JSON response and the resulting database state. There is very little pure-unit mocking. That is a conscious trade: the suite is slower per test, but each passing test is strong evidence that the wired-together system actually works, not just that an isolated function does.

🔬 Under the Hood

This is the "honeycomb" or "testing trophy" stance popular in service backends: for an I/O-bound API whose logic is the coordination between HTTP, validation, and SQL, integration tests catch the bugs that matter (wrong status codes, missing ownership filters, broken serialization) that unit tests structurally cannot. The cost — real DB, real event loop — is paid back in confidence.

The tooling: pytest + pytest-asyncio

The entry point is a four-line pytest.ini. The decisive setting is asyncio_mode = auto, which tells pytest-asyncio to treat every async def test_* as a coroutine to run on an event loop — no per-test @pytest.mark.asyncio decorator required.

pytest.iniINI

Why does async testing need this machinery at all? A coroutine isn't a test until something awaits it on a running loop. Plain pytest would call test_foo(), get back an un-awaited coroutine object, and report a pass without executing a single assertion — a silently false green. pytest-asyncio (the same role anyio plays in other suites) supplies the missing event-loop fixture and drives each coroutine to completion. VitalTrack also pins a session-scoped event_loop fixture in tests/conftest.py:82-86 so session-lifetime async resources share one loop.

⚠️ Pitfall

Forget to install/configure pytest-asyncio (or drop asyncio_mode) and your async tests don't fail — they're skipped or pass vacuously. A suite that's green because nothing ran is the most dangerous kind of green. Interviewers love asking "how do you know your async tests actually executed?"

In-process testing: httpx AsyncClient over ASGITransport

The client fixture is the heart of the harness. It wires httpx's AsyncClient to an ASGITransport wrapping the real app object.

ASGI is a contract: an application is just an async callable app(scope, receive, send). There is no Uvicorn, no port binding, no network. Yet because the call enters at the very top of the stack, the entire middleware and routing chain runs for real: CORS, security headers, the request-id middleware, exception handlers, dependency resolution, and Pydantic (de)serialization all execute exactly as in production.

How a commit on main becomes a running, schema-correct API serving traffic — the multi-stage image, the fail-closed boot, the Infrastructure-as-Code blueprint, the migration chain, and the CI gates that stand between you and a bad deploy.

The image is a classic multi-stage build from python:3.12-slim. Docker images are stacks of content-addressed layers. Because the compiler, the apt build caches, and the pip download cache live only in the discarded builder stage, none of those bytes land in the final image — you ship the resolved wheels, not the means to produce them. The one subtlety: any runtime shared library the wheels link against (here libpq5 for asyncpg) must be re-installed in the production stage, since libpq-dev existed only in the builder.

Least privilege: a non-root user

The production stage creates an explicit group/user pair at gid/uid 1000, copies the app with --chown=appuser:appgroup, and switches USER appuser before the process ever runs. Dockerfile:41-60 Containers share the host kernel; by default a container process runs as root (uid 0), so a container escape inherits root on the host. An unprivileged uid shrinks the blast radius and means the process cannot bind privileged ports (< 1024) — which is exactly why the app listens on 8000.

🛡️ Security / OWASP

Running as non-root maps to OWASP/CIS container hardening and the principle of least privilege. Combined with the slim image (no gcc/make in production), an attacker who lands in the container has neither root nor a compiler to build an exploit — a tangible reduction in supply-chain and post-exploitation attack surface.

The serving model: Gunicorn master, Uvicorn workers

The CMD runs Gunicorn as a pure process supervisor that forks Uvicorn ASGI workers. Note the count: 2 workers, not 4 — the count was reduced 4→2 during pre-launch hardening, so any older doc citing 4 workers is stale.

DockerfileDockerfile
CMD gunicorn app.main:app \
    --worker-class uvicorn.workers.UvicornWorker \
    --workers 2 \
    --bind 0.0.0.0:${PORT:-8000} \
    --access-logfile - --error-logfile - --capture-output

You get two layers of scaling. Multiple processes give true parallelism across CPU cores, sidestepping the GIL; within each process the asyncio event loop multiplexes thousands of in-flight awaits on a single thread. Gunicorn's job is to fork the workers, monitor them, and restart any that crash or hang — which plain Uvicorn alone would not supervise. The CMD is deliberately in shell form so ${PORT:-8000} is expanded by /bin/sh at runtime (Render injects PORT); exec-form JSON would treat it as a literal string. Dockerfile:80-86

⚠️ Pitfall

A single blocking call — a sync DB driver, time.sleep, or heavy CPU work on the loop thread — freezes the entire event loop of that worker. Every concurrent request on that worker stalls until it returns. This is why the app is fully async (asyncpg, async handlers) end-to-end; one stray blocking call effectively serializes a whole worker.

💡 Interview Insight

"Why 2 workers?" is really probing whether you understand the worker × pool-size vs. max_connections relationship. Each Gunicorn worker is a full process with its own connection pool. With pool_size=5 and max_overflow=10, 2 workers can open up to 2 × 15 = 30 connections. That number must stay under Postgres/Neon's connection ceiling, or checkouts start timing out (pool_timeout 30s). More workers isn't free — it multiplies your connection footprint linearly.

Boot sequence: migrate-on-boot, fail-closed

The container's ENTRYPOINT is docker-entrypoint.sh, which runs before the server. It parses the DB host/port out of DATABASE_URL if not set, polls pg_isready for up to 30 × 2s = 60s, then runs the migrations. The critical line is the fail-closed migration step:

docker-entrypoint.shbash
echo "Running database migrations..."
# Abort startup if migrations fail — booting against a schema-mismatched DB
# produces confusing runtime 500s. A failed deploy keeps the previous instance.
if ! alembic upgrade head; then
    echo "ERROR: Database migration failed. Aborting startup."
    exit 1
fi
echo "Starting application server..."
exec "$@"

A non-zero exit aborts the new container, so the orchestrator keeps the previous healthy instance running rather than rolling out an app bound to a half-migrated schema (this is the OBS-3 hardening fix). The DB-wait loop, by contrast, is best-effort — after 60s it proceeds with only a WARNING, but the migration step that follows would then fail and fail-close anyway. docker-entrypoint.sh:49-59

📝 Note

The final exec "$@" matters: it replaces the shell process (PID 1) with Gunicorn so signals — notably SIGTERM on deploy or scale-down — reach the server directly for a graceful shutdown. A lingering shell wrapper would swallow them, and the container would be force-killed instead of draining in-flight requests.

⚠️ Pitfall — the health check must NOT probe the DB

VitalTrack runs on Neon serverless Postgres, which scales to zero and cold-starts — the DB is briefly unreachable for a few seconds even when nothing is wrong. The platform health check decides whether to restart the container. If Render or Docker probed /health, a transient Neon cold start would return 503, the orchestrator would kill and restart the container, and you'd get a restart loop that takes down a perfectly healthy app over a DB blip. Probing /live ties restarts to actual process death; DB readiness is surfaced to clients via /health instead.

The whole story end to end — a great "walk me through a deploy" answer:

16 System Design & Scaling to 1M Users

The system-design round is where senior offers are won or lost. The good news: CareKosh is not a toy — its current single-service shape already makes the right structural bets. This chapter takes that one service and grows it to a million users, one bottleneck at a time, so you can walk an interviewer from "one box" to "global fleet" while pointing at real lines of code.

In one line The app is stateless (JWT, no server session) so it scales out trivially; every step after that is really about protecting the one shared resource — Postgres — with pooling, replicas, caching, and queues.

A scaling answer is a story, not a checklist. Each move below follows the same arc: symptom → bottleneck → fix → new trade-off. We start from what CareKosh runs today and only add machinery when a measured pain forces it. That discipline — never scaling speculatively — is itself a senior signal.

💡 Interview Insight

When the interviewer says "how would you scale this to 1M users?" they are not testing whether you can name Redis. They are testing your order of operations: do you clarify and estimate before you architect, do you find the real bottleneck before you fix something cheap, and do you name the new problem each fix introduces. Answer in that order — clarify → estimate → bottleneck → iterate — and you sound like someone who has run a production system.

Step 0 — Clarify and estimate before you draw a box

Before scaling anything, pin the numbers. 1M registered users is not 1M concurrent requests. For an inventory app used a few times a day, a sane back-of-envelope is roughly 1M users × ~10 requests/day ≈ 10M req/day ≈ ~115 req/s average, with peaks maybe 5–10× that. That is a modest number — the honest senior framing is "this fits on a handful of app instances and one well-tuned Postgres for a long time; let me show where it breaks first." Read-heavy too: listing categories and items dominates; writes (orders, stock edits) are rare by comparison. That read/write asymmetry drives almost every decision that follows.

  1. Clarify — 1M total or concurrent? Read- or write-heavy? Latency SLO? Multi-region users? (CareKosh: total, read-heavy, single-region-friendly.)
  2. Estimate — req/s average and peak, payload sizes, DB row counts, hot vs cold data.
  3. Find the bottleneck — for this app it is almost never the stateless Python tier; it is the single Postgres and its connection pool.
  4. Iterate — apply the cheapest fix that moves the bottleneck, then re-measure.

Step 1 — Scale the app tier out (it's already stateless)

Symptom: one instance saturates CPU under peak login/list traffic. Bottleneck: a single Uvicorn process. Fix: run N identical app instances behind a load balancer. The reason this is trivial here — and the first thing to say out loud — is that the app holds no per-user server state. Authentication is a signed JWT verified locally on every request; there is no in-memory session to pin a user to a box.

Any instance can serve any request. Add a box, register it with the load balancer, done. There is no allow_credentials cookie session to coordinate; even CORS is configured for token-based clients, not cookie sessions.

🔬 Under the Hood

Horizontal scaling is only "trivial" when the tier is a pure function of (request, shared backing store). The instant you keep mutable per-user state in process memory — a session dict, a websocket map, an in-process cache of "who is logged in" — you need sticky sessions or a shared store, and round-robin load balancing breaks. CareKosh's stateless JWT design is what buys the easy win. The one place it cheats — in_memory_fallback_enabled on the rate limiter app/utils/rate_limiter.py:41 — is exactly the kind of in-process state that becomes inconsistent across instances, which we fix in Step 4.

New trade-off: per-instance config must be identical, and anything you kept in memory (rate-limit counters, the refresh-token denylist behind swallow_errors) is now wrong across the fleet because each box counts independently. Hold that thought.

Step 2 — Postgres is the bottleneck: vertical first, then PgBouncer

Symptom: as instances multiply, the database starts refusing connections (FATAL: too many connections) long before its CPU is busy. Bottleneck: connection count, not query throughput. This is the single most important scaling fact about this codebase, and it falls straight out of the pool config.

Each engine caps at pool_size + max_overflow = 15 connections per process app/core/database.py:84-92. Now do the multiplication an interviewer is waiting for: 10 app instances × 4 Gunicorn workers × 15 = 600 connections demanded of Postgres. Managed Postgres (Neon, RDS) typically allows a few hundred. The math explodes before you have meaningful traffic, because every worker keeps its own idle pool open.

Fix, in order:

  1. Vertical scale first — bump the DB to more vCPU/RAM. Cheapest, no code change, buys real headroom. Always the first DB move.
  2. Introduce PgBouncer in transaction pooling mode — put a lightweight pooler between the app fleet and Postgres. Apps open thousands of cheap connections to PgBouncer; PgBouncer multiplexes them onto a small fixed set of real Postgres connections, handing a backend connection to a transaction only for its duration.
  3. Shrink the app-side pool — with PgBouncer absorbing the fan-in, each worker needs a far smaller SQLAlchemy pool, so pool_size and max_overflow drop.
⚠️ Pitfall

PgBouncer transaction pooling forbids session-level state across statements: server-side prepared statements, SET session variables, and LISTEN/NOTIFY break because consecutive statements may land on different backends. With asyncpg you must disable statement caching (statement_cache_size=0) or asyncpg's auto-prepared statements will fail intermittently. Mention this unprompted — it is the classic "have you actually run PgBouncer?" tell.

📝 Note

The 8-second statement_timeout baked into connect_args app/core/database.py:74-82 becomes more valuable behind a pooler, not less. A pooler makes backend connections scarce and shared, so one runaway query pinning a backend is now everyone's problem. The server-side timeout force-aborts it and returns the backend to the pool — exactly the pool-exhaustion guard from the resilience chapter, now protecting the whole fleet instead of one process.

Step 3 — Read replicas and read/write splitting

Symptom: the primary's CPU is now the limit, and it's dominated by SELECTs — listing items, categories, orders. Bottleneck: read load on a single writer. Fix: add one or more read replicas (streaming replication) and route read-only queries to them, keeping writes on the primary.

CareKosh is a natural fit because its hot path is overwhelmingly reads — inventory lists, item detail, category trees — while writes (creating an order, editing stock) are infrequent. You'd introduce a second async engine bound to the replica DSN and send GET handlers there, leaving the mutating handlers and the per-request commit() in get_db app/core/database.py:106-123 pointed at the primary.

⚠️ Pitfall

Replication lag means eventual consistency. A user creates an order on the primary, then immediately lists orders from a replica that hasn't caught up — and the order is missing. For a medical inventory app that "lost" a stock change reads as a bug. Mitigations to name: read-your-writes routing (pin a user to the primary for a short window after they write), or accept staleness only for cold/aggregate reads and serve fresh-critical reads from the primary. Don't blanket-route everything to replicas.

New trade-off: you've traded write-primary CPU for application complexity and a consistency model you now have to reason about per endpoint.

Step 4 — Redis: cache hot reads, and fix the state we cheated on

Symptom: the same category and item lists are read thousands of times between rare writes; replicas still burn CPU re-running identical queries. Bottleneck: redundant reads of slowly-changing data. Fix: a shared Redis cache in front of the DB for hot reads, with a TTL and explicit invalidation on write.

Because the data model is keyed by user_id end to end (every Category, Item, Order, ActivityLog carries a user_id FK), cache keys are clean and naturally namespaced — e.g. items:{user_id}:page:{n}. On a stock edit you invalidate just that user's item keys. A modest TTL bounds staleness even if an invalidation is missed.

✅ Pro Tip

State your cache invalidation story before the interviewer asks. "Cache-aside with a short TTL and explicit delete-on-write" beats "I'll cache the list" every time. The two famous hard problems are cache invalidation and naming things — show you respect the first by pairing every TTL with a write-path invalidation, and by accepting bounded staleness via TTL as the safety net for missed invalidations.

Step 5 — Move async work to a real queue

Symptom: email sends, PDF/report generation, and exports run inside the request via FastAPI BackgroundTasks; they're tied to the worker that served the request. Bottleneck (and risk): BackgroundTasks run in the same process and die with the worker — a deploy, crash, or autoscale-down loses in-flight jobs, with no retry and no visibility. Fix: a dedicated job queue (Celery, RQ, or SQS-backed workers) with a broker.

The app already does the right first thing — verification email is queued after commit so a flaky mail provider never rolls back a signup — but a background task is fire-and-forget within one process. Promoting these to a queue gives you durability, automatic retries with backoff, dead-letter handling, and independent scaling of workers from the API tier. The web tier enqueues a lightweight message and returns immediately; a separate worker fleet consumes it.

In-process, no retry, lost on restart. Fine for best-effort, post-commit email at small scale.

📨

Durable broker, retries, DLQ, independent scaling. The right home for email, PDFs, reports, exports.

Step 6 — Partition / shard by user_id when one Postgres won't fit

Symptom: even a vertically maxed primary with replicas and caching can't hold the working set or write throughput. Bottleneck: a single Postgres node. Fix: partition, then shard — and CareKosh's aggregate-root design makes this unusually clean.

Every table fans out from the User aggregate root: Category, Item, Order, ActivityLog, RefreshToken all carry a user_id foreign key with ondelete='CASCADE', and a user's data never joins across user boundaries. That means user_id is a natural shard key: route all of a user's rows to the same shard and queries stay single-shard — no cross-shard joins, no scatter-gather. Start with table partitioning (range/hash within one Postgres) for the big append-only tables like audit_log, and graduate to sharding (the user partition lives on a different physical node) when one node is genuinely full.

💡 Interview Insight

This is where your data model pays off in the scaling round, and interviewers love when those connect. The "everything keyed by user_id, no cross-user joins" property isn't an accident — it's what makes the system embarrassingly shardable. Say it explicitly: "Because the schema is a per-user aggregate, I can shard by user_id with zero cross-shard joins; the only global lookups are by token jti and by email at login, which I'd handle with a small routing/directory table or a global secondary index."

New trade-off: sharding kills cross-user analytics in SQL (you now need an analytics replica or warehouse), complicates the rare global lookups (login-by-email, refresh by jti), and makes schema migrations a fleet-wide operation.

Step 7 — Observability and abuse control at scale

You cannot operate a fleet you can't see. CareKosh already stamps an X-Request-ID on every response and ships Sentry — the foundation is there. At scale you graduate to: metrics (Prometheus: req/s, p50/p95/p99 latency, error rate, pool checkouts, cache hit ratio), distributed tracing (OpenTelemetry, propagating that request ID through queue and DB spans), and dashboards + alerts on SLOs. The readiness probe that runs SELECT 1 under a timeout is what lets the load balancer depool a sick instance instead of restarting it — keep that liveness-vs-readiness split, it's load-balancer-correct behavior under autoscaling.

On abuse: the per-IP rate limiter (login 5/min, register 3/hour) app/utils/rate_limiter.py:10-32 moves to Redis (Step 4) so limits are global, and the proxy-aware CF-Connecting-IP / X-Forwarded-For keying assumes the origin only accepts traffic through the CDN — at fleet scale you enforce that with a firewall/allowlist so attackers can't bypass the limiter by hitting an instance directly.

Step 8 — CDN for static and images

Symptom: item images and any static assets eat bandwidth and add latency for distant users. Fix: serve them from object storage (S3) behind a CDN. The API stores only the image URL (image_uri, validated to http(s) at the schema boundary) — the bytes never flow through your Python tier. A CDN caches them at the edge near each user, slashing latency and offloading origin bandwidth entirely. This keeps the app tier doing what it's good at: JSON over a fast DB, nothing else.

The whole story, in one breath

Tie it together the way you'd close the round: "The app is stateless JWT auth, so I scale it out behind a load balancer for free. The real bottleneck is the one Postgres — I vertical-scale it, then add PgBouncer because workers×pool explode connections, then read replicas for the read-heavy inventory lists with read-your-writes to handle lag, then Redis to cache hot per-user lists and to make rate limiting and token revocation correct across instances, then a durable queue for email and reports so they survive restarts, and when one node truly can't hold it, I shard by user_id — which is clean because the whole schema is a per-user aggregate. Application Factory — construction decoupled from instantiation

The app is not a module-level global created at import. It is built by a function, create_app() -> FastAPI app/main.py:81-164, that registers — in one explicit, ordered place — CORS, the correlation-ID and security-headers middleware, the versioned router, the rate limiter, the exception handlers, and the health endpoints. Only afterward does a single module-level statement instantiate it. The payoff is testability and control: each test can build an isolated app (and override its dependencies), there are no import-time side effects beyond the one construction line, and the middleware ordering — which determines the runtime onion — lives in one readable function rather than scattered across decorators.

That is the textbook Unit of Work — a single atomic boundary around everything the request touched. A handler that stages an item, an ActivityLog, and an AuditLog either persists all three or none.

        yield session
        await session.commit()      # success → make the whole request durable

🔬 Under the Hood

The session is a SQLAlchemy Identity Map + Unit of Work: it tracks loaded objects and accumulates pending changes, flushing SQL lazily and committing once. The handlers treat the session as a repository-ish data-access seam — queries like select(Item).where(Item.user_id == current_user.id) stand in for a repository method. There's no explicit Repository class here, but the Session is the abstraction that hides connection management and change tracking from the handler.

Singleton settings via lru_cache

Configuration is parsed and validated exactly once. @lru_cache on get_settings() turns the typed Settings object into a process-wide singleton, so the environment is read a single time at startup — one fail-fast point, one source of truth — rather than re-parsed on every import or DI call. Service layer that defers the commit

The audit service app/services/audit.py:8-27 shows a subtle, mature decision. log_audit builds an AuditLog and db.add()s it but deliberately does not commit — the explicit comment says to let the caller's transaction commit the mutation and the audit row together. This keeps a single responsibility (record an audit entry) in one place while respecting the Unit of Work owned by get_db. If the service committed internally, you could end up with an audit row for a change that later rolled back — a corrupted ledger.

⚠️ Pitfall

A helper that commits "to be safe" silently fractures the transaction boundary. Here the service stages the row and trusts the request-level Unit of Work to make it atomic with the business mutation. Committing inside the helper would create phantom audit entries (audit persisted, mutation rolled back) or unaudited mutations — the exact failure modes a compliance-grade trail must never have.

Each middleware is a handler in a chain: it does a little work, calls await call_next(request) to delegate inward, then optionally mutates the response on the way out. Q01FoundationalWhat is ASGI, and why does FastAPI need it instead of WSGI?›

WSGI (Web Server Gateway Interface) is the old synchronous contract between Python web servers and apps: one callable, app(environ, start_response), called once per request, blocking until it returns. It has no concept of await, long-lived connections, or WebSockets. ASGI (Asynchronous Server Gateway Interface) generalises it: the app is async def __call__(self, scope, receive, send), where scope describes the connection, and receive/send are async channels for streaming events.

FastAPI is built on Starlette, an ASGI framework, and an app object is an ASGI callable — that is exactly what Uvicorn imports as app.main:app and invokes. ASGI is what lets a single worker handle thousands of concurrent connections cooperatively (every await on I/O yields control), and it's what makes WebSockets, SSE, and long-polling possible. With WSGI you'd need one thread or process blocked per in-flight request. The trade-off: in async code you must never call a blocking library, or you stall the whole event loop.

Q02FoundationalWhat does async/await actually do, and how does one worker serve many requests?›

An async def function is a coroutine — calling it doesn't run it, it returns a coroutine object that the event loop drives. await is a cooperative yield point: when you await something I/O-bound (a DB query, an HTTP call), the coroutine suspends and hands control back to the event loop, which is free to advance other coroutines while the OS waits on that socket. When the I/O completes, the loop resumes your coroutine where it left off.

So a single-threaded worker interleaves many requests: while request A is blocked on await db.execute(...), the loop runs request B's code. This is concurrency without parallelism — one CPU, many in-flight I/O operations. It beats a thread-per-request model for I/O-bound web work because there's no thread context-switching or memory overhead, and it sidesteps the GIL (which only matters for CPU-bound parallelism). The catch: the model only works if every request spends its time waiting, not computing. A CPU-bound chunk (e.g. password hashing) must be pushed off the loop — see Q03.

Q03AdvancedWhat's the single biggest async pitfall, and how does this codebase avoid it?›

Q08AdvancedIn FastAPI, when should a route be def vs async def?›

FastAPI accepts both and treats them differently. An async def route runs directly on the event loop — correct when its body only awaits non-blocking I/O (async DB driver, async HTTP client). A plain def route is run by FastAPI in an external threadpool, so it won't block the loop even if it does blocking work. So the decision rule is about what the body does, not preference:

  • Body awaits async I/O (like every handler here, using asyncpg) → async def.
  • Body calls a blocking/sync library (a sync DB driver, requests, heavy CPU) and you can't make it async → def, and let FastAPI offload it.
  • Worst case: async def that calls a blocking function directly — it stalls the loop for every concurrent request. That's the trap Q03 is about; the fix is either def or anyio.to_thread.run_sync.

CareKosh uses async def everywhere because its I/O (DB, email) is async, and surgically threads only the CPU-bound Argon2 hash.

Q11IntermediateExplain the transaction-per-request pattern — when does the commit actually fire?›

Each request gets one AsyncSession from the get_db yield-dependency, wrapped in a single transaction (the Unit of Work pattern). In practice the handler does its db.add() / db.execute() calls and then commits explicitly with await db.commit() before returning — so it can refresh() and serialize fresh data (e.g. app/api/v1/auth.py:157). The dependency's post-yield code also commits on success, but that's a fallback; its load-bearing job is to roll back on error:

flow: handler runs (and usually commits) → returns → get_db resumes past its yield → await session.commit() (a no-op if already committed) → session.close(). If the handler raises anything, control hits the except branch and the whole transaction is rollback-ed, so a request is all-or-nothing. This is why a handler can create a user, a refresh-token row, and an activity-log row and trust they either all persist or none do. Handlers that need an ID mid-flow use await db.flush() (sends SQL, gets the PK, stays in the transaction) rather than committing early. The mental model: the route describes intent; the dependency owns the transaction boundary.

Q13AdvancedOptimistic vs pessimistic concurrency — how does item update implement it?›

CareKosh uses optimistic control via the items.version column, doing the check-and-bump inside a single SQL UPDATE (a compare-and-swap):

Q15AdvancedWhy does order_items.item_id have no foreign key — isn't that a bug?›

It's deliberate denormalization for historical accuracy. An OrderItem stores a snapshot of the item at order time — its name, brand, unit, current_stock, minimum_stock — and references the source item only by a plain item_id string with no FK constraint. Two reasons: (1) an order is a financial/audit record that must remain truthful even if the item is later renamed or deleted — with a hard FK + cascade, deleting an item would erase order history (or be blocked entirely); (2) the snapshot captures what was true then, not what's true now (a price/quantity at order time).

The trade-off is the classic normalization vs. denormalization tension: you accept duplicated data and give up referential integrity on that column in exchange for immutable history and decoupled lifecycles. The interview-grade framing: normalize for the operational truth, snapshot for the historical truth. Invoices, ledgers, and audit logs almost always snapshot for exactly this reason.

Q16AdvancedWhat does expire_on_commit=False do, and what async trap does it dodge?›

Q17IntermediateWhy store enums as VARCHAR (native_enum=False) instead of a native Postgres ENUM?›

Q18IntermediateWhat pagination does this use, and what's wrong with OFFSET at scale?›

Q19IntermediateWhat's inside a JWT, and why HS256 with a short access-token TTL?›

A JWT is three base64url parts: header.payload.signature. The header names the algorithm; the payload carries claims (sub = user id, exp, iat, plus this app's custom type of "access"/"refresh" and a jti); the signature authenticates the first two. HS256 is HMAC-SHA256 — a symmetric MAC: the same SECRET_KEY signs and verifies. That's simple and fast and fine for a single service that both issues and checks tokens. RS256 is asymmetric (private key signs, public key verifies) — you'd switch to it when many independent services must verify tokens without holding the signing secret.

The token is stateless: any worker validates it with the secret, no DB lookup — that's what makes the API horizontally scalable. The cost is you can't instantly un-issue a stateless token, so the access token is deliberately short-lived (30 minutes) and paired with a long-lived (30-day) refresh token that is tracked in the DB and can be revoked. Short access TTL bounds the damage window of a leaked token app/core/security.py:40-56.

Q20AdvancedJWTs are stateless — so how do you actually revoke one?›

You can't truly revoke a stateless access token, so you design around it: keep access tokens short (30 min) and make the long-lived refresh token the revocable anchor. Each refresh token's jti is persisted as a RefreshToken row with is_revoked, expiry, device, and IP. Revocation then becomes a DB write:

  • Logout sets is_revoked=True on that one token.
  • Password change / reset bulk-revokes all of a user's refresh tokens app/api/v1/auth.py:1220-1224 — so changing your password really does log out every other device.
  • Rotation (Q21) revokes the presented token every time it's used.

A revoked or unknown refresh token fails at /auth/refresh, so within at most one access-token lifetime the session is dead. Q21StaffWalk through refresh-token rotation and prove it's race-safe.›

On each /auth/refresh, the presented token is verified, then single-use rotated: the old token is revoked and a brand-new pair is minted. The safety lives in how the revoke is done — an atomic conditional update that claims the token in one statement:

app/api/v1/auth.pyPython
claim = await db.execute(
    update(RefreshToken)
    .where(RefreshToken.jti == jti,
           RefreshToken.user_id == user_id,
           RefreshToken.is_revoked.is_(False))
    .values(is_revoked=True)
    .returning(RefreshToken.id)
)
if claim.scalar_one_or_none() is None:
    raise HTTPException(401, "Invalid or expired refresh token")

The WHERE is_revoked = false + RETURNING means the check and the revoke are the same operation, serialized by the row lock. If the same token is submitted twice concurrently (double-submit, or a stolen replay), exactly one UPDATE flips false → true and gets a row back; the other matches nothing and gets a 401 app/api/v1/auth.py:778-793. Contrast the naive version — SELECT, check in Python, then UPDATE — which has a TOCTOU window where both requests read "not revoked" and both succeed. CAS closes that window; a refresh token is provably usable at most once.

Q23AdvancedHow does the auth flow prevent email enumeration, and why are reset tokens hashed?›

Enumeration is when an attacker learns which emails are registered by watching how the API responds. The forgot-password and resend-verification endpoints defend against it by returning the same generic success message whether or not the account exists (or is already verified) — "If an account exists with this email, a link will be sent." No timing or status-code tell app/api/v1/auth.py:434-444. That denies attackers a user-list oracle.

Reset/verification tokens are treated like passwords: the raw token is emailed to the user, but only its SHA-256 hash is stored in the DB, with an expiry. Verification re-hashes the incoming token and compares to the stored hash. So a database leak yields hashes, not usable tokens — and the constant-time-ish comparison + short expiry shrink the attack surface further. On a successful reset, all refresh tokens are revoked, so a compromised session can't survive a password reset. Same defense-in-depth philosophy as the password storage itself.

Q24AdvancedWhat's the difference between liveness and readiness probes, and why does conflating them cause outages?›

They answer different questions. Liveness = "is the process alive, or wedged and in need of a restart?" Readiness = "is it able to serve traffic right now?" CareKosh exposes both app/main.py:250-299: /live returns 200 unconditionally (process-only, never touches the DB), while /health probes the DB with SELECT 1 under a 2-second asyncio.wait_for and returns 503 if the database is unreachable.

The failure mode of conflating them: if your liveness probe checked the DB, then a transient Neon blip would make every container report "dead," the orchestrator would kill and restart healthy app processes en masse, and the restart storm would amplify the outage — even though the apps were fine and just needed the DB to come back. Correct design: liveness must depend only on the process; readiness can depend on dependencies and is used to pull a pod out of the load-balancer rotation without killing it. The container HEALTHCHECK and Render both probe /live for exactly this reason.

Q26StaffHow would you scale this service to 1,000,000 users?›

ASGI vs WSGI?WSGI is the synchronous one-request-per-callable Python web interface; ASGI is its async successor with async def __call__(scope, receive, send), supporting websockets and lifespan. Uvicorn speaks ASGI.
What does await actually do?Suspends the coroutine at that point and yields control back to the event loop, which runs other ready coroutines until the awaited awaitable resolves.
Process vs thread vs coroutine?Process = own memory + own GIL (true parallelism). Thread = shared memory, OS-scheduled, GIL-bound for Python bytecode. Coroutine = cooperative, single-thread, yields at await — concurrency without parallelism.
What is the GIL?The Global Interpreter Lock: one thread executes Python bytecode at a time. C extensions (Argon2, asyncpg) release it during native work — which is why hashing in a worker thread gives real parallelism.

What is N+1?One query loads N parents, then one lazy query per parent loads its children — 1+N round-trips. Fixed by eager batching (selectin) or a join.

What is CORS preflight?A browser OPTIONS request sent before a "non-simple" cross-origin call, asking the server which methods/headers/origins are allowed. CORSMiddleware short-circuits it.

1. Add multi-tenant orgs to CareKosh

  • Decide the isolation model first — shared-DB + org_id discriminator column (cheapest, what I'd choose) vs schema-per-tenant vs DB-per-tenant. State the trade-off: row-level is simplest to operate but leans entirely on query scoping for isolation.
  • Add org_id to every tenant-scoped table and a composite index (org_id, ...). Today rows are scoped by user_id; the org becomes the new tenancy boundary above the user.
  • Enforce scoping centrally, not per-handler — push the tenant filter into a dependency (a current_org dep derived from the JWT) so every query is automatically WHERE org_id = :org. Forgetting it on one endpoint is a cross-tenant data leak.
  • Defense-in-depth at the DB — Postgres Row-Level Security policies keyed on a session variable, so even a query that forgets the filter can't read another org's rows.

2. Add real-time low-stock alerts

  • Detect at the write boundary — the stock-apply path already knows the new quantity and the item's minimum_stock; emit an event when it crosses below the threshold, inside the same transaction that mutated stock.
  • Don't push from the request thread — enqueue the alert (outbox row committed atomically with the stock change, drained by a worker) so a flaky notification provider never rolls back the inventory update. CareKosh already uses post-commit BackgroundTask for emails as a lighter version of this.
  • Choose the transport by latency need — WebSocket/SSE for a live dashboard, push notification or email for a phone. ASGI makes WebSockets first-class.
  • Debounce — suppress repeat alerts for an item already below threshold so a steady drain doesn't spam; track "last alerted at" per item.

3. Make it offline-first with conflict resolution

📝 Note

This one is a gift — the schema was designed for it. Every entity already carries a nullable indexed local_id (client-generated UUID) and Item carries a version. app/models/item.py:113-125

  • Identity mapping via local_id — the client creates rows offline with its own UUID; the API persists it as local_id and returns the canonical id so the client can reconcile. (Scaffolding only — there is no live /sync/upsert endpoint today; CI even rejects /sync routes.)
  • Conflict detection via version — the optimistic-lock version is exactly the conflict signal. If the client's base version is stale, the server rejects (409) and the client must reconcile.
  • Pick a resolution policy — last-write-wins (simplest), server-wins, or field-level merge. For medical inventory I'd surface the conflict to the user rather than silently LWW, since stock counts are safety-critical.
  • Pull deltas with a watermark — clients pull changes since their last updated_at/sequence to stay incrementally in sync.

4. Add role-based access control (RBAC)

  • Extend, don't replace, the dep chain — CareKosh already layers active < verified < superuser through composed dependencies. RBAC generalizes that boolean ladder into roles/permissions.
  • Model it — roles table, permissions table, role_permissions join, and a user_roles assignment (org-scoped if combined with prompt 1).
  • Enforce declaratively — a require_permission("item:delete") dependency factory, so routes read Depends(require_permission(...)) and authz stays in the type-annotated DI layer, self-documenting in OpenAPI.
  • Audit the grants — write role/permission changes into the existing AuditLog with JSONB before/after — privilege escalation is exactly what auditors check.

5. Export 10M-row reports without OOM

  • Never materialize the full result set — the failure mode is loading 10M ORM rows into memory. Stream instead.
  • Server-side cursor + chunked rows — use a streaming/server-side cursor and yield_per so the DB sends rows in batches; map each batch to CSV lines.
  • Stream the HTTP response — FastAPI's StreamingResponse over an async generator so bytes flow to the client as they're produced; constant memory regardless of row count.
  • For very large or slow exports, go async-job — kick off a background job that writes to object storage (S3) and email a signed download link, so a 10-minute export doesn't hold an HTTP connection or a pooled DB connection (recall the 8s statement_timeout would kill a single giant query anyway).

6. Add full-text search over items

  • Start in Postgres, not a new system — a tsvector column over name/brand/supplier_name with a GIN index handles ranked, stemmed search with zero new infrastructure.
  • Keep the index fresh — a generated column or a trigger updates the tsvector on write; query with @@ to_tsquery(...) and rank with ts_rank.
  • Scope it to the tenant — combine the FTS predicate with the existing user_id/org_id filter so search never crosses tenancy.
  • Reach for Elasticsearch/OpenSearch only when you outgrow PG — typo-tolerance, faceting, cross-entity search, or huge scale. Name the cost: a second datastore to keep in sync (CDC/outbox), which is real operational weight.

🔬 Under the Hood

With the default expire_on_commit=True, SQLAlchemy marks every attribute stale after commit(). The next attribute access triggers a fresh SELECT — fine synchronously, but under asyncio that lazy load is implicit blocking I/O on an already-closed transaction, which raises a greenlet/MissingGreenlet error. Setting it False keeps committed objects usable for serialization. Knowing this single line is the difference between "I copied a tutorial" and "I understand the async session lifecycle."

Second, the primitive that keeps Argon2 — deliberately slow, CPU-bound hashing — from freezing the entire event loop:

from anyio import to_thread

# Argon2 is CPU-bound; run it on a worker thread so the
# event loop stays free to serve other requests.
hashed = await to_thread.run_sync(hash_password, payload.password)

The fix is exactly the line above — and being able to explain why (the GIL, one event-loop thread, run_in_executor underneath AnyIO) is what the question is really testing.

Runtime & concurrency

This cluster is about how Python serves a request without blocking. The single most important fact: CPython has a GIL, so CPU-bound work like Argon2 must be pushed off the event loop or every concurrent request stalls behind it.

TermOne-line definition
ASGIAsynchronous Server Gateway Interface — the async contract app(scope, receive, send) that lets a server stream requests into an async app; FastAPI is an ASGI callable.
WSGIASGI's synchronous predecessor (one request per worker thread, blocking) — Flask/Django-classic; cannot natively express await.
event loopThe single-threaded scheduler that runs coroutines, suspending each one at await so the thread can serve other requests during I/O waits.
coroutineA function defined with async def that returns an awaitable; it can pause and resume rather than running to completion in one go.
awaitYields control back to the event loop until an awaitable resolves — the suspension point that makes single-threaded concurrency possible.
GILCPython's Global Interpreter Lock — only one thread executes Python bytecode at a time, so threads help I/O concurrency but not CPU parallelism.
UvicornThe ASGI server that parses HTTP into a scope and calls the app on the event loop; runs app.main:app.
GunicornA production process manager that forks multiple UvicornWorker processes for parallelism and graceful restarts.
workerOne OS process serving requests; N workers sidestep the GIL by giving you N independent interpreters (and N copies of in-memory state).
middlewareA layer wrapping the app like an onion; add_middleware prepends, so the last registered runs outermost (CORS → request-id → security headers).
🔬 Under the Hood

The reason VitalTrack threads Argon2 off the loop with anyio.to_thread.run_sync is purely the GIL plus the event loop: hashing is CPU-bound and does not release at an await, so running it inline would freeze the single event-loop thread and stall every other in-flight request for ~100ms. See app/api/v1/auth.py:121.

Auth, tokens & hashing

TermOne-line definition
JWTJSON Web Token — a signed, self-contained credential (header.payload.signature) the server can verify without a DB lookup.
claimA key/value statement inside a JWT payload; VitalTrack issues sub, exp, iat, jti, plus a custom type (app/core/security.py:40-131).
jtiJWT ID — a unique token identifier; refresh tokens store it (unique=True) so a specific token can be revoked or rotated.
HS256 / HMACSymmetric signing: HMAC-SHA256 with a shared secret. Fast, but verifier and signer share the same key (vs. asymmetric RS256).
access tokenShort-lived JWT (type=access) sent on every request; stateless, so no DB hit to authenticate.
refresh tokenLonger-lived token (type=refresh) used only to mint new access tokens; stored server-side by jti so it can be revoked.
token rotationEach refresh atomically revokes the old token and issues a new pair, so a stolen refresh token works at most once (app/api/v1/auth.py:778-820).
Argon2The memory-hard, GPU-resistant password hash (Argon2id) and VitalTrack's preferred scheme — RFC 9106, OWASP-recommended.
bcryptAn older adaptive hash kept as a fallback scheme so legacy hashes still verify and silently rehash to Argon2 on next login.
saltPer-password random bytes mixed into the hash so identical passwords produce different digests, defeating rainbow tables (Argon2 embeds it).
hashingOne-way transformation of a password into a verifiable digest; deliberately slow for passwords (the opposite goal of a fast hash).
app/core/security.pyPython
pwd_context = CryptContext(schemes=["argon2", "bcrypt"], deprecated="auto")

def get_password_hash(password: str) -> str:
    return pwd_context.hash(password)          # first scheme = argon2

def verify_password(plain: str, hashed: str) -> bool:
    return pwd_context.verify(plain, hashed)    # verifies argon2 OR legacy bcrypt

The deprecated="auto" flag is the key word: any hash not made with the first scheme is marked stale, so on a successful verify the app can transparently re-hash with Argon2 — algorithm agility without a forced password reset.

🛡️ Security / OWASP

Per the OWASP Password Storage Cheat Sheet, the modern answer to "how do you store passwords?" is Argon2id (≈19 MiB memory, t=2, p=1) with bcrypt as an acceptable fallback. Quoting the exact scheme list at app/core/security.py:21-34 and the threaded-off-the-loop hashing turns a textbook answer into a code-grounded one.

SQLAlchemy & the session

TermOne-line definition
SQLAlchemyThe Python ORM + Core toolkit; VitalTrack uses 2.0-style typed mapping (Mapped[...] / mapped_column) over async asyncpg.
ORMObject-Relational Mapper — maps rows to Python objects and tracks their changes, so you write objects instead of raw SQL.
sessionThe per-request workspace holding loaded objects and pending changes; one is opened, yielded, committed, and closed per request.
unit of workThe pattern where all changes in a request are tracked and flushed in one atomic commit — get_db implements it (commit on success, rollback on raise).
identity mapThe session guarantee that one row maps to exactly one Python object within that session, so reads are consistent and de-duplicated.
connection poolA reused set of open DB connections; VitalTrack runs pool_size=5 + max_overflow=10 → 15 max (app/core/database.py:84-92).
pool_pre_pingA cheap liveness check on checkout that discards connections silently killed by an idle timeout or proxy, avoiding "server closed connection" errors.
statement_timeoutA Postgres-side 8s per-statement cap (server_settings) so one runaway query can't pin a pooled connection and cascade into pool exhaustion.
expire_on_commitSet False so attributes stay readable after commit — critical in async, where a post-commit reload would raise MissingGreenlet.
lazy / eager loadingLazy fetches a relationship on first access (risking N+1); eager pre-fetches it with the parent query.
selectinAn eager strategy issuing one extra query WHERE fk IN (...) to batch-load children — the canonical N+1 fix, and async-safe (app/models/user.py:116-145).

Forgetting expire_on_commit=False in an async app is a classic production bug: everything passes in tests that don't serialize after commit, then real responses raise greenlet_spawn has not been called because Pydantic touches an expired attribute and triggers implicit lazy I/O. See app/core/database.py:97.

TermOne-line definition
transactionA group of statements that commit all-or-nothing; VitalTrack's boundary is the request — the audit row and the mutation commit together.

💡 Interview Insight

"You already have ondelete=CASCADE in the DB — why also cascade in the ORM?" is a trap. The crisp answer: they cover different deletion paths. ORM delete-orphan only fires for objects deleted through the session; the DB cascade protects deletes that bypass the ORM (raw SQL, bulk delete, admin tooling). Defense in depth — and you'd add passive_deletes=True to stop them duplicating work.

How the URL space is assembled

FastAPI builds the final path by stacking router prefixes. The app includes one umbrella router with prefix /api/v1 app/api/v1/__init__.py, which in turn includes five resource routers each carrying their own prefix (/auth, /categories, /items, /orders, /activities). Reading the "Auth required?" column

There is no auth middleware. A route requires authentication if and only if its handler signature includes the CurrentUser dependency alias — Annotated[User, Depends(get_current_user)] app/api/deps.py — which extracts the bearer token, decodes the access JWT, loads the user with noload("*"), and rejects inactive accounts. Routes without it are public by construction. The email-link HTML routes (verify-email, reset form, confirm-delete) are intentionally public because the high-entropy token in the URL is the credential.

✅ Pro Tip

This is a cheap, high-value pattern worth name-dropping in an interview: a "route inventory" assertion catches an entire class of accidents — a router that fails to register, a duplicate decorator, a debug endpoint shipped by mistake, a public route added without review. The diff that changes the count forces a human to update the expected number and, by extension, to consciously acknowledge that the API surface changed.

🛡️ Security / OWASP

An enumerated, count-pinned endpoint catalog is a direct control against OWASP API9:2023 — Improper Inventory Management. Undocumented or forgotten endpoints (a stale debug route, a deprecated /sync handler) are a classic breach vector. Here the inventory is asserted in CI, the legacy /sync routes are explicitly forbidden from returning, and the interactive docs at /docs are gated behind DEBUG app/main.py:88-90 so production never advertises its own schema.
