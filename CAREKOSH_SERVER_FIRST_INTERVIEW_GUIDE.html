Interview-Ready Architecture Guide

Server-First vs Offline-First

A real-world case study from CareKosh — how a data corruption bug in a medical inventory app forced a complete architecture migration, and what it teaches about state management, data integrity, and production engineering.

Chapter 1

The Incident — What Went Wrong

CareKosh is a home ICU medical inventory management app for family caregivers. It tracks life-critical supplies — oxygen cylinders, suction machines, ventilator filters, medications. Running out of stock is not an inconvenience; it's a medical emergency.

The app originally used an offline-first architecture: all data lived locally on the phone in Zustand stores persisted to AsyncStorage, with a background sync engine pushing changes to the PostgreSQL server. This worked well in development. Then it destroyed production data.

The Data Corruption Event

After a user logged out and logged back in, 31 out of 33 inventory items had their quantities reset to zero. The suction machine showed 0 units. The oxygen cylinder showed 0 units. The ventilator filter showed 0 units. For a caregiver managing a home ICU, this is catastrophic — the app told them they had nothing.

Timeline of the Bug

User builds inventory

32 items created with correct stock quantities. Server has accurate data.

User creates and receives order

Stock quantities updated. Server still accurate.

User logs out and logs back in

Zustand persist middleware rehydrates stale local state from AsyncStorage — all quantities show 0.

Sync engine fires automatically

scheduleSyncToBackend() pushes local state (with 0 quantities) to server via blind UPSERT. Server data overwritten.

Server now has all zeros

31 of 33 items: quantity = 0. All with identical timestamps proving a bulk overwrite event.

The root cause was a rehydration race condition: Zustand's persist middleware loaded stale data from AsyncStorage before the server fetch completed, and the sync engine treated local data as authoritative, pushing zeros to the server.

Chapter 2

Offline-First Architecture (Before)

The original architecture followed the offline-first pattern. Every piece of data existed in three places simultaneously, creating a tri-state consistency problem.

The Three Fatal Flaws

Flaw What Happened Why It's Dangerous
1. Rehydration race Zustand's persist loads stale data from AsyncStorage before server fetch completes Stale data (quantity=0) becomes "current" state, triggering sync
2. Blind UPSERT sync.ts pushes ALL local data to server without comparing timestamps or versions Server's correct data gets overwritten by stale local data
3. No sync guard scheduleSyncToBackend() fires on ANY Zustand mutation, including rehydration The act of loading stale data triggers pushing it to server — a self-reinforcing corruption loop
Chapter 3

Server-First Architecture (After)

The migration (PR #4) replaced the entire data flow. The server is now the single source of truth. Local state is ephemeral — it exists only for the current session's UI rendering.

The Tradeoff

The offline-first version let users create and edit inventory items without internet. The server-first version requires connectivity for mutations. For CareKosh, this tradeoff was correct: caregivers managing home ICU supplies are in homes with WiFi. They are not in remote areas without connectivity. Data accuracy matters more than offline editing for medical inventory.

Chapter 5

The Root Cause — Deep Dive

The bug was a rehydration race condition — a class of problem specific to offline-first architectures that use persist middleware with background sync.

The Race Condition (step by step)

TIMELINE OF A LOGIN (what happened in v1):
════════════════════════════════════════════

t=0ms    User enters credentials, taps Login
t=50ms   Server authenticates, returns JWT tokens
t=51ms   App stores tokens in SecureStore
t=52ms   App navigates to Dashboard

t=53ms   ⚠️ Zustand persist middleware REHYDRATES from AsyncStorage
         → Loads items[], categories[], orders[] from PREVIOUS session
         → Previous session had stale data (quantity = 0 for most items)
         → Zustand state is now populated with stale zeros

t=54ms   ⚠️ Zustand mutation detected (rehydration counts as mutation)
         → scheduleSyncToBackend() fires (debounced 2 seconds)

t=100ms  Server fetch begins: GET /items
         → This will return CORRECT data from PostgreSQL

t=2054ms ⚠️ SYNC FIRES before server fetch response arrives
         → POST /sync/push with ALL local items (quantity = 0)
         → Server receives 33 items, all with quantity = 0
         → Server does UPSERT — overwrites correct quantities with zeros

t=2200ms Server fetch response arrives with... zeros
         → Because the sync ALREADY overwrote the server data
         → UI shows zeros. Server has zeros. Backup has zeros.
         → Data is irreversibly corrupted.

Why this is a CLASS of problem, not a one-off bug

This wasn't a coding mistake that could be fixed with a null check. It's an architectural inevitability when you combine: (1) a persist middleware that rehydrates state, (2) a sync engine that treats local state as authoritative, and (3) no version checking on the server. Any offline-first app with these three properties will eventually hit this bug. The fix was to eliminate the architecture, not patch the symptoms.

Chapter 6

Why Not Just Fix the Sync?

A reasonable question. Why delete 1,711 lines of code instead of adding a sync guard? Six reasons:

Proposed Fix Why It's Insufficient
Add isInitialSyncComplete flag Prevents the rehydration race but doesn't fix timestamp-based conflicts, orphan data, or concurrent edits from multiple devices
Compare timestamps before UPSERT Requires clock synchronization between phone and server. Mobile device clocks drift. Timezone bugs are ubiquitous. Timestamp comparison is unreliable across devices
Add version column and check on sync This IS the right approach — but now you've reinvented server-side OCC while still maintaining 1,711 lines of sync code. Why not just call the server directly?
Queue mutations and replay them CRDTs (Conflict-free Replicated Data Types) solve this properly but add enormous complexity. Overkill for an app where users have WiFi
Disable persist middleware Fixes the rehydration race but removes the core benefit of offline-first (data survives app restart). Now you have sync code that serves no purpose
Server-wins conflict resolution Fixes corruption but means offline edits are silently discarded. User makes changes offline, comes back online — changes gone. Bad UX

The Engineering Decision

Every proposed fix either partially solved the problem while adding complexity, or fully solved it by effectively becoming server-first with extra steps. The cleanest solution was to acknowledge that CareKosh users always have internet (home WiFi) and eliminate the entire sync layer, replacing it with direct CRUD calls via TanStack Query — an industry-standard library that handles caching, refetching, and optimistic updates out of the box.

Chapter 7

Data Integrity Mechanisms Added

The migration didn't just swap the data flow — it added four layers of data protection that didn't exist before.

1. Optimistic Concurrency Control (OCC)

Every table has a version integer column. On every UPDATE, the backend checks WHERE version = expected_version. If another client modified the row since you read it, the version won't match and you get a 409 Conflict instead of silently overwriting.

# Backend: items.py
async def update_item(item_id, update_data, expected_version, db):
    result = await db.execute(
        update(Item)
        .where(Item.id == item_id, Item.version == expected_version)
        .values(**update_data, version=Item.version + 1)
    )
    if result.rowcount == 0:
        raise HTTPException(409, "Item was modified by another session")
    return await get_item(item_id, db)

2. Audit Log (JSONB Snapshots)

Every write operation logs a before/after JSONB snapshot to the activity_log table. If data ever looks wrong, you can trace exactly what changed, when, and what the previous values were.

# Audit entry example:
{
  "entity_type": "item",
  "entity_id": "uuid-...",
  "action": "update",
  "before_snapshot": {"name": "Suction Machine", "quantity": 2, "version": 3},
  "after_snapshot":  {"name": "Suction Machine", "quantity": 5, "version": 4},
  "created_at": "2026-04-19T14:30:00Z"
}

3. Database-Level CHECK Constraint

-- Migration 0004:
ALTER TABLE items ADD CONSTRAINT check_quantity_non_negative
    CHECK (quantity >= 0);

Even if application code has a bug that tries to set quantity to -1, PostgreSQL will reject it. Defense in depth.

4. Atomic Apply-Stock Endpoint

When a purchase order is received and stock needs updating across multiple items, the POST /orders/{id}/apply-stock endpoint wraps all item updates in a single database transaction. Either all items update successfully, or none do. No partial stock updates.

Chapter 8

When to Use Each Architecture

Neither architecture is universally better. The choice depends on your specific constraints.

Use Offline-First When

Users are genuinely offline for extended periods: field workers, rural areas, subway commuters, airplane mode

Data conflicts are rare or easily resolvable (e.g., personal notes, local preferences)

Eventual consistency is acceptable (data can be temporarily wrong)

You have resources to implement CRDTs or operational transforms

Examples: Google Docs (OT), Notion (offline blocks), Todoist (task sync), git (distributed VCS)

Use Server-First When

Users have reliable connectivity (home WiFi, office, mobile data)

Data accuracy is critical (financial, medical, legal)

Multiple users may edit the same data (collaboration)

Audit trail and regulatory compliance required

You're a solo developer or small team (simpler to maintain)

Examples: Banking apps, medical records, e-commerce inventory, SaaS dashboards, CareKosh

The Middle Ground: Stale-While-Revalidate

TanStack Query supports cache persistence via @tanstack/query-async-storage-persister. This lets you show cached data instantly on app launch while fetching fresh data in the background. The cache is read-only for display — mutations always go to the server first. This gives "instant load" UX without the data corruption risks of offline-first. It's the approach CareKosh plans to adopt in v1.1.

Chapter 9

Key Concepts for Interviews

Optimistic Concurrency Control (OCC)

A strategy where you don't lock rows during reads. Instead, each row has a version number. When you write, you include the version you read. If someone else modified the row in between (version changed), the write fails with a conflict error. The client can then re-read, re-apply their change, and retry. Used by: DynamoDB (conditional writes), Kubernetes (resourceVersion), HTTP (ETag/If-Match), CareKosh (version column).

Pessimistic Concurrency Control (PCC)

Lock the row when you read it. Nobody else can modify it until you release the lock. Prevents conflicts entirely but reduces throughput. Used by: traditional banking systems, database SELECT ... FOR UPDATE. CareKosh doesn't use this — OCC is sufficient for a single-user-per-account app.

CQRS (Command Query Responsibility Segregation)

Separate read models from write models. Reads can hit caches or denormalized views; writes go through validation and business logic. TanStack Query naturally implements this: useQuery = read side, useMutation = write side. CareKosh follows this pattern implicitly.

Event Sourcing vs Audit Log

Event sourcing stores every change as an immutable event and rebuilds state by replaying events. CareKosh uses a simpler approach: JSONB before/after snapshots in an audit log. This gives traceability without the complexity of event replay. It's not event sourcing — it's change capture.

Stale-While-Revalidate

A caching strategy where stale cached data is served immediately while fresh data is fetched in the background. When the fresh data arrives, it replaces the stale cache. TanStack Query does this by default when cached data exists. The key: stale data is only used for display, never for server mutations.

12-Factor App

A methodology for building SaaS apps. CareKosh follows Factor III (Config) rigorously: the same Docker image runs in development, staging, and production with only environment variables changing (DATABASE_URL, ENVIRONMENT, SECRET_KEY). Zero code changes needed to switch environments.

Idempotency

An operation is idempotent if calling it multiple times has the same effect as calling it once. CareKosh's OCC provides natural idempotency for updates: if you retry a failed update with the same version, it will either succeed (first try) or fail with 409 (already updated) — never corrupt data by applying the change twice.

Chapter 10

Interview Questions & Answers

These are real questions you might face when discussing CareKosh's architecture in an SDE interview. Each answer references the actual codebase.

Tell me about a time you had to make a significant architectural change. What drove the decision?

Behavioral

CareKosh originally used offline-first architecture with Zustand persist middleware syncing to PostgreSQL. A user logged out and back in, and the Zustand rehydration loaded stale data with zero quantities from AsyncStorage. The sync engine then pushed these zeros to the server, overwriting correct data for 31 of 33 medical inventory items. I confirmed this by querying the server directly — all items had identical timestamps proving a bulk overwrite.

I migrated to server-first architecture using TanStack Query. The Zustand store went from 1,100 lines to 61 (UI state only). The 611-line sync engine was deleted entirely. I added Optimistic Concurrency Control with a version column and JSONB audit logging. The entire class of rehydration-race bugs was eliminated because local state is never persisted or pushed to the server — the server is always authoritative.

Key result: Zero data corruption incidents since the migration. Codebase reduced by ~1,650 lines of state management code.

How do you handle concurrent modifications to the same resource?

Concept — OCC

Every mutable entity (items, categories, orders) has a version integer column. When the mobile app reads an item (e.g., version=3), it includes that version in the update request. The backend's UPDATE query has WHERE id = :id AND version = 3 and increments to version=4. If another client already updated it to version=4, the WHERE clause matches zero rows and we return 409 Conflict. The client then re-fetches, shows the updated data, and lets the user retry.

This is Optimistic Concurrency Control — we don't lock anything during reads, we just detect conflicts at write time. It's the same pattern DynamoDB uses for conditional writes and Kubernetes uses for resourceVersion.

Why did you choose server-first over offline-first? What are you giving up?

Tradeoff Analysis

I'm giving up the ability to create and edit inventory items without internet connectivity. In return, I get guaranteed data integrity, a simpler codebase (1,650 fewer lines), and audit traceability for every change.

This tradeoff is correct for CareKosh's domain: the target users are family caregivers managing home ICU setups — they're in homes with WiFi, not in remote areas. The cost of incorrect medical inventory data (thinking you have zero oxygen cylinders when you actually have five) is far higher than the cost of needing WiFi to update stock levels.

For the "instant load" UX benefit of offline-first, I plan to add TanStack Query cache persistence — it shows cached data immediately on app launch while fetching fresh data in the background. The cache is read-only for display and never pushed to the server, so the data corruption risk is zero.

What is a rehydration race condition? How did you solve it?

Concept — Race Condition

When a React Native app with Zustand persist middleware starts up, it asynchronously loads saved state from AsyncStorage. This "rehydration" restores the previous session's data — including potentially stale inventory quantities. The problem: Zustand's persist fires a state mutation when rehydration completes, and if you have a sync engine that triggers on mutations, it will push the stale rehydrated data to the server before the actual server fetch completes.

The timing: rehydration completes at ~53ms, sync fires at ~2054ms (debounced 2s), server fetch response arrives at ~2200ms. The sync pushes zeros to the server 150ms before the server could tell the app the real values.

I solved it by eliminating the architecture that made it possible: no persist middleware for inventory data, no sync engine, no local-to-server push. TanStack Query fetches from the server on mount, caches in memory, and never persists inventory data to AsyncStorage. The race condition cannot occur because there's nothing to rehydrate.

How do you ensure data consistency in a mobile app where the network is unreliable?

System Design

Four layers: (1) Server-side OCC prevents concurrent overwrites — version column + 409 Conflict. (2) JSONB audit log captures before/after snapshots for every write, enabling forensic analysis. (3) Database-level CHECK constraint (quantity >= 0) prevents invalid states even if application code has bugs. (4) Atomic transactions for multi-item operations (apply-stock endpoint wraps all item updates in a single transaction — all succeed or all rollback).

For network unreliability: TanStack Query retries failed requests automatically (configurable), shows stale cached data while retrying, and surfaces errors to the user explicitly rather than silently corrupting data. The app shows a clear "Unable to connect — server may be starting up" message with auto-retry on cold starts.

What's the difference between TanStack Query and Zustand? Why use both?

Concept — State Management

TanStack Query manages server state — data that lives on the server and is fetched via HTTP. It handles caching, background refetching, stale-while-revalidate, retry logic, and cache invalidation. In CareKosh: items, categories, orders, activity log.

Zustand manages client state — data that exists only on the device and has no server representation. In CareKosh: current theme (dark/light), selected tab, search query text, whether the profile menu is open.

Using both is the industry-standard pattern. They solve different problems. Putting server data in Zustand (like v1 did) forces you to build caching, invalidation, and sync logic yourself — poorly reimplementing what TanStack Query does out of the box.

How did you handle the migration without data loss?

Behavioral

The migration (PR #4) was executed across multiple sessions with strict principles: change only the data flow plumbing, not the features. I ran the Alembic migration (0004_version_audit_log) on staging first, verified via Neon console that the version and audit columns were added correctly, then tested all CRUD operations via curl against staging before merging.

A frustration during the process: the initial migration accidentally removed working features (seed data restore, backup/export, Build Inventory prompts). I explicitly established the principle that "architectural migrations must preserve existing functionality" and spent PRs #5–#9 restoring every feature that was lost. Lesson learned: migration PRs should have a pre/post feature checklist.

Explain the 12-Factor App principle and how your project follows it.

Concept — 12-Factor

The 12-Factor App is a methodology for building SaaS apps that are portable, scalable, and deployable. CareKosh follows several factors rigorously:

Factor III (Config): All configuration is in environment variables (DATABASE_URL, SECRET_KEY, ENVIRONMENT, REQUIRE_EMAIL_VERIFICATION). The same Docker image runs on development (localhost), staging (Render Singapore), and production (Render Singapore) — zero code changes. When we migrated from Railway to Render, zero Python code changed because config was already externalized via pydantic-settings.

Factor IV (Backing Services): PostgreSQL (Neon), email (Brevo) are attached resources configured via URLs. Swapping Neon for local Docker PostgreSQL is just changing DATABASE_URL.

Factor X (Dev/Prod Parity): Same Docker image, same Alembic migrations, same FastAPI code in all environments. Only env vars differ.

What would you do differently if you were starting this project from scratch?

Behavioral

Three things: (1) Start with server-first from day one. The offline-first architecture was premature complexity for a use case where users have WiFi. It added 1,711 lines of sync code that ultimately caused data corruption. (2) Write tests before features. The test gap is the single largest portfolio weakness — 40 of 53 tests are currently failing because they were written for the pre-migration architecture. Interviewers check tests first. (3) Set up Sentry on day one. We had no observability when the data corruption happened — I had to diagnose it by manually querying the database with curl commands.

How do you handle account deletion in compliance with data protection regulations?

System Design — GDPR/DPDP

Two-step email-confirmed deletion flow: (1) User taps "Delete Account" → app calls DELETE /auth/me → backend generates a SHA-256 hashed token, stores it with 24-hour expiry, sends confirmation email via Brevo with a link containing the raw token. (2) User clicks the email link → GET /auth/confirm-delete/{token} → backend hashes the incoming token, queries DB for a matching hash that hasn't expired, cascade-deletes the user and ALL associated data (categories, items, orders, activity log, refresh tokens), returns an HTML confirmation page.

This prevents accidental deletion (requires email confirmation), gives a 24-hour window to cancel (POST /auth/cancel-delete), and ensures complete data removal (PostgreSQL CASCADE). The confirmation page is a server-rendered HTML response — no mobile app needed to complete deletion, which satisfies Play Store requirements for account deletion being accessible outside the app.
