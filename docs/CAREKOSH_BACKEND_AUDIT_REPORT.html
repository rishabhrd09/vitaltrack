01 · Executive summary

The one paragraph a non-engineer can act on

This backend is correct and safe for your launch scale — ~100 concurrent and ~500 total users is modest for this design. The security-critical paths are well built: every data query is scoped to the logged-in user (no cross-user leaks — verified in code and covered by tests), the order → apply-stock flow is genuinely concurrency-safe, refresh tokens rotate, and secrets/tokens are handled properly. The deployment setup (GitHub Actions CI → Render Docker → Neon, with an EAS mobile pipeline) is already a mature, modern solo-dev stack.

You do not need any big scaling work — no Redis, no caching layer, no microservices, no Kubernetes, no service-layer rewrite. The scary numbers in this report are all worst-case saturation figures that only bite at 1,000–10,000 concurrent users, far beyond your needs.

Before the first public deploy, do a focused ~1-day hardening branch (§03): a handful of small correctness/validation fixes that can bite even with 5 users, add error tracking so you aren't blind in production, and — most important in the real world — move off Render's free tier so cold starts don't make the app feel broken to your first users and the Play reviewer (§04).

This revision — reconciled with two independent audits & the test suite

Findings were cross-checked against a second independent review and re-verified against the code and tests. New issue added & verified: negative order-item quantities (VAL-1). Two races folded in: refresh rotation (SEC-3) and order-status (CONC-3). Correction: an earlier draft claimed items/categories had no IDOR tests — that was wrong; test_items.py:404 and test_categories.py:233 both exist. The genuine test gap is concurrency tests (see §09). All three passes reached the same launch decision.

02 · Launch decision for your scale

Target: a maximum of ~100 users active at once and ~400–500 total registrations.

🚀 Go / no-go

Ship it — after a ~1-day pre-launch hardening branch.

Scale is not the blocker. "100 users at a time" is not 100 simultaneous requests — a medical-inventory app is bursty and mostly idle, so realistic peak in-flight requests is more like 10–30 at once. Every load-related risk below has large headroom at that level.

1Block negative / zero order-item quantitiesapp/schemas/order.py:27S
Today quantity: int = 1 has no lower bound, so a negative quantity flows into quantity = Item.quantity + order_item.quantity on apply and reduces stock. Self-scoped (your own inventory only), but it breaks the "orders only add stock" invariant and can make total_units negative.
# app/schemas/order.py — OrderItemBase
- quantity: int = 1
+ quantity: int = Field(1, gt=0, le=999999)

# (optional) belt-and-braces DB guard via a new Alembic migration:
op.create_check_constraint("chk_order_items_quantity_positive", "order_items", "quantity > 0")
2Make item stock/version updates atomic (real optimistic lock)app/api/v1/items.py:373, 531S→M
The version check runs in Python and the flushed UPDATE has no version in its WHERE — so two concurrent edits both pass and the second silently overwrites the first. (A sequential stale submit is already caught; the concurrent race is not.) Push the compare-and-swap into SQL. Apply to both update_stock and update_item.
# app/api/v1/items.py — update_stock (same shape for update_item's field set)
res = await db.execute(
    update(Item)
    .where(Item.id == item_id,
           Item.user_id == current_user.id,
           Item.version == data.version)          # <-- the guard the DB enforces
    .values(quantity=data.quantity, version=Item.version + 1)
    .returning(Item.version)
)
if res.scalar_one_or_none() is None:               # 0 rows = stale version or not found
    return JSONResponse(status_code=409, content={"error": "Version conflict", ...})
3Make refresh-token rotation a single-use atomic claimapp/api/v1/auth.py:771–814S
Rotation reads the token, checks is_revoked in Python, then revokes + mints a new one. Two simultaneous refreshes of one token can both pass and both mint valid tokens. Replace the read-check with an atomic conditional claim.
# app/api/v1/auth.py — refresh_token, replacing the select + Python is_revoked check
claim = await db.execute(
    update(RefreshToken)
    .where(RefreshToken.jti == jti,
           RefreshToken.user_id == user_id,
           RefreshToken.is_revoked.is_(False))
    .values(is_revoked=True)
    .returning(RefreshToken.id)
)
if claim.scalar_one_or_none() is None:
    raise HTTPException(status_code=401, detail="Token has been revoked")
# ... then mint + insert the new refresh token as before
4Make order-status transitions conditional (no last-writer-wins)app/api/v1/orders.py:352–402S
Optional but cheap and the same pattern. Two concurrent transitions (e.g. pending→received and pending→declined) can both validate and the last commit wins, with both activity logs written. Guard the UPDATE on the old status.
# app/api/v1/orders.py — update_order_status, after validating the transition
res = await db.execute(
    update(Order)
    .where(Order.id == order.id,
           Order.user_id == current_user.id,
           Order.status == old_status)             # only transition from the state we read
    .values(status=new_status, **timestamps)
    .returning(Order.id)
)
if res.scalar_one_or_none() is None:
    raise HTTPException(status_code=409, detail="Order changed, please refresh and retry")

6Add Sentry + a request-ID middlewareapp/main.py · requirements.txtS
Highest operability ROI for a solo dev. Without it, your first production bug is invisible — a user says "it's broken" and you have only unstructured stdout. Sentry's free tier is enough.
# requirements.txt:  sentry-sdk[fastapi]==2.*
# app/main.py
import sentry_sdk
if settings.ENVIRONMENT == "production" and settings.SENTRY_DSN:
    sentry_sdk.init(dsn=settings.SENTRY_DSN, environment=settings.ENVIRONMENT,
                    traces_sample_rate=0.1)

@app.middleware("http")
async def request_id(request, call_next):
    rid = request.headers.get("X-Request-ID") or uuid4().hex
    response = await call_next(request)
    response.headers["X-Request-ID"] = rid
    return response

Correction: an earlier draft of this report recommended pointing Render at /health — that is wrong for this stack and has been reverted. Two deployment facts worth knowing

1. Migrations run inside the container entrypoint on every deploy. Fine on a single free instance (one entrypoint runs). The moment you run 2+ instances, every instance races alembic upgrade head on boot → move migrations to a one-off release/pre-deploy step (see §11). Step 7 (removing || echo) already makes a failed migration abort the deploy.

2. The production AAB build is disabled in CI (if: false, "while we iterate on the server-first refactor"), so the Play Store artifact is currently produced manually via eas build --profile production. Re-enabling it (or building manually) is part of the launch path — not a backend blocker.

05 · Verdict — order → apply-stock concurrency

✅ Concurrency-safety verdict

The apply-stock path is SAFE — proven, with one scope caveat.

POST /orders/{id}/apply is safe against N concurrent applies to the same item and against duplicate/retried requests, on two independent grounds. (There is even a passing concurrent test for it — test_orders.py:424.)

① Double-apply is blocked by an atomic compare-and-swap claim

vitaltrack-backend/app/api/v1/orders.py:479–498

update(Order)
  .where(Order.id == order.id, Order.user_id == current_user.id,
         Order.status == OrderStatus.RECEIVED)        # <-- the guard
  .values(status=OrderStatus.STOCK_UPDATED, applied_at=now)
  .returning(Order.id)
# losers match 0 rows → claimed_order_id is None → HTTP 400, no stock touched

Under Read Committed, two concurrent applies of the same order serialize on the order row lock. The first flips RECEIVED → STOCK_UPDATED; the loser re-evaluates the WHERE against the committed row, matches zero rows, and 400s before touching inventory. STOCK_UPDATED is terminal, so this is effectively idempotent.

② The stock increment is a DB-atomic read-modify-write

vitaltrack-backend/app/api/v1/orders.py:503–514

update(Item).where(Item.id == order_item.item_id, Item.user_id == current_user.id)
    .values(quantity=Item.quantity + order_item.quantity,   # evaluated server-side
            version=Item.version + 1)
    .returning(Item.id, Item.name, Item.quantity)

Item.quantity + order_item.quantity is computed inside Postgres under a row lock — re-reads the latest committed value, adds, writes. Two different orders touching one item serialize on that lock; no lost update. All writes ride the single get_db transaction and commit together — no partial commit.

Scope caveat — this is where the real bugs live

The order apply path is safe, but the direct stock-edit endpoints are not (CONC-1), and order-item quantity isn't validated (VAL-1). Both are fixed in §03, steps 1–2.

08 · Part A — System-design findings

Each finding states current behaviour (with evidence), the failure mode under load, the blast radius, and a concrete fix sized S / M / L.

A1 · Concurrency & data consistency

VAL-1Negative order-item quantity can reduce stock on applyMediumS
Evidence
schemas/order.py:27 — quantity: int = 1 (no gt=0); orders.py:292 stores it; orders.py:510 applies it 
quantity: int = 1                                  # OrderItemBase — unbounded sign
# apply:  quantity = Item.quantity + order_item.quantity   # negative → stock goes DOWN
Impact
A crafted or buggy-client order with a negative quantity reduces inventory through receive→apply (as long as the result stays ≥0, so the CHECK constraint doesn't catch it), and makes total_units negative. Self-scoped — orders only reference the caller's own items (validated at orders.py:115–140), so no cross-tenant impact and a user can already set their own stock via the stock endpoint. Genuine input-validation gap that breaks the "orders only add stock" invariant. Surfaced by the second audit (F04) and verified here.
Fix
One line: quantity: int = Field(1, gt=0, le=999999), optionally backed by a DB CHECK (quantity > 0) on order_items. §03 step 1.
CONC-1Direct stock-edit optimistic lock is a Python TOCTOU — lost updatesHighS→M
Evidence
items.py:530–544 (update_stock); same pattern items.py:373–460 (update_item) 
if item.version != data.version:          # check, in Python
    return JSONResponse(status_code=409, ...)
item.quantity = data.quantity             # act
item.version += 1
await db.commit()    # UPDATE items SET quantity=?, version=? WHERE id=?  (no version predicate)
Verified: no __mapper_args__ = {"version_id_col": ...} and no with_for_update anywhere — the flushed UPDATE carries no version in its WHERE.
Impact
Two requests with the same data.version=N both pass the != check, both write version=N+1; the second silently overwrites the first. A sequential stale submit is correctly rejected (test_items.py:146), which can give false confidence — but the concurrent race is real. Blast radius is one user's own data, but this is an offline-first app (local_id/version exist for sync) — two devices or a replayed sync queue is a realistic trigger, and for medical stock a vanished decrement matters.
Fix
Push the compare-and-swap into the statement (add Item.version == data.version to the UPDATE WHERE, treat 0 rows as the 409) — or set version_id_col on the model. §03 step 2. Add the concurrent test (§03 step 5).

orders.py:352 reads order, :367 derives old status, :375 order.status = new_status, :402 commit — no conditional WHERE

Concurrent pending→received and pending→declined can both validate; last commit wins and both activity logs are written (one "lies"). Single-user, low probability. Raised by the second audit (F05).

DB-1Every authenticated request eager-loads the entire accountHighS
Evidence
user.py:116–145 — all five relationships; deps.py:55–57 
categories / items / orders / activity_logs / refresh_tokens = relationship(..., lazy="selectin")
# get_current_user: select(User).where(User.id == user_id)  → fires ~6 extra SELECTs
Impact
One select(User) triggers selectin loads of all items, categories, orders (and Order.items is selectin too → order_items), activity logs and refresh tokens — on every authenticated call, even GET /items. activity_logs grows unbounded, so per-request cost rises over the account's lifetime. Dominant latency cost; amplifies DB-2.
Fix
Surgical (pre-launch): in get_current_user add .options(noload("*")) (or raiseload("*")) — the auth user is used for scalar fields only, so this kills the cascade with zero risk. Broader (later, test-gated): drop lazy="selectin" on the model + add explicit selectinload(...) where needed (a blind model change risks MissingGreenlet elsewhere). §03 step 8.
DB-2Connection math has no headroom on Neon freeHighS
Evidence
database.py:77–85 (pool_size=5, max_overflow=10) · Dockerfile CMD --workers 4
Impact
Each worker = its own pool. Worst case 15 × 4 = 60 connections per instance. Neon free (0.25 CU) allows ~104 max / ~97 app-available. A rolling deploy briefly runs old+new (~120 → "too many connections"); two instances steady = 120, over the ceiling. At your scale you'll use ~15–30 — a "before scaling" risk; trim workers→2 now.
Fix
Pre-launch: --workers 2. Before scaling: Neon pooled (-pooler) endpoint with statement_cache_size=0 (DB-3), or smaller pools. §11.
DB-3 · DB-4asyncpg + PgBouncer caveat, and no pool_recycleMediumS
Evidence
database.py:84 (connect_args); config.py:55 strips Neon ?sslmode query params; alembic/env.py reuses the same URL
Impact
When you adopt the pooled endpoint (DB-2 fix), asyncpg's prepared-statement cache breaks under PgBouncer transaction pooling. Alembic would then also run through the pooler — migrations should use a direct URL. No pool_recycle means Neon idle drops rely solely on pre_ping.
Fix
Set statement_cache_size=0; add pool_recycle=300; use DATABASE_URL (pooler) for the app and a separate DATABASE_DIRECT_URL for Alembic. (Second audit's F08 — agreed.)
DB-5 · DB-6Stats count in Python; missing composite indexesMediumM
Evidence
items.py:128–139 loads all items then sum(1 for …); orders status unindexed (migration:114); activity orders by created_at
Impact
Stats/feed endpoints scale with total rows per user; fine at your size, degrades as inventories grow. Compounded by DB-1.
Fix
SQL conditional aggregates (count() filter); composite indexes (user_id,status,exported_at), (user_id,created_at), (user_id,is_active). DB-7 (Low): collapse the per-item SELECT in apply into one IN (...).

A3–A5 · Statelessness, blocking work & rate limiting

RATE-1Rate limiter is per-process, in-memory, per-IP, no lockoutHighM
Evidence
rate_limiter.py:38–42 — no storage_uri; trusts CF/XFF headers; auth.py:185 5/minute login
Impact
(1) Not global — each worker keeps its own bucket, so 5/min is ~20/min per IP per box and resets on spin-down. (2) Per-IP, no account lockout — IP rotation defeats it; no failed-login counter. (3) Spoofable — direct-to-origin requests forge the client IP. Otherwise the app is stateless (JWT + Postgres refresh state).
Fix
Before scaling, not pre-launch. Redis-backed slowapi (Upstash) for a global limit; per-account failed-login backoff; restrict trusted proxy headers to Cloudflare ranges. §11.
BLOCK-1Argon2 hashing blocks the event loopHighS
Evidence
security.py:27–34; called directly at auth.py:120/715/1208 (hash) and 217/1202 (verify)
Impact
Argon2 is CPU- and memory-bound (~50–150ms + tens of MB/hash). On Render free (0.1 CPU / 512 MB) each hash is slow; while it runs the worker's event loop is frozen. At your login volume this is minor — but the fix is small.
Fix
await anyio.to_thread.run_sync(...) / threadpool. §03 step 9.
Confirmed good — email is NOT in the request path

Register/reset enqueue email via background_tasks.add_task(...) (auth.py:161/452/519/730/1012) and the Brevo HTTP call has timeout=10.0. Request latency is decoupled from Brevo, and a Brevo outage doesn't 500 the registration.

A6 · Resilience & failure modes

RESIL-1No database statement timeoutHighS
Evidence
database.py:77–85 — no statement_timeout / asyncpg command_timeout
Impact
A slow/hung Neon query blocks its request indefinitely and holds its pooled connection. A handful of stuck queries exhausts the 15-conn pool; new requests wait pool_timeout=30s then 500. One slow query → cascading outage.
Fix
connect_args={"server_settings": {"statement_timeout": "8000"}} (8s). §03 step 8.
RESIL-2Brevo has timeouts but no retry / persistenceMediumM
Evidence
email.py:102–118 — single attempt, returns False on failure; runs in an in-process BackgroundTask
Impact
A transient Brevo blip or a worker restart mid-task loses the verification email with no retry — the user is stuck unverified.
Fix
A small retry inside the task, or an email_outbox table + resweep. Defer. Confirmed good: pool_pre_ping=True; dispose_engine() on shutdown.

A7 · Multi-tenant security

SEC-5No IDOR — every resource query is user-scoped (code + tests)Passverified
Evidence
Every CRUD handler filters on user_id == current_user.id (items.py:59/226/…; categories.py:43/114/…; orders.py:163/213/…; activity.py:34). And it's test-covered: stranger-access returns 404 in test_items.py:404, test_categories.py:233, and order scoping in test_orders.py; stats scoping in test_items.py:308.
Impact
User A guessing user B's id gets 404. Schemas don't expose user_id/id/is_verified/is_superuser → no mass-assignment. All queries parameterised → no SQLi. Tokens use secrets.token_urlsafe(32) stored as SHA-256, compared with compare_digest; JWT algorithm pinned. Strongest part of the codebase.
SEC-1Access tokens are not revocableMediumM
Evidence
deps.py:47–58 trusts any unexpired access JWT; logout/reset/change revoke refresh only
Impact
A stolen or post-logout access token works for ≤30 min. Standard stateless-JWT tradeoff, self-healing, no privilege escalation — Medium.
Fix
Defer. Add a token_version / password_changed_at claim and check it in get_current_user.
SEC-2 · SEC-3Token-scan lookups; refresh rotation not atomic & no reuse detectionMediumS
Evidence
SEC-2: auth.py:300–312 loads all unexpired-token users and loops in Python. SEC-3: auth.py:771–814 read-check-write rotation; a revoked token just 401s
Impact
SEC-2: O(N) over pending users, on the event loop (the deletion flow already does the indexed lookup). SEC-3: two simultaneous refreshes of one token can both mint valid tokens; and presenting a revoked token doesn't revoke the family (reuse detection). SEC-3's atomicity was raised by the second audit (F03).
Fix
SEC-3 atomic claim is in §03 step 3 (pre-launch). SEC-2 indexed token lookup + family revocation: defer.

09 · Part B — SOLID & clean architecture

Clean layout, shallow layering. Models/schemas import nothing from api/ or services/; no circular imports. But there is no service/repository layer — services/ contains only audit.py — so routers do everything. "Shallow but not tangled": the risk is duplication and untestable logic, not spaghetti. None of this blocks your launch — it's maintainability for later.

Testability — stronger than first reported

The one real gap is concurrency tests: the only asyncio.gather test is order-apply — there is no concurrent test for the item-version race (CONC-1), refresh-rotation race (SEC-3), order-status race (CONC-3), or the create_order ID race. §03 step 5 closes exactly these.

Bottom line

The dangerous-by-default things — multi-tenant isolation, the order→apply-stock race, token handling — were done right, and the deployment stack already matches modern solo-dev practice. The exposure is operational scale (whole-account eager load + connection ceiling + no statement timeout) and a few cheap correctness gaps (CONC-1, VAL-1, SEC-3). The §03 list is ~1 focused day and removes the launch-relevant risks. For ~100 concurrent / ~500 total users: do §03, move off free tier, then ship.
