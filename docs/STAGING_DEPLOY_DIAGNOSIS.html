Why staging looks frozen — and why it's actually correct.

A plain-language walkthrough of how the two Render services deploy, why they diverged in the event log after April 20, and — most importantly — what it means for the mobile testing done in the days since.

Repo: rishabhrd09/vitaltrack Staging last deployed: Apr 20, 3:18 PM · PR #17 · de39437 Production last deployed: Apr 22, 5:16 AM · PR #31 · 7a9c652 Main HEAD: 7a9c652
Bottom line first

Nothing is broken. Both Render services are configured identically — same repo, same branch main, same Root Directory vitaltrack-backend, same Auto-Deploy "On Commit." The Root Directory setting tells Render to only auto-deploy when files inside that directory change. PRs #18 through #31 are frontend-only, so Auto-Deploy on both services correctly skipped them. Staging sat still. Production would have sat still too — except CI also curls production's Deploy Hook on every main push, which ignores the Root Directory filter and force-redeploys it anyway.

Staging is running the latest backend code. git log de39437..origin/main -- vitaltrack-backend/ returns zero rows, so staging's "frozen" commit and production's current commit contain byte-identical backend code. On-device testing against staging was valid.

There is no defect here — only an asymmetry. Production force-redeploys backend Docker images even when the backend didn't change (wasteful but harmless). Staging doesn't. If that asymmetry ever matters (next backend change lands, you want staging to pick it up automatically), the mitigation is either trimming production's over-eager CI hook or adding a symmetric hook for staging. Both are tracked in Section 05.

01 — CONTEXT

Two backends, one codebase.

CareKosh runs two copies of the FastAPI backend, both served from the same GitHub repo at vitaltrack-backend/:

Production
vitaltrack-api.onrender.com
Purpose: what real users eventually hit
Database: production Neon Postgres (real user data)
Mobile source: production AAB builds, Play Store releases
Deploy trigger: Deploy Hook called by CI on every main push — and Render auto-deploy
Staging
vitaltrack-api-staging.onrender.com
Purpose: what Preview APKs talk to during testing
Database: staging Neon Postgres (test data only)
Mobile source: Preview APKs built from PRs via the build-apk label
Deploy trigger: Render auto-deploy only — no CI hook

The mobile app has its backend URL baked in at compile time via EXPO_PUBLIC_API_URL in eas.json. Preview APKs always point at staging. Production AABs always point at production. So when you install a Preview APK and hit Sign In, the request goes to staging. Whether staging is up-to-date determines whether your testing is valid. Whether production is up-to-date determines what your real (eventual) users hit.

Both services should run identical vitaltrack-backend/ code from main. If they diverge, any code path that changed between them will behave differently on the two surfaces.

02 — HOW DEPLOYS WORK

Two ways to tell Render "new code is ready."

Render is a hosting platform. It promises to rebuild + redeploy whenever new commits land. But it has two separate mechanisms for noticing commits, and they have different failure modes.

Mechanism A — Auto-Deploy (Render's built-in)

How Render's built-in auto-deploy works
01
The service is configured: "watch branch main, auto-deploy on new commits."
02
Render registers a webhook on the GitHub repo — a URL GitHub will POST to on every push.
03
Someone merges a PR into main. GitHub POSTs to Render's webhook.
04
Render pulls the code, rebuilds the Docker image, redeploys. Event: "New commit via Auto-Deploy".

This is how staging is set up. Simple. No glue code. Entirely dependent on one healthy GitHub → Render webhook.

Mechanism B — Deploy Hook (CI-triggered, explicit)

How the explicit Deploy Hook works
01
Render generates a secret URL (the "Deploy Hook"). POSTing to it triggers a deploy.
02
The URL is stored as a GitHub repo secret: RENDER_DEPLOY_HOOK.
03
The deploy-backend job in .github/workflows/ci.yml runs on every main push after tests pass. It runs curl -s "${{ secrets.RENDER_DEPLOY_HOOK }}".
04
Render receives the hook POST, pulls code, rebuilds, redeploys. Event: "Triggered via Deploy Hook".

This is how production is set up. The single RENDER_DEPLOY_HOOK secret in GitHub points to the production service — confirmed by the deploy summary step logging URL: https://vitaltrack-api.onrender.com. Staging has no equivalent entry in CI.

Key difference: Auto-Deploy is "Render listens for GitHub pushes" (via webhook subscription). Deploy Hook is "CI actively tells Render" (via curl). If the GitHub → Render webhook path breaks, Auto-Deploy silently stops. Deploy Hook is unaffected — it's a completely separate channel that goes from GitHub Actions straight to Render.

03 — WHAT ACTUALLY HAPPENED

Not a break. A Root Directory filter doing its job.

The real cause came from the staging service's Settings → Build & Deploy page. Both services have Root Directory set to vitaltrack-backend. Render's own description of that field (visible on the dashboard) says:

Quoted from Render's Root Directory help text
◆
"If set, Render runs commands from this directory instead of the repository root. Additionally, code changes outside of this directory do not trigger an auto-deploy. Most commonly used with a monorepo."

That's exactly what happened. Reading the timeline with that in mind:

APRIL 20 · 3:18 PM
Last staging Auto-Deploy — PR #17 (backend-touching)
Deploy live for commit de39437. Label: "New commit via Auto-Deploy." PR #17 touched vitaltrack-backend/, so it matched the Root Directory filter and triggered a build.
APRIL 20 · 6:57 PM
PR #18 merged — frontend-only — correctly skipped
PR #18 sits inside vitaltrack-mobile/ only. Render's Auto-Deploy on both services correctly ignored it. Production's event log shows "Triggered via Deploy Hook" — CI force-redeployed production's backend even though the backend didn't change. Staging's event log simply shows nothing, because no CI hook is configured for staging.
APRIL 20 – APRIL 22
PRs #19–#31 — same pattern
64 commits merged across 14 PRs. Every one is frontend-only. Neither service's Auto-Deploy triggered. Production shows 14 "Triggered via Deploy Hook" events (CI-driven, each rebuilding backend code that hadn't changed). Staging shows nothing, which is the correct behavior for its configuration.

Evidence that settles the question

Both services' dashboards (Settings → Build) show Auto-Deploy = "On Commit" right now. Nobody toggled it off. Nobody broke a webhook. The webhook is firing on every push; Render is correctly matching those pushes against the Root Directory filter; the filter is correctly rejecting non-backend commits for both services. Prod only appears "up to date" because a second, redundant trigger (CI's Deploy Hook) fires unconditionally on every push.

What this rules out

✗ Auto-Deploy toggled off. Both services show "On Commit." Previous hypothesis was wrong.

✗ GitHub → Render webhook broken. If the webhook were broken, production's Auto-Deploy wouldn't fire either (and it didn't, for these commits — CI's hook was the only trigger). But the webhook was evaluated by Render: it just didn't match the Root Directory filter.

✗ Branch drift. Both services point at main. Verified in the screenshots.

04 — THE CRITICAL QUESTION

Have you been testing against stale code?

This is what actually matters. Two days of silent drift could have meant every test session debugging against the wrong backend. The answer depends on one query: what changed in the backend between PR #17 and PR #31?

(no output — zero commits touched vitaltrack-backend/)

$ git diff --shortstat de39437..origin/main -- vitaltrack-backend/

(no output — zero lines changed in backend)

$ git log --oneline de39437..origin/main | wc -l
64

# 64 commits on main since the last staging deploy.
# Zero of them touched the backend. All inside vitaltrack-mobile/.
Verdict
Your testing has been valid — and staging's behavior is correct by design.

Between April 20 (last staging deploy, de39437) and April 22 (current main, 7a9c652), not a single backend file changed. Every one of the 64 commits in that window sits inside vitaltrack-mobile/: branding, icons, UX polish, PDF restyling, profile-sheet gesture, register screen changes.

Staging's Auto-Deploy correctly skipped all of them because of the Root Directory filter. Functionally, staging at de39437 runs the byte-identical backend code as production at 7a9c652. Any request your Preview APK has made to staging got the same response it would have gotten from production.

Specifically the client-side changes in this PR window — the 401 interceptor in services/api.ts, the register-without-tokens change in useAuthStore.ts, the email-required gate — all live in the mobile app's code. They change how the app treats API responses, not what the API itself returns. The backend change that would truly matter here (the /register endpoint dropping the token fields from its response) hasn't shipped yet and would need a backend commit to take effect.

One thing to keep in mind
Staging only rebuilds when backend code changes.

Because both services have Root Directory = vitaltrack-backend, staging's Auto-Deploy only fires on commits that touch backend files. The next backend PR you merge will correctly trigger a staging rebuild automatically — no action needed. But if you ever push a mobile-only change that depends on a backend state (e.g. testing against a pre-populated staging DB that got reset), you may want to bump staging manually via "Manual Deploy" even though nothing in vitaltrack-backend/ changed. This is a known quirk of the Root Directory filter, not a flaw.

Section 05 lists three optional structural tweaks. None are urgent.

05 — WHAT (IF ANYTHING) TO CHANGE

Nothing urgent. Three optional improvements.

Since nothing is broken, there's no mandatory fix. The question is whether the current asymmetry ever bothers you. Three candidate changes, in ascending order of effort and value.

i
Do nothing (defensible)
Staging deploys on backend changes; redundant mobile-only rebuilds are avoided. Production has the safety net of a CI force-deploy. Costs: production wastes Render build minutes rebuilding unchanged backend code; staging has to be manually bumped via "Manual Deploy" the one time you ever need it to match main exactly (e.g. cache warmup testing). If those costs don't hurt, leaving everything as-is is a valid answer.
ii
Symmetry: add a staging Deploy Hook to CI (recommended)
Create a GitHub secret RENDER_DEPLOY_HOOK_STAGING with the staging hook URL (visible on the staging service's Settings → Deploy page). Extend the deploy-backend job in .github/workflows/ci.yml to curl both hooks in sequence whenever it already runs. Staging then mirrors production's "deploy on every merge" behavior. Only worth doing if you ever want staging to auto-rebuild on frontend-only PRs (most relevant for asset changes like new splash PNGs — though those don't actually require a backend redeploy).
iii
Efficiency: trim production's redundant redeploys
The cleaner alternative to option ii: instead of adding a staging hook, add a path filter to the CI deploy-backend job so it only runs when vitaltrack-backend/** changed. Use GitHub Actions' paths filter at the job level. Both services then behave identically: deploy only when the backend actually changes. Builds 14 fewer Docker images per week on production. Most disciplined outcome. (If a rebuild-on-mobile-change ever becomes necessary for staging, add option ii on top.)
iv
Long-term: put staging into render.yaml
The existing vitaltrack-backend/render.yaml defines only the production service. Adding a second services: entry for vitaltrack-api-staging with its own envVars block puts both services' config under version control. Config drift between prod and staging becomes detectable via git diff. Unrelated to the deploy-trigger question above — this is about auditability and disaster recovery.
v
When you actually need to force-deploy staging
If you're about to test something backend-specific and want absolute certainty staging matches main (e.g. reproducing a bug, verifying a migration), click Manual Deploy → Deploy latest commit on staging's Render dashboard. Takes 60–90 seconds. This is the pragmatic escape hatch regardless of which of the options above you pick.
APPENDIX

Terms used in this report.

Auto-Deploy
Render's built-in feature that watches a Git branch for new commits and automatically rebuilds + redeploys when one arrives. Implemented via a GitHub webhook subscription.
Deploy Hook
An explicit HTTPS URL Render generates per service. POSTing to it triggers a deploy. Typically stored as a CI secret and called from a GitHub Actions workflow step.
Webhook
A URL one service gives to another so the second can "call back" when something happens. GitHub posts to Render's webhook when you push commits. If that HTTP call fails repeatedly, GitHub pauses future deliveries.
Preview APK
An installable Android build produced by EAS (Expo Application Services) from a pull request. Its EXPO_PUBLIC_API_URL is hardcoded at build time to the staging backend URL via the preview profile in eas.json.
render.yaml
A YAML file defining Render services as code. In this repo it lives at vitaltrack-backend/render.yaml and currently defines only the production service. Staging is configured through the Render dashboard UI, not in Git.
Idle spin-down
Unrelated to this diagnosis, but visible on the dashboard: Render's free tier puts services to sleep after 15 minutes of inactivity. The first request after sleep takes 30–60 seconds to wake the container. Different problem from deploys not firing.
