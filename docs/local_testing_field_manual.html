Chapter 01

Why this manual exists

The dev loop you use every day is a stack of abstractions. Each abstraction is a tiny miracle. Each one hides something useful. When the miracle stops working, you need to see what it hid.

Most React Native developers learn the dev loop the same way they learn to drive a car: as a sequence of motions that produce a desired outcome. Plug in phone. Run a magic command. Run another magic command. Scan a QR code. App appears. When the sequence works, nobody asks why. When it doesn't — when the QR code scan hangs, when the bundle never finishes loading, when the app starts but the dashboard is empty — most people Google the symptom and copy whatever fix is on the top Stack Overflow answer. Sometimes that works. Sometimes it makes things worse. Almost always, it leaves you no smarter than before.

This manual is for the developer who wants the other thing — to actually understand the stack underneath the dev loop, so that debugging becomes a process of localizing the failure rather than guessing in the dark. By the end of this book you will be able to answer, off the top of your head:

  • Why React Native needs a "native shell" and a separate JavaScript bundle that ship together.
  • What Expo Go actually is — why it can run any Expo-compatible app, and why some libraries crash inside it.
  • What Metro is, what it serves, and what it has nothing to do with.
  • What ADB does, and why we use exactly one of its many features.
  • How the phone reaches the laptop — three possible paths, with sharply different trade-offs.
  • Why the word localhost means different things in different places, and why this fact alone explains most "I can't reach the backend" bugs.
  • Step by step, what happens between the moment you plug in your phone and the moment the dashboard appears.
Knowing the model means you can debug anything in this stack — when "Connecting…" hangs, you'll know which of the four hops it could be stuck on, and you'll fix it in minutes instead of hours. — The thesis of this book

Chapter 02

The fundamental problem

React Native is neither a webpage nor a regular Android APK. It is a hybrid. The hybrid shape dictates everything that follows — the existence of Expo Go, the role of Metro, the need for ADB. Understand the hybrid, and the rest is consequence.

CareKosh is a React Native app. To explain why our development setup looks the way it does, we have to explain what a React Native app actually is — and the answer is more interesting than the marketing suggests. A React Native app is two halves welded together at runtime, and the welding is the source of nearly every confusion in this book.

02.01 Two halves, welded at runtime

The first half is real native Android. Buttons in your app are android.widget.Button. Text inputs are android.widget.EditText. Scroll views are android.widget.ScrollView. There is no embedded WebView, no headless browser. When you tap a button, you tap a real native widget that responds at native speed. This is what gives React Native apps their "real app" feel — they don't suffer the input lag of HTML rendered inside a Cordova shell.

The second half is JavaScript. Your component tree, your state management, your business rules, your API calls — all written in TypeScript and JSX, all run inside a JavaScript engine that's bundled into the app process. Hermes, in our case — a small, fast VM Facebook built specifically for React Native.

These two halves communicate through a bridge (or in newer React Native versions, JSI — same concept, more direct binding). When JS says "render <Text>Hello</Text>," the bridge translates that instruction into a real TextView instance with its text set to "Hello".

CH 02 · The fundamental problem Local Testing · Field Manual 5

02.02 The two halves must coexist on the phone

A complete React Native app therefore has two parts that need to coexist on the phone at the same time:

  1. The native shell — an APK that provides UI primitives, navigation, camera, file access, network stack, and the JavaScript engine itself.
  2. The JavaScript bundle — a single large .js file (around 10 MB for our app) containing your component tree, dependencies, and business logic.

How these two parts get onto the phone is where production and development diverge — sharply.

02.03 How production ships this

For the Play Store, both halves are baked into one binary. EAS Build does the work for us:

# Run from your laptop after merging to main
eas build --profile production
  ↓
Compiles your JS → bundle ┐
Packages with native shell┤→ AAB → uploaded to Play Console
Signs the binary          ┘

A full production build takes 10–30 minutes. The user downloads the AAB from the Play Store; both halves of the app are already inside, fused; the app just runs. Beautiful for shipping. Disastrous for iteration.

02.04 Why we don't do that for development

Imagine making a one-line change to a TypeScript file and waiting fifteen minutes for a fresh APK build. Multiply by hundreds of edits per day. Multiply by the number of times you'll get the change wrong on the first try. The math is fatal.

So for development, we use a clever trick: a fixed, pre-built native shell that knows how to fetch any compatible JavaScript bundle from a server. The shell never changes. Only your JS bundle changes. The iteration loop becomes:

Edit a .tsx file → save in the editor
        ↓ (< 1 second)
Metro re-bundles incrementally
        ↓ (< 1 second)
Expo Go fetches the patch over your USB cable or Wi-Fi
        ↓
Hermes hot-reloads the affected modules
        ↓
You see the change — round trip in seconds, not minutes.

That fixed shell is Expo Go — an APK you install once from the Play Store. The server is Metro — a Node.js process you run on your laptop. Your JS bundle is the only thing that changes between edits. This is the entire reason this whole setup exists.

The thesis of the dev loop

Production fuses the native shell and the JS bundle into one APK. Development separates them — fixed shell on the phone (Expo Go), JS bundle served fresh from your laptop (Metro). Everything else in this manual is consequence: how the bundle gets from your laptop to the phone, how the phone reaches the right backend, how the cable and the network divide labor.

02.05 The trade-off this creates

Splitting the shell from the bundle is wonderful for iteration speed but it does cost something: you can only use native modules that Expo Go's shell already contains. If your code imports a library that depends on its own native code (Swift, Kotlin, C++) — for example, burnt, which provides iOS-style toasts using SwiftUI / Material You APIs — Expo Go's shell will not have those native classes loaded, and the app will crash on the import.

This is why CareKosh deliberately picks pure-JavaScript dependencies whenever possible. react-native-toast-message instead of burnt. react-native-svg from the Expo SDK rather than a custom drawing library. The constraint of "must work in Expo Go" keeps our development loop fast, at the cost of occasionally picking the second-best library.

For the rare native dependency we truly need, the escape hatch is a development build — a custom Expo-Go-like shell with our specific native modules baked in. We haven't needed to do this yet. If you're reading this and you're about to add a native dependency, that decision is bigger than it looks; talk it through first.

Chapter 03

The four actors

Metro, Expo Go, ADB, and the backend. Three of them live on different machines. Each plays one role. None of them know about each other directly. They talk through specific channels and only those channels.

Now that you understand why the development setup splits the shell from the bundle, the cast of characters is short. There are exactly four pieces. Two of them live on your laptop. One lives on your phone. One lives in the cloud. Each plays a single role; none of them know more about the others than they have to. Let's introduce them in the order traffic flows.

CH 03 · The four actors Local Testing · Field Manual 9

03.01 Metro — the JavaScript packager

Metro is a JavaScript bundler in the same family as Webpack or Vite, but purpose-built for React Native. It knows about .tsx and .jsx files, knows React Native's resolver rules (which platform-specific file to pick, e.g. Foo.android.tsx versus Foo.ios.tsx), and knows how to serve incremental hot-reload patches.

When you run npm run start:staging, the npm script executes cross-env EXPO_PUBLIC_API_URL=https://… expo start --clear, which boots Metro. From that moment, Metro:

  1. Reads everything under vitaltrack-mobile/app/, components/, hooks/, utils/, store/, and the project root.
  2. Resolves every import statement — your code and all of node_modules.
  3. Transpiles TypeScript and JSX into plain JavaScript.
  4. Bundles everything into one large .bundle file — typically around 10 MB for our app.
  5. Starts an HTTP server on port 8081 to serve the bundle when asked.
  6. Watches the filesystem; on every save, recomputes which parts of the bundle need to change and pushes a patch via WebSocket to whoever's listening (Expo Go, in our case).
  7. Prints a QR code to the terminal pointing at itself.

Metro never runs your app. It just packages it. When you press Save in VS Code, Metro recomputes the bundle. When Expo Go asks for the bundle, Metro hands it over. Once the bundle is in Hermes, Metro's job is done — it just sits there waiting for the next file change.

Local-only by design

Metro is a development tool. It never ships to production. Your Render-hosted backend has nothing to do with Metro and never talks to it. If your phone can reach the backend but can't reach Metro, the problem is between the phone and your laptop — not between the phone and the cloud.

03.02 Expo Go — the generic native shell

Expo Go is a regular Android app. You installed it from the Play Store. Inside that APK, three things sit waiting:

  • The native shell that React Native needs. The bridge, the React Native runtime, and every native module the Expo SDK ships — camera, file system, secure storage, gestures, async storage, the lot.
  • A JavaScript engine. Hermes — a small VM Facebook built specifically for React Native. Smaller and faster to start than V8.
  • A bundle loader. A piece of code that knows how to take a Metro URL, fetch the bundle, hand it to Hermes, and call AppRegistry.registerComponent.

When you point Expo Go at exp://localhost:8081 (or scan a QR code that contains the same URL), the loader does the following:

1. Parse the exp:// URL → host=localhost, port=8081
2. Open HTTP connection to http://localhost:8081/index.bundle?platform=android
3. Receive ~10 MB of JavaScript over the connection
4. Pass the bundle string into Hermes
5. Hermes parses + executes
6. The bundle ends with: AppRegistry.registerComponent('main', App)
7. Expo Go's native shell sees the registration, mounts your App
8. React reconciler turns the component tree into native Android views
9. Splash screen disappears, your app's first screen appears

Expo Go is a generic shell. It can load any Expo-compatible JS bundle — CareKosh, a different app, a tutorial demo. They all work because the native APIs they call (camera, file system, etc.) are all already present in Expo Go's shell.

The cost of generality

Expo Go's shell is fixed. You cannot add new native dependencies at runtime. If a library requires its own native code that isn't in Expo's SDK, importing it inside Expo Go will crash. This is exactly why CareKosh sticks to pure-JS dependencies — and why we picked react-native-toast-message instead of burnt for the toast UI in commit #3 of the audit branch.

03.03 ADB — the Android Debug Bridge

ADB is a tool from the Android SDK that lets your laptop talk to your Android phone over USB. It has three components, and most confusion about ADB stems from not knowing which component is doing what.

ComponentWhere it runsRole
adb (client)Your laptop, the adb.exe you typeSends commands. Lightweight.
adb serverBackground process on your laptop, port 5037Multiplexes between any number of clients and any number of connected devices.
adbd (daemon)On the phone, runs whenever USB Debugging is onReceives commands from the server, executes them on the phone.

When you type adb devices in PowerShell, here is the actual sequence:

ADB can do many things — install APKs, read system logs, run remote shells, push and pull files, forward TCP ports. For our development setup, we use exactly one feature: port forwarding. Specifically the adb reverse command, which lets traffic on the phone's localhost get tunneled back to the laptop's localhost. We will explain that command in detail in Chapter 4.

"Unauthorized" means: trust not yet given

If adb devices shows your phone with status unauthorized, the daemon on the phone has not yet been told to trust your laptop's RSA key. The phone shows a dialog containing your laptop's key fingerprint; tapping "Always allow from this computer" stores the key on the phone permanently. From then on adb works without prompting.

03.04 The backend — FastAPI on Render (or Docker)

Your backend choice is independent of the dev loop. The mobile app makes HTTPS calls to some URL, and that URL responds with JSON. Which URL is determined by an environment variable, EXPO_PUBLIC_API_URL, which Metro reads when bundling and which gets baked into the JS bundle as a string constant.

Backend choiceURLWhen you'd use it
Local Dockerhttp://localhost:8000
(with adb reverse tcp:8000 tcp:8000)		Day-to-day dev with simultaneous changes on both ends.
Render staginghttps://vitaltrack-api-staging.onrender.comReproduce a staging-only bug; demo to others; CI parity.
Render productionhttps://vitaltrack-api.onrender.comManual smoke test of the live system. Use sparingly.

The package.json npm scripts wire the right URL into the right environment variable:

"start:local":   "cross-env EXPO_PUBLIC_API_URL=http://localhost:8000 expo start --clear"
"start:staging": "cross-env EXPO_PUBLIC_API_URL=https://vitaltrack-api-staging.onrender.com expo start --clear"
"start:prod":    "cross-env EXPO_PUBLIC_API_URL=https://vitaltrack-api.onrender.com expo start --clear"

When Metro bundles your code, it reads process.env.EXPO_PUBLIC_API_URL and inlines the value into the bundle as a literal string. The mobile app reads this baked-in value from vitaltrack-mobile/services/api.ts:

const API_BASE_URL = process.env.EXPO_PUBLIC_API_URL || 'http://localhost:8000';
Restart matters

That value is baked into the bundle at bundle time. Changing the env var requires restarting Metro with the --clear flag — otherwise Metro will serve a stale bundle with the old URL still inlined and you'll see traffic going to the wrong backend, with no obvious error message. Most "why is it hitting the wrong backend?" mysteries are solved by npm run start:staging with --clear.

Chapter 04

The three connection modes

Once Metro is running on your laptop and Expo Go is installed on your phone, the question becomes: how does Expo Go reach Metro? Three answers exist. Each has sharp trade-offs. Picking the right one is half the battle.

The bundle has to get from the laptop to the phone somehow. The internet connection between them — over Wi-Fi, USB, or a public tunnel — determines almost everything else about your dev experience. The decision tree below tells you which mode to pick first; the rest of this chapter explains why.

CH 04 · The three connection modes Local Testing · Field Manual 17

04.01 LAN mode — over Wi-Fi

The default for plain npx expo start. Both devices sit on the same Wi-Fi network. The phone reaches the laptop using the laptop's LAN IP — typically something like 192.168.1.5:8081.

The QR code in your terminal encodes exp://192.168.1.5:8081 with whatever your actual LAN IP is. Expo Go scans the code, fetches the bundle directly over Wi-Fi. When this works, it's beautiful — no cable, no fuss.

When LAN fails — and it fails more often than you'd expect — it's almost always one of three reasons:

  • The Wi-Fi access point has client isolation enabled (devices on the same network can't see each other). This is common on guest networks, university Wi-Fi, hotel Wi-Fi, and corporate guest networks.
  • The laptop and phone are on different subnets — wired desktop on 192.168.0.x, phone on a separate guest network 192.168.10.x. Same physical building, different logical networks.
  • The laptop's firewall (Windows Defender on first run, especially) is blocking inbound TCP on port 8081.

04.02 USB mode — adb reverse

The phone is plugged into the laptop with a USB data cable. The adb reverse command sets up a TCP tunnel: any time the phone tries to connect to its own localhost:8081, the connection is intercepted by adbd and forwarded over USB to the laptop's localhost:8081, where Metro is listening.

The same trick can forward the API port: adb reverse tcp:8000 tcp:8000 lets the phone hit http://localhost:8000 and reach the local Docker FastAPI backend running on the laptop.

USB mode is the most reliable mode. It doesn't depend on Wi-Fi state, doesn't depend on the network at all (for Metro traffic), works through any firewall configuration. The trade-off is small: the phone is tethered to the laptop. Fine at a desk; awkward when you want to walk around showing the app to someone.

CH 04 · The three connection modes Local Testing · Field Manual 19

04.03 Tunnel mode — public URL

When LAN won't work and USB isn't an option (a colleague wants to test on their phone, or you're presenting a demo from across the room), Tunnel mode is the universal fallback.

Run npx expo start --tunnel instead of plain expo start. Expo will install a small extra dependency (an ngrok-like tunneling client) and give you a public URL like exp://abc-xyz-yourname.exp.direct:80. Your laptop's Metro server is now reachable over the public internet through this URL.

Trade-offs to know about:

  • Slower. Every byte of the JS bundle now travels phone → exp.direct → your laptop and back. The first bundle load takes noticeably longer than LAN or USB. Hot reloads also have higher latency.
  • Both sides need internet. If your laptop is offline, tunnel mode can't work even though both phone and laptop can otherwise see each other.
  • Doesn't help your local Docker backend. The phone can hit Metro over the tunnel, but EXPO_PUBLIC_API_URL=http://localhost:8000 won't work because the phone's localhost is no longer connected to your laptop. In tunnel mode you should hit staging or production: EXPO_PUBLIC_API_URL=https://vitaltrack-api-staging.onrender.com.

04.04 Picking the right mode for the task

The orthogonal axes

Two questions, decided independently. (1) How does the phone reach Metro? — LAN, USB, or Tunnel. (2) Which backend should the bundle hit? — local Docker, staging, or production. The npm script picks the second; your network situation picks the first. They never collide. Mix and match freely.

Chapter 05

End-to-end flow

From the moment your USB cable touches the laptop to the moment the dashboard renders with real data — every hop, every protocol, in order. USB mode against staging backend, traced step by step.

This is the central reference of the book. Whenever something breaks during the dev loop, the failure is at one of the 38 specific steps below. Find the last step that obviously worked, find the next step that obviously didn't, and the broken layer is between them. The failure is almost never mysterious; it's usually just further upstream than you assumed.

The flow has four phases. Phase 1 happens once per dev session. Phase 2 happens once per fresh app start. Phase 3 happens on every API call after that. Phase 4 happens on every code save.

CH 05 · End-to-end flow Local Testing · Field Manual 23

05.01 Phase 1, narrated

You sit down at your laptop. Your phone is charged but not connected. CareKosh source is checked out at ~/code/vitaltrack/vitaltrack-mobile. PowerShell is open in that directory.

  1. Plug in the USB data cable. Establishes the physical channel between phone and laptop. Make sure it's a data cable, not charge-only — many bundled USB-C cables don't carry data.
  2. Phone shows "Allow USB debugging?". The dialog displays your laptop's RSA key fingerprint. Tap "Always allow from this computer" so the trust persists across sessions.
  3. adb devices in PowerShell. Phone shows up with status device. If it shows unauthorized, you missed the dialog or the trust didn't store; re-plug.
  4. adb reverse tcp:8081 tcp:8081. The laptop's adb client tells the laptop's adb server to push a rule to the phone's adbd: "any connection to your own port 8081, tunnel to laptop's port 8081." The phone's TCP stack now intercepts.
  5. npm run start:staging. The npm script runs cross-env EXPO_PUBLIC_API_URL=https://… expo start --clear. Metro starts, reads source files, builds the initial bundle (~10 MB, takes 10–30 seconds the first time), opens the HTTP server on port 8081, prints a QR code containing exp://localhost:8081.
  6. Open Expo Go on the phone, scan the QR code (or type the URL manually).

05.02 Phase 2, narrated

From the QR scan to the dashboard appearing — between five and twenty seconds, depending on your laptop and how warm Metro's cache is.

  1. Expo Go parses the URL exp://localhost:8081 and opens an HTTP connection to that address.
  2. The phone's TCP stack sees a connection request to localhost:8081; the rule from step 4 fires; the connection is intercepted by adbd.
  3. The connection is tunneled over the USB cable, through the adb server on the laptop, to the laptop's own localhost:8081 — where Metro is listening.
  4. Metro receives an HTTP request like GET /index.bundle?platform=android&dev=true&hot=true. If it's the first request after a code change, Metro builds. Otherwise it serves cached output.
  5. ~10 MB of JavaScript travels back through the same tunnel: laptop's Metro → laptop's adb server → USB cable → phone's adbd → Expo Go's HTTP client.
  6. Expo Go feeds the bundle to Hermes. Hermes parses, executes. The bundle ends with AppRegistry.registerComponent('main', App).
  7. Expo Go's native shell mounts your App component. React's reconciler builds the native view tree. Splash screen disappears. Your auth screen or dashboard appears.

05.03 Phase 3, narrated

The app is running. Now it needs data from the backend. This is where the USB cable becomes irrelevant.

  1. The app calls fetch() with the URL baked into the bundle at build time — https://vitaltrack-api-staging.onrender.com/api/v1/auth/me with a JWT in the Authorization header.
  2. The native fetch implementation issues an HTTPS request — not over the USB cable. Over the phone's own internet connection, whichever it is (Wi-Fi or LTE).
  3. DNS resolves the hostname to Render's edge IP. TLS handshake. Encrypted request travels over the public internet.
  4. Render's load balancer routes to your staging FastAPI container. The container authenticates the JWT, queries Postgres on Neon, returns JSON.
  5. The response travels back: Render → public internet → phone's Wi-Fi/LTE → app's fetch promise resolves. TanStack Query stores the result in its cache. Components subscribed to that query re-render with the data.

05.04 Phase 4, narrated

You spot a typo. You fix it in VS Code and press Save. The fix appears in the running app within a couple of seconds, without restarting.

  1. You save the .tsx file in VS Code.
  2. Metro's filesystem watcher detects the change.
  3. Metro recomputes which parts of the bundle changed (incremental rebuild — typically under one second).
  4. Metro pushes a "hot update" message to Expo Go via WebSocket on the same port 8081. (Same tunnel as the bundle load — adb reverse handles WebSocket too.) Hermes hot-reloads the changed modules. React re-renders affected components. You see your change.
The single most useful sentence in this book

The USB cable is involved in steps 8–11 (initial bundle load) and step 21 (hot reload). The internet is involved in steps 15–18 (API calls). The cable and the internet are unrelated. You can unplug USB after the bundle loads and the app keeps working — until you change code and need a re-bundle.

Chapter 06

The setup, with
the why

The same six commands you'd find in any operational guide, but each one annotated with what it does at the network level — and what would happen without it.

Operational guides skip the why. They tell you to run adb reverse tcp:8081 tcp:8081 without telling you what would happen if you didn't, or what changes if you ran adb forward instead, or why kill-server sometimes "fixes" things. This chapter is the operational guide annotated. You can copy any command verbatim; you can also now reason about every one of them.

06.01 Step 1 — One-time prerequisites

On the phone

Settings → About Phone → Build Number → tap 7 times
        ↓
"You are now a developer!"
        ↓
Settings → Developer Options → USB Debugging → ON

Why: Toggling USB Debugging starts the adbd daemon on the phone. Without it, every adb command fails immediately because there's no daemon to talk to. The daemon is the phone-side endpoint of the entire ADB conversation.

Install Expo Go from the Play Store

Why: Expo Go is the native shell that will load your JS bundle. Without it, the phone has no way to execute the bundle Metro produces. You only need to install Expo Go once per phone — not per project.

On the laptop

Install Android SDK Platform Tools (gives you adb.exe)
Add to PATH: C:\platform-tools  # or wherever you extracted it

Why: Without adb.exe on your PATH, the entire ADB chapter of this book is unavailable. Platform Tools is a tiny ZIP from Google; no Android Studio install required.

cd vitaltrack-mobile
npm install --legacy-peer-deps

Why: Installs all JavaScript dependencies into node_modules/. The --legacy-peer-deps flag tells npm to be lenient with the React Native ecosystem's perpetually-mismatched peer-dependency declarations — a known quirk that's easier to swallow than fight. Without this, npm install may fail with peer-dep conflicts.

06.02 Step 2 — Plug in the phone, authorize ADB

Plug USB cable from phone to laptop

Why: Establishes the physical channel. Use a data cable. Many bundled USB-C cables are charge-only and won't pass adb traffic. (You can tell empirically: if adb devices shows nothing even after authorization, swap cables before debugging anything else.)

On the phone, accept "Allow USB debugging?" dialog.
Tick "Always allow from this computer."

Why: The phone's adbd daemon refuses commands from any computer it doesn't trust. The dialog shows your laptop's RSA key fingerprint — that's the laptop's identity from the phone's point of view. Tapping "Always allow" stores the public key on the phone permanently (until you revoke it via Developer Options).

adb devices

Why: Lists every Android device the laptop can see. Should show your phone with status device. The full state machine:

OutputMeaningAction
(empty)Phone not detected at all.Cable, port, or driver issue.
unauthorizedPhone seen, trust not given.Look for the dialog on the phone; re-plug if missing.
deviceAuthorized and ready.Continue to step 3.
offlinePhone seen, daemon unresponsive.adb kill-server && adb start-server.

06.03 Step 3 — Set up the USB tunnel

adb reverse tcp:8081 tcp:8081

Why: This is the core trick of USB mode, and probably the single command in this book most worth understanding. Read carefully — the syntax is asymmetric.

The two arguments are tcp:<phone-side-port> tcp:<laptop-side-port>. The command is read aloud as: "any time something on the phone tries to connect to TCP port 8081 on the phone, intercept it and instead forward to TCP port 8081 on the laptop."

You can verify the rule exists at any time:

adb reverse --list

# Output:
(reverse) tcp:8081 tcp:8081
adb reverse is not sticky

The rule is lost when any of these happen: phone unplugged, phone rebooted, laptop rebooted, adb kill-server run, USB cable physically jiggled hard enough to disconnect briefly. After any of those, re-run the command. Symptom: Expo Go suddenly hangs on "Loading from localhost:8081" when it was working a minute ago — you've lost the tunnel.

If you also want the phone to reach a local Docker backend on your laptop, add a second tunnel for the API port:

adb reverse tcp:8000 tcp:8000

Why: Same trick, different port. Now localhost:8000 on the phone reaches the FastAPI backend running in Docker on the laptop. Skip this if you're hitting staging or production — those are real public URLs reached over the phone's own Wi-Fi/LTE, not over USB.

06.04 Step 4 — Start Metro with the right backend

cd vitaltrack-mobile
npm run start:staging

Why — let's unpack this script the way a debugger would. The line in package.json looks innocent:

"start:staging": "cross-env EXPO_PUBLIC_API_URL=https://vitaltrack-api-staging.onrender.com expo start --clear"

But there are four moving parts here, each with its own job:

npm run start:staging

  └─→ which is: cross-env EXPO_PUBLIC_API_URL=https://… expo start --clear
       │              │                                    │            │
       │              │                                    │            └── ④ flush Metro's cache
       │              │                                    │                necessary after any env-var change
       │              │                                    │
       │              │                                    └── ③ start the Expo dev server

       │

Metro boots. After a few seconds (or up to 30 on a cold start with cleared cache), the terminal prints:

› Metro waiting on exp://localhost:8081
› Scan the QR code above with Expo Go (Android) or the Camera app (iOS)

The QR code in your terminal encodes that exp://... URL. (LAN mode would print your real LAN IP instead of localhost; USB mode keeps it as localhost because that's what reaches the laptop through the tunnel.)

The --clear flag, demystified

Metro caches transformed modules aggressively to make subsequent rebuilds fast. The cache key does not include environment variables. So if you run start:staging, then later start:local without --clear, Metro will happily serve the still-cached staging bundle to your local-mode phone. Symptoms range from "wrong backend" to "it inexplicably works fine on Tuesday." All our start:* scripts pass --clear for this reason. If you start Metro any other way and switch backends, restart with --clear.

06.05 Step 5 — Connect Expo Go

Two equivalent options. Pick whichever your fingers prefer.

Option A — Scan the QR code

Open Expo Go on the phone. There's a "Scan QR code" button on its home screen. Point at the QR code in your terminal. Expo Go opens the URL automatically. This is the path most people use.

Option B — Type the URL manually

Open Expo Go. There's a URL input field labelled "Enter URL manually." Type either:

  • exp://localhost:8081 — works in USB mode only, because of the adb reverse tunnel.
  • exp://192.168.1.5:8081 (substituting your actual laptop LAN IP) — works in LAN mode.

Either way, the same thing happens behind the scenes:

  1. Expo Go opens HTTP connection to the URL it parsed.
  2. Connection traverses the chosen path (USB tunnel or Wi-Fi).
  3. Metro receives the request, builds (or pulls from cache), responds with the bundle.
  4. Bundle arrives back at Expo Go, gets handed to Hermes, executes.
  5. The app's first screen appears — auth screen on a fresh install, dashboard if you're already logged in.

06.06 Step 6 — Verify it actually works

Loading the bundle proves Metro reached you. To prove the backend is reachable too, do anything in the app that triggers an API call: log in, scroll the dashboard, edit an item. If you see real data — your inventory, your orders, your previous edits — the entire chain is working end-to-end. Metro served the bundle, Expo Go executed it, the app reached the backend over the phone's internet, the backend returned data, TanStack Query cached it, the UI rendered.

If instead you see "Connecting…" forever, or the dashboard renders empty with no error, or you see a red toast labelled "Network request failed" — something's broken in the chain. The next chapter narrows down where.

A debugging mantra

Whenever something hangs or fails, the question to ask first is not "why doesn't this work?" but "which of the four actors is the problem?" Metro? Expo Go? ADB? The backend? The diagrams in chapters 3 and 5 are the map. The question always has an answer.

Chapter 07

Three things that confuse everyone

If you only remember three things from this book, make it these. They explain the majority of "wait, why doesn't this work?" moments in the dev loop.

Every developer who works on a React Native app for long enough hits the same three confusions. Each is a category of bug that produces dozens of distinct symptoms. Each has a single conceptual fix. Once you have the model, the symptoms collapse into one diagnosis.

07.01 localhost is ambiguous

localhost is not a global address. It always means this machine, and this machine changes depending on whose perspective you're taking. The same word, used by code on the laptop versus code on the phone, points to two completely different computers.

The fix is what we covered in section 06.03: adb reverse tcp:8000 tcp:8000 bridges the two interpretations. After that command, the phone's localhost:8000 does reach the laptop's localhost:8000 — but only by special arrangement, and only over USB.

This single confusion explains many recurring bug categories:

  • "My local Docker backend isn't reachable from the app" — usually because EXPO_PUBLIC_API_URL=http://localhost:8000 was set but no adb reverse tcp:8000 tcp:8000 was run.
  • "It worked yesterday and now it doesn't" — the reverse rule was lost when the phone rebooted overnight.
  • "It works in USB mode but not LAN mode" — LAN mode needs the laptop's actual IP (http://192.168.1.5:8000), not localhost.
Rule of thumb

Never let localhost appear in EXPO_PUBLIC_API_URL unless you've also run a matching adb reverse. If you'd rather not think about it, point at staging or production — those are real public URLs that work the same from any machine.

07.02 The USB cable carries code, not data

This is the most common mistake new developers make on this stack: "I'm on USB, why doesn't the phone reach my Docker backend?" The answer is that USB and "the phone's internet" are two completely independent channels.

Concretely, here's how the two channels divide labor:

  • If EXPO_PUBLIC_API_URL is https://vitaltrack-api-staging.onrender.com, every API call goes over the phone's own internet (Wi-Fi or LTE) — directly to Render. The USB cable is irrelevant for these calls.
  • If EXPO_PUBLIC_API_URL is http://localhost:8000, the phone tries to reach its own port 8000 — which only succeeds if a matching adb reverse rule turned that into a tunnel back to the laptop's Docker.

The consequence is that you can be in USB mode (Metro over USB) and hitting staging (API over the phone's Wi-Fi) at the same time. They're independent. Two separate channels, two separate questions, two separate things to debug.

This is also why you can unplug the USB cable mid-session and the app keeps working. The bundle is already loaded into Hermes; the app's API calls don't use USB. You only need the cable back when you change code and need a fresh bundle (or a hot-reload patch).

07.03 Metro and the backend never meet

Metro serves your code. The backend serves your data. They run on different machines, on different ports, with different protocols.

MetroBackend
WhereYour laptopRender's data centre (or your local Docker)
Port8081443 (or 8000 locally)
ProtocolHTTP (+ WebSocket for hot reload)HTTPS (or HTTP locally)
LifetimeWhile you have the dev session runningAlways (or while Docker is up)
Talks toExpo GoYour running app

Both are involved when the app first loads — Metro to deliver the code, then the backend to deliver the data once the code is running. But they don't know about each other and don't share state. They run completely separate processes on completely separate machines.

So when your dashboard renders empty, the question is sharp: did Metro deliver the bundle? (You can tell — the app rendered something; if it didn't, you'd see a "Network request failed" loading the bundle screen.) Did the backend not respond? (You'd see a red error toast or a forever-spinning loader.) These are two different problems with two different fixes. Knowing which side broke is half of debugging.

A simple discriminator

If the app shows your splash, your nav bar, your fonts, your icons — Metro worked. The bundle is in Hermes. Any subsequent emptiness or error is a backend problem (or a backend-reachability problem). If the app never gets past the Expo Go splash or shows "Loading from localhost:8081" forever — Metro is the problem. The bundle never made it to Hermes.

08.03 A worked example: "I'm running npm run start:staging, why does adb reverse fail?"

This question came up recently. Here's the diagnosis in the model from this book:

The actual problem

If adb reverse returns an error, that's an ADB authorization issue, not a code or network issue. The phone needs to authorize the laptop for USB debugging first. Until adb devices shows your phone with status device (not unauthorized, not empty), no adb reverse will succeed.

Fix sequence:

  1. Look at the phone screen — Android shows an "Allow USB debugging?" dialog with the laptop's RSA key fingerprint when you connect to a new computer. Tap "Always allow from this computer" → OK.
  2. If you don't see the dialog: make sure the phone is unlocked when plugged in (locked phones don't show the prompt); try a different USB cable (some are charge-only); try a different USB port.
  3. If the dialog still doesn't appear: adb kill-server && adb start-server && adb devices — that re-triggers the prompt.
  4. Last resort — phone Settings → System → Developer Options → "Revoke USB debugging authorizations" → unplug, replug → fresh prompt.

And the second half of that same question — do I even need adb reverse?

A useful realization

If you're running npm run start:staging, the app points at https://vitaltrack-api-staging.onrender.com. That's a real public URL — the phone reaches it directly over its own Wi-Fi/LTE. You do not need adb reverse for the API. You only need adb reverse tcp:8081 tcp:8081 for Metro, and only if your phone's Wi-Fi can't reach your laptop's LAN IP. If LAN mode works (QR scan, bundle loads), skip the reverse entirely. Try the QR code first; come back to adb reverse only if Metro hangs.
