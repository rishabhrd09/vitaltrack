01

Final Verdict

The seven original backend goals are necessary and well scoped. For a stricter backend production finish, add Goal 8. For Play Store publication and real users, add two separate release-readiness goals that do not disturb the backend branch pack.

Short answer

The original 7 goals close every defect that was verified in the current backend code. They should be run in order, one branch at a time. They are not speculative feature work; they are production guard work for a working app.

Backend production-grade finish line: run those 7 goals, then run Goal 8 only after the required CORS origin decision is known. Goal 8 covers CORS, unauthenticated email diagnostics, and server-side default-category protection.

Play Store / real-user finish line: after the backend pack, run Goal 9 for mobile release privacy/platform hardening and Goal 10 for backup, monitoring, and launch operations. These are separate because they target the installable app and production operations, not the FastAPI defect pack.

Layer Decision Why
Must do Goals 1-7 Close the destructive sync route, GET account deletion, reset XSS, missing domain tests, order race/workflow holes, fake health, plain-string secret reprs, advisory CI, and stale docs.
After pack Goal 8 Closes three known deferred backend security gaps that are small and important, but one of them requires a real production CORS origin list before merge.
Before public launch Goals 9-10 Cover mobile Play Store safety, Android platform configuration, production logging, privacy disclosures, backups, monitoring, and rollback. These are release guardrails, not backend behavior changes.
Park Migration hygiene and template cleanup Useful, but not necessary to keep the current working app functionally correct or materially safer right now.
Final recommendation: Treat CareKosh backend production guard as an 8-goal program, with Goals 1-7 as the verified-defect pack and Goal 8 as the minimal backend security finish. Treat Play Store release as a separate 2-goal launch guard: Goal 9 for mobile privacy/platform hardening and Goal 10 for production operations. Do not mix Goals 9-10 into the backend branches.

02

Play Store And Solo-Developer Fit

Goals 1-8 are backend production guardrails. Goals 9-10 are the separate mobile and operations guardrails that make sense before inviting real users into a health-adjacent Play Store app.

Decision

For a solo developer targeting real users, Goals 1-7 are justified and necessary for the verified backend risks. Add Goal 8 for the cleanest backend posture, then Goals 9-10 for Play Store/mobile/operations readiness. This is especially true because CareKosh handles accounts, authentication, email, caregiver inventory records, and home ICU/health-adjacent data.

Question Answer
Are these required by Google Play? No. Google Play also requires app-content declarations such as privacy policy, Data Safety, app access instructions for login-gated apps, target audience, content rating, ad declaration, and sensitive-permission declarations when applicable. These backend goals complement those requirements; they do not replace them.
Are they necessary before real users? Yes for Goals 1-7, because each closes a verified backend security, correctness, operational, test, or CI/doc honesty issue. Goal 8 is the minimal post-pack backend hardening. Goals 9-10 are necessary before a serious public Play Store release because they cover mobile privacy/platform safety and production operations.
Are they overkill for 10 users? No. The riskiest defects are not scale-dependent: a crawler can trigger GET deletion with one user, a destructive sync route can delete data for one user, and double-apply can corrupt one family's inventory.
Are they enough for 1000-5000 users? They are enough for the verified backend defects in this audit. For 1000-5000 real users, add Goal 9 and Goal 10 before broad release because Google Play privacy, mobile platform hardening, backups, monitoring, and rollback are outside the backend-only pack.
Will they break existing functionality? No plan can honestly guarantee zero regression. This pack is designed to minimize breakage by splitting changes into branches, avoiding mobile changes, preserving API contracts, adding tests before risky logic changes, and requiring CI/staging verification after each branch.

Play Store checklist that is separate from this backend pack

  • Publish and link an active privacy policy that covers app data collection, use, sharing, and deletion.
  • Complete the Play Console Data Safety form accurately, including data handled by third-party SDKs.
  • Provide reviewer access instructions if login is required to inspect the app.
  • Declare ads, target audience, content rating, and any high-risk/sensitive permissions truthfully.
  • Ensure sensitive user data is transmitted securely over HTTPS and used only for expected app functionality.

Risk posture

For a medical/home-ICU inventory app, user trust matters before the user count is large. The goal pack is intentionally conservative because data loss, account deletion, and stock corruption are high-impact even at small scale.

04

Goal Sequence: Required Verified-Defect Pack

Run these as separate branch-sized changes. Each one is deliberately scoped to protect the already-working app.

Goal 1

security/remove-legacy-sync-router

Remove legacy destructive sync API

Objective: remove the unused /api/v1/sync/* surface that can delete user data based on incomplete offline payloads.

Implementation

  • Remove sync import and router.include_router(sync.router) from app/api/v1/__init__.py.
  • Delete or quarantine app/api/v1/sync.py and app/schemas/sync.py.
  • Remove sync schema imports and __all__ entries from app/schemas/__init__.py.
  • Keep local_id columns, migrations, and SYNC_PUSH/SYNC_PULL enum values.
  • Update backend README count from stale 42 to 38 after sync removal.

Compatibility guard

  • No mobile edits. Current mobile is server-first and no longer calls sync.
  • No database migration. Leaving local_id is harmless and safer for rollback.
  • Historical docs may mention old sync as history, but active backend docs must not advertise it.

Verification

  • pytest tests/ -q --cov=app --cov-report=term-missing
  • ruff check app/
  • python -c "from app.main import app; print(len([r for r in app.routes if hasattr(r, 'path') and r.path.startswith('/api/v1')]))"

Accept: all tests pass, import succeeds, sync route count is 0, API route count is 38.

Goal 2

security/account-deletion-post-confirm

Require POST confirmation for account deletion

Objective: prevent account deletion by link previews, crawlers, scanners, or accidental email-link opens.

Implementation

  • Keep GET /confirm-delete/{token} but make it render a confirmation page only.
  • Add POST /confirm-delete/{token} for the destructive action.
  • Reuse the existing SHA-256 token hash and expiry validation.
  • Prefer a simple HTML form with method="post" and action="".
  • Keep DELETE /me, POST /cancel-delete, token format, and cascades unchanged.

Compatibility guard

  • Email links still work; users just get a deliberate final confirm button.
  • Mobile does not call the confirm route, so mobile auth flow stays intact.
  • No schema or token generation changes.

Tests

  • GET valid token returns page and user remains.
  • POST valid token deletes user.
  • Invalid/expired POST returns error and user remains.
  • Invalid GET returns error and never deletes.

Accept: pytest tests/test_auth.py -q and full backend pytest pass; final API count becomes 39.

Goal 3

security/reset-password-token-escaping

Escape reset token in password reset HTML

Objective: close reflected-token XSS without changing the password reset API contract.

Implementation

  • Import stdlib html in auth.py.
  • Render safe_token = html.escape(token, quote=True) into a data-* attribute.
  • Have JavaScript read the token from the DOM.
  • Remove all raw {token} interpolation from <script> content.

Compatibility guard

  • Do not change POST /reset-password request body.
  • Do not change token hashing, expiry, password rules, or email link format.
  • Do not migrate all auth HTML to templates here.

Tests

  • Injection probe with quotes and <script> renders escaped.
  • Raw injected script does not appear in the response.
  • Normal reset flow still works.

Accept: auth tests and full backend tests pass.

Goal 4

test/domain-inventory-order-coverage

Add domain coverage for inventory and orders

Objective: create a behavioral safety net around items, categories, and orders before changing correctness logic.

Implementation

  • Add tests/test_items.py and tests/test_orders.py; optionally add tests/test_categories.py.
  • Cover item CRUD, filters, stats, needs-attention, OCC conflicts, and per-user scoping.
  • Cover category CRUD and per-user scoping.
  • Cover order create, forward lifecycle, apply success, apply invalid status, missing item, sequential double-apply, and scoping.
  • Add a test-only guard so destructive test isolation refuses non-test DB targets.

Compatibility guard

  • This goal is tests-first and should not change production behavior.
  • Do not refactor fixture isolation unless absolutely required.
  • If a new test reveals a real bug, stop and report instead of silently fixing it in this branch.

Accept: existing tests plus new tests pass; items.py >= 70% and orders.py >= 70% coverage.

Goal 5

correctness/atomic-apply-order-stock

Make order stock application atomic

Objective: prevent double stock application and enforce the order workflow the server already documents.

Implementation

  • Add failing concurrency test first: two apply requests, only one should win.
  • Claim the order with a guarded DB update where status is still received.
  • Increment item quantity and version with DB-level updates, not Python object math.
  • Preserve missing-item fail-fast validation, audit logs, and activity logs.
  • Validate order item existence and ownership during order creation.
  • Enforce legal status transitions; stock_updated should only be reached via apply.

Compatibility guard

  • Search mobile order screens before finalizing the transition map.
  • Allow same-status PATCH as an idempotent retry-safe no-op.
  • Do not redesign order schema, reservations, or mobile behavior.

Accept: concurrency test fails before fix and passes after; Goal 4 lifecycle tests remain green.

Goal 6

ops/health-and-secret-types

Make health truthful and protect secret representations

Objective: make runtime health honest and stop sensitive config from appearing as plain strings in repr/log contexts.

Implementation

  • Change /health into a DB-backed readiness check using SELECT 1 and timeout.
  • Add /live as a no-DB liveness endpoint.
  • Move Render healthCheckPath to /live so DB outages do not trigger restart loops.
  • Convert SECRET_KEY and MAIL_PASSWORD to SecretStr.
  • Add explicit accessors for raw secret values and update JWT/email call sites.
  • Keep DATABASE_URL as str in this goal.

Compatibility guard

  • Mobile already treats /health 503 as a retryable cold-start/server-unavailable signal.
  • Do not change JWT algorithm, token expiry, or secret values.
  • Do not convert DATABASE_URL until alembic, conftest, and engine consumers are audited.

Accept: /health 200 with DB, /health 503 when probe fails, /live 200 always, all auth/JWT/email tests pass, API route count remains 39.

Goal 7

ci/block-quality-gates-and-docs

Make backend quality gates blocking and docs honest

Objective: enforce the standard the code now meets and remove stale production-readiness claims.

Implementation

  • Keep Ruff and pytest blocking.
  • Make mypy blocking only if the baseline is green; otherwise move it to a separate clearly advisory job.
  • Make Trivy blocking for high/critical after baseline triage.
  • Add per-file coverage gates for items.py and orders.py using coverage JSON/XML.
  • Set default GitHub workflow token permissions to read-only and review third-party action refs; pin high-risk third-party actions where practical.
  • Update README, roadmap, and developer docs for route counts, sync removal, test status, health semantics, CORS truth, and deferred follow-ups.

Compatibility guard

  • Do not change runtime behavior in this goal.
  • Do not make CORS stricter in this branch; document it as Goal 8 because it needs real origin values.
  • Do not pretend mypy/Trivy are clean until the baseline run proves it.

Accept: CI passes with blocking tests, route floor, coverage gates, and triaged security scan; docs no longer contain stale 42 total, 40/53, or active legacy sync claims.

05

Critical Goal 8: Post-Pack Security Hardening

This goal is not part of the original verified-defect pack, but it is the honest minimal add-on if "production-grade" means no known backend security gaps left untriaged.

Why Goal 8 is separate

Goals 1-7 should land first because they fix proven defects and build the test/CI foundation. Goal 8 depends on one product/deployment decision: the real allowed CORS origins. Without that list, a CORS validator can break the Render deployment.

Goal 8

security/post-pack-hardening

Harden CORS, email diagnostics, and default categories

Objective: close the three important deferred backend security gaps while preserving the working mobile app.

Item Implementation Breakage guard
CORS hardening Replace production CORS_ORIGINS=["*"] with explicit origins in Render/env, then add a production validator that rejects wildcard origins. Do not merge until actual web/admin origins are known. Native mobile requests are not governed by browser CORS, so mobile should keep working.
Email service status Restrict GET /api/v1/auth/email-service-status to authenticated users, superuser/admin if such a role is used, or disable it in production. Keep non-production diagnostics available if useful. Add tests for unauthenticated 401/403 and allowed diagnostic access.
Default-category deletion Reject deletion of Category.is_default == true on the backend. Return 400/409 with clear detail. Add test proving non-default category deletion still works.

Verification

  • Backend tests pass.
  • CORS validator test proves wildcard is rejected only in production.
  • Configured Render env uses explicit origins before production deploy.
  • Email diagnostic is no longer public in production.
  • Default categories cannot be deleted through the API.

Accept: Goal 8 closes known deferred security gaps with no mobile behavior regression and no deployment break from missing CORS configuration.

Additional Release Goals

These are not replacements for Goals 1-8. They are separate, low-regression launch guardrails needed because the app will be installed from Google Play and used by real people.

Decision

Yes, there are two additional important goals outside the backend production guard. They should be done before a serious public Play Store launch or before inviting non-test users at scale. They should not break current functionality because they mostly tighten configuration, logging, docs, and operational readiness.

Goal 9

mobile/playstore-release-hardening

Harden the Android release and Play Store privacy posture

Objective: make the installable Android app match Google Play and OWASP mobile expectations without changing the working server-first product flow.

Implementation

  • Disable cleartext traffic for production Android builds, or introduce an environment-specific debug-only exception so local development can still use http://localhost:8000.
  • Audit Android permissions in app.json; remove legacy storage permissions unless the built APK/AAB proves they are required for image/PDF flows.
  • Gate or remove production console.log, console.warn, and console.error statements that include email, username, URLs, error payloads, or operational details.
  • Review persisted AsyncStorage query cache: document exactly what inventory/order/category data is stored, ensure it is cleared on logout/login, and decide whether backup exclusion/encryption is needed for health-adjacent data.
  • Confirm SecureStore plugin configuration and Android backup behavior for token storage.
  • Create a Play Data Safety and privacy-policy data inventory: account info, email, username, phone, inventory items, photos/images if used, files/PDF export, diagnostics/crash logs if any, and third-party SDK data.
  • Update Play Store release docs with app access instructions for reviewers, privacy policy URL, target audience, permissions, content rating, and account deletion path.

Compatibility guard

  • Do not remove features. If a permission is required for image picking, document it and keep it.
  • Keep development builds usable with local HTTP through a dev-only configuration path.
  • Keep production EXPO_PUBLIC_API_URL on HTTPS.
  • Run a preview APK smoke test after permission or cleartext changes.

Accept: production build uses HTTPS-only network behavior, no sensitive production logs remain, permission list is justified and minimal, Play Data Safety inputs are documented, and registration/login/inventory/order/PDF/image flows still work in a preview build.

Goal 10

ops/launch-readiness-runbook

Add backup, monitoring, and rollback readiness

Objective: make the production service operable for real users without changing app behavior.

Implementation

  • Document Neon/Postgres backup schedule, retention, owner, and a restore drill against a disposable database.
  • Add a release checklist for backend deploy, mobile preview build, Play internal testing, smoke account, rollback decision, and post-deploy verification.
  • Add minimal uptime/error monitoring for /live, /health, login, register, inventory list, and order apply flows.
  • Define log redaction and retention rules so health-adjacent user data, tokens, passwords, and DB URLs are not retained in logs.
  • Record incident contacts, Render/Neon/Brevo dashboards, environment variables that must exist, and how to rotate SECRET_KEY with user-impact warning.
  • Run a small load/cold-start smoke test sized for the expected early audience rather than a large enterprise benchmark.

Compatibility guard

  • This goal should be docs, monitoring config, and smoke scripts only unless a real defect is discovered.
  • Do not change database schema or app flows.
  • Do not run restore drills against production.
  • Keep free-tier Render cold-start behavior documented honestly.

Accept: there is a tested restore path, a clear launch/rollback checklist, basic monitors exist, logs have a redaction policy, and a solo developer can handle a small production incident without guessing under stress.

Execution Rules For Every Goal Agent

These rules are what keep a production-guard branch from accidentally disturbing a working app.

Start from fresh, up-to-date main; create only the named branch for that goal.

Implement only the goal scope. Do not bundle refactors, UI polish, or mobile changes unless explicitly required.

Run backend commands from vitaltrack-backend/.

Never run pytest with staging or production DATABASE_URL; the fixture drops tables.

Use a disposable Postgres test DB and ENVIRONMENT=testing.

Preserve current mobile server-first behavior and API response shapes unless the goal explicitly changes a browser-only HTML route.

If a test exposes a new production bug outside the branch scope, stop and report it instead of quietly expanding the branch.

Open a PR and merge before beginning the next goal.

10

Final Acceptance Gate

CareKosh should be considered through the backend guard and launch-readiness pack only when these conditions are true.

Security

Sync routes gone, deletion POST-only, reset token escaped, public email diagnostic restricted, wildcard production CORS removed, default categories protected server-side.

Correctness

Order creation validates owned items, status transitions are enforced, stock application is atomic, and double apply cannot inflate inventory.

Reliability

Health reports DB readiness truthfully, /live supports platform liveness, secrets use safe reprs, and tests cover core item/order flows.

Governance

CI blocks on tests, route floor, coverage gates, and triaged security scans; docs match current server-first backend behavior.

Mobile Release

Production Android build uses HTTPS-only behavior, permissions are minimal and justified, sensitive production logs are removed, and Play Data Safety inputs are documented.

Operations

Backup restore path, launch checklist, monitors, rollback notes, and log-retention/redaction rules are documented and tested away from production data.

Final answer to the central question

Are only the original 7 goals needed? They are enough for every backend defect verified in code. For an honest backend production-grade posture, add Goal 8. For a Play Store app targeting real users, add Goal 9 and Goal 10 as separate release-readiness goals. After Goals 1-10 are green, do not add more engineering goals until staging evidence, Play internal testing, and real operational checks say something specific is still missing.
