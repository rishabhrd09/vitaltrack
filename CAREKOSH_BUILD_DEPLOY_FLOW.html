The questions

Question 1

I pushed to my feature branch. What just happened?

Answer

Nothing on GitHub. Nothing on Render. Nothing on EAS. CI only triggers on pull_request, push to main, or workflow_dispatch — not on arbitrary branch pushes.

The only side effects of a feature-branch push are: the commits land on GitHub, and if your branch is being compared in an open PR, the PR's CI re-runs.

Misconception

"Render deploys every push" — no. Render watches main only. Until you merge, your code is not running anywhere except your laptop.

Question 2

I opened a PR. What runs?

Answer

Four jobs run in .github/workflows/ci.yml, in parallel where possible:

JobDoes~Time
test-backendpytest (postgres:16 service), ruff lint, mypy1–2 min
test-frontendTypeScript check, ESLint, expo-doctor30–60 s
security-scanTrivy filesystem scan (CRITICAL + HIGH)~1 min
pr-checkMerge gate — succeeds only if backend + frontend passinstant

Neither build-preview nor build-production runs by default. The first needs the build-apk label; the second is disabled.

Question 3

How do I get a preview APK from a PR?

Answer

Add the label build-apk to the PR.

The build-preview job is gated on this label. Once applied, it runs eas build --profile preview --platform android against the PR branch. Artifact: an APK bound to the staging backend (https://vitaltrack-api-staging.onrender.com), because EXPO_PUBLIC_API_URL is baked into the binary at build time.

Why label-based

EAS build credits are finite. Running a build on every PR push burns them. A label makes each build an explicit reviewer decision.

Question 4

I merged the PR. Now what?

Answer — two things happen in sequence

1. CI re-runs on main. Same four jobs. If any fails, the deploy is skipped.

2. deploy-backend runs. It POSTs to the Render deploy hook URL (stored as the RENDER_DEPLOY_HOOK repository secret). Render pulls the new image and starts a rolling deploy of both the production and staging services — both watch main; their only difference is env vars.

TIMELINE FOR A TYPICAL MERGE t+0 merge button clicked t+0.1 GitHub creates merge commit on main t+0.2 CI triggers on push event t+2m backend/frontend/security pass on main t+2m deploy-backend POSTs Render hook t+2.5m Render pulls image t+3m docker-entrypoint.sh: waits for DB, runs alembic upgrade head t+4m Gunicorn replaces old workers; health check passes t+4m production + staging both live on new code

No mobile build is kicked off by merging. Mobile releases are a separate manual step (Q6).

Question 5

What exactly does the backend server do on startup?

Answer — docker-entrypoint.sh is the whole story
  1. Parse DATABASE_URL into host/port/user/db (handles postgresql:// and postgresql+asyncpg://)
  2. Wait for DB with pg_isready (up to 30 retries × 2 s = 60 s)
  3. Run alembic upgrade head — idempotent; skips already-applied migrations
  4. exec the CMD: gunicorn -k uvicorn.workers.UvicornWorker -w 4 app.main:app
  5. Gunicorn binds on 0.0.0.0:$PORT (Render injects $PORT); health check at /health
Quirk you'll see in Render logs

The entrypoint first tries localhost:5432 before falling through to the real DATABASE_URL. You'll see ~30 failed connection attempts in the first minute of every deploy. This is harmless — it's the local-dev fallback running before the Neon URL takes over.

Question 6

How does a mobile release actually happen?

Answer — manual, from your laptop, today
# Prep
cd vitaltrack-mobile
npx expo-doctor                              # sanity check

# Build AAB (production profile bakes in vitaltrack-api.onrender.com)
eas build --profile production --platform android

# Upload to Play Console internal track
eas submit --profile production --platform android

The production EAS profile sets channel: production and autoIncrement: true (bumps versionCode automatically). Submit target is track: internal.

Planned change

The CI file has a disabled build-production job ready to automate this. When Play Console production is active, flip if: false to if: github.ref == 'refs/heads/main' and wire the EAS robot token as a secret.

Question 7

Why does the same code produce different behavior in staging vs production?

Answer — env vars, not artifacts

Render runs the same Docker image in both environments. The differences are injected via env vars on the Render service:

VarStagingProduction
ENVIRONMENTstagingproduction
DATABASE_URLNeon branch vitaltrack_stagingNeon branch neondb
FRONTEND_URLstaging domaincarekosh.com
CORS_ORIGINSstaging domainproduction domain

12-factor

This is the orthodox 12-factor split: one immutable artifact, per-environment configuration. Data never crosses; a staging DB wipe touches zero production rows.

Question 8

Which backend does a given APK talk to?

Answer — whichever URL was baked in at build time

EXPO_PUBLIC_API_URL is a compile-time constant. Once an APK/AAB is built, its backend is fixed — changing a Render URL later won't redirect existing installs.

EAS profileArtifactPoints at
developmentAPKhttp://localhost:8000
previewAPKhttps://vitaltrack-api-staging.onrender.com
productionAAB (Play Store internal)https://vitaltrack-api.onrender.com
Sharp edge

If a teammate gets confused test data on "preview APK," the fix is a rebuild, not a server config change. Always verify which URL an APK is pointed at before blaming the backend.

Question 9

What does the backend actually serve for non-JSON requests?

Answer — a few HTML pages, rendered server-side

Most endpoints return JSON, but three return HTML because the user arrives via an email link:

Question 10

The first request after idle is slow. Why?

Answer — Render free-tier cold start

Free Render web services sleep after 15 min of inactivity. The first request wakes them — container start, Alembic check, Gunicorn boot — which can take 30–60 s.

Mitigation status:

  • Client-side UX (shipped, audit/cold-start-mutation-ux track): ambient StatusPill escalates from "Connecting…" to "Connecting… server warming up"; per-item "Updating…" indicators on rows with in-flight saves; cold-start pre-flight banner on Edit Item; MutationResultDialog centred summary for slow-success and connection-failure outcomes; tappable Retry button that re-fires the same mutation through Mutation.execute(); auto-clear of the cold-start flag on any successful 2xx so the pill doesn't get stuck. Cache persistence (PR #16) means cached data renders instantly on launch while the fresh fetch is still in flight.
  • UptimeRobot pinging /health every 14 min keeps the service warm — not configured today, but recommended.
  • Paid Render tier removes the sleep behaviour entirely — the most reliable fix once the project monetises.
Question 11

My PR says "2 commits behind main." Should I rebase?

Answer — usually no

GitHub counts the merge commits of other PRs against your branch. After every merge on main, every open PR goes up by 1. Rebase only if you have actual merge conflicts, not because of the count.

When to rebase anyway

(1) You want a recent fix from main that your work depends on. (2) Your branch is weeks old and ancestors matter for review. (3) CI is failing because of a main-side change that your branch doesn't include.
