Complete System Architecture

Three tiers — mobile, backend API, Postgres — joined by a single invariant: the server is the source of truth. Local state is UI-only; server state is TanStack Query.

Before — Offline-first (pre-PR #4)

  • Zustand (1,100 lines) + persist middleware
  • AsyncStorage holding inventory rows
  • services/sync.ts — 611 lines of push/pull/full
  • Rehydration race: stale local overwrote server → data loss
  • Merge conflicts on life-critical stock counts

After — Server-first (PRs #4–#8)

  • TanStack Query owns all server state
  • Zustand 61 lines — UI-only
  • sync.ts deleted; backend /sync/* unused
  • OCC version column + HTTP 409
  • Single writer wins; loser re-fetches and retries
Why server-first for medical inventory

Offline editing of life-critical stock counts creates merge conflicts with real-world consequences — two caregivers silently overwriting each other's updates can result in running out of oxygen. Server-first eliminates the whole conflict class; OCC handles the last remaining race.

Mobile cache persistence layer (PR #16)

Between the mobile app and the server, TanStack Query now keeps a read-only snapshot of inventory/order/category/activity data in AsyncStorage (@tanstack/react-query-persist-client + @tanstack/query-async-storage-persister, key carekosh-query-cache). On app launch, the last-seen data renders instantly while fresh data refetches in the background (staleTime: 30s, gcTime: 24h). Mutations bypass the cache and go direct to the server — the cache is never pushed back, so the data-corruption failure mode of the old offline-first architecture is structurally impossible. Cache is cleared on both logout and login for shared-device privacy; a schema buster tied to app.json's version auto-invalidates on upgrade; and a ENABLE_CACHE_PERSISTENCE kill switch in providers/QueryProvider.tsx disables the whole layer with one line. focusManager is wired to React Native's AppState so returning to the foreground triggers a refetch.

Diagram 02 / 05

Auth & Account Lifecycle

Six flows — register, login, refresh, password change, account delete, cancel delete. Every refresh rotates; every password change revokes all sessions; every deletion cascades.

Why two-step deletion

Google Play policy (and GDPR / DPDP Act) require full in-app account deletion. A single-click delete would be attractive for attackers with a stolen access token; a two-step flow that requires email access adds an out-of-band check. The raw token is emailed but never stored — the DB keeps only the SHA-256 hash.

Diagram 03 / 05

CI/CD & Deployment Pipeline

GitHub Actions on PR and push. Render deploy hook on merge. EAS builds on demand. Production AAB job exists but is disabled pending Play Store launch.

Key invariant

The Docker image that runs in staging and production is byte-identical. Every environmental difference is an env var injected by Render. This is what makes "staging works but production doesn't" a tractable question — the delta is always in config or data, never in code.

Data Model & Integrity

Seven tables, one aggregate root (users), a version column that makes OCC possible, a JSONB audit log that makes every write traceable, and CASCADE FKs that let an account-deletion touch only one row and unwind cleanly.

Mobile Screen Map

Expo Router's file-based navigation. Two groups — (auth) and (tabs) — plus four top-level modals. The Profile screen and its swipe-down popup menu shipped in PR #13.

Why file-based routing

Expo Router's file-to-route mapping is a constraint that doubles as documentation — the app/ tree literally is the navigation graph. A new contributor can grep the directory and know every screen the app has. This is worth more than the flexibility a code-based navigator would buy.
