Compile-time mobile baking

EXPO_PUBLIC_API_URL is set per EAS profile in eas.json. Once an APK or AAB is built, this URL is a compile-time constant — you cannot change which backend an installed build hits without rebuilding. A "preview APK hitting production" is not a toggle; it's a new binary.

3 · Data isolation

Three databases, three concerns, three operating rules.

EnvDatabaseWhat lives thereWhen to wipe
Dev Local Docker postgres:16 Whatever you created last session docker compose down -v — any time, no consequence
Staging Neon branch vitaltrack_staging QA test accounts, curl-created inventory, E2E dry runs Any time via Neon Console. Tell teammates first (they may have a test account).
Production Neon branch neondb Real user data — caregivers' medical inventories, orders, activity Never. Not even "just this one user." Use the account-deletion flow (PR #13).
There is no cross-environment replica

Staging does not replicate production. If a bug only repros with real-looking data, you create a synthetic analog in staging — you do not copy production rows. Copying production to staging would violate both data minimisation principles and likely the privacy policy you'll ship for the Play Store launch.

4 · Docker + deployment pipeline

docker-entrypoint.sh — the first thing Render runs

#!/bin/sh set -e 1. Parse DATABASE_URL into host/port via sed (handles postgresql:// and postgresql+asyncpg://) Note: query-param stripping (Neon's ?sslmode=require etc.) is NOT done here — that happens in app/core/config.py via the ensure_asyncpg_driver validator. The entrypoint only extracts host + port for the readiness check. 2. Wait for DB: for i in 1..30: pg_isready -h $DB_HOST -p $DB_PORT sleep 2 If both DATABASE_HOST and DATABASE_URL are unset (or sed parse fails), falls back to db:5432 / localhost:5432 and you'll see 30 failed attempts in the logs. With the Neon URL set in Render env vars (the normal case), the parse succeeds and the first attempt hits Neon directly. 3. alembic upgrade head Idempotent. Applies any new migrations; skips already-applied. 4. exec "$@" Which resolves to: gunicorn -k uvicorn.workers.UvicornWorker -w 4 \ --bind 0.0.0.0:$PORT app.main:app

5 · Known quirks & gotchas

Quirk 1 · localhost:5432 failures on Render

docker-entrypoint.sh's dev fallback tries localhost first. Render logs show ~30 failed attempts in the first minute of every deploy. This is not a bug — it's the script probing the dev path before using DATABASE_URL. Ignore.

Quirk 2 · "Your branch is 2 commits behind main"

Merge commits on main tick every open PR up by 1. Unless you have actual conflicts, this count is cosmetic. Rebase only when needed.

Quirk 3 · Render free-tier cold starts

15 min idle → sleep. First request after sleep: 30–60 s to wake. Mitigate with UptimeRobot pinging /health every 14 min, or upgrade tier.

Quirk 4 · EXPO_PUBLIC_API_URL is baked, not runtime

A "staging APK" and a "production AAB" are two separate binaries. You cannot toggle. Changing the Render URL in eas.json then shipping an app update requires a new build for each environment.

Quirk 5 · Both Render services auto-deploy from main

There is no "deploy to staging first, soak, then production" today. Both services watch main and get the same code at the same time. If you need time separation, that's Strategy C in the deployment strategy doc — needs a staging long-lived branch.

Quirk 6 · REQUIRE_EMAIL_VERIFICATION true on staging

Historical docs claimed staging had email verification off. It does not — both Render services set this to true. If registration-without-verification behaviour is what you want for quick testing, run locally (REQUIRE_EMAIL_VERIFICATION=False) instead of spinning up staging accounts.

Quirk 7 · Render shell migrations on a bad branch

Running alembic upgrade head manually on Render shell when the code is pre-migration will happily apply the migration, potentially leaving the DB ahead of the code. Always let docker-entrypoint.sh run migrations on deploy — don't trigger them by hand unless you're recovering from a failure.

Quirk 8 · Mobile cache persistence lives in AsyncStorage (PR #16)

TanStack Query's cache is persisted to AsyncStorage under key carekosh-query-cache by providers/QueryProvider.tsx. Two knobs matter operationally: (1) ENABLE_CACHE_PERSISTENCE is a top-of-file boolean kill switch — flip to false to restore pre-PR-#16 behavior without a rebuild. (2) The buster is bound to Constants.expoConfig.version — bumping app.json's version auto-invalidates every persisted cache on first launch after upgrade, so you don't ship stale shapes when response schemas change. Auth queries are excluded from dehydration via shouldDehydrateQuery; cache is cleared on both logout and login for shared-device privacy. Auth tokens continue to live in SecureStore — they are never persisted through TanStack Query.

6 · In one sentence

CareKosh is a 12-factor app: the same immutable Docker image runs in every environment; all environment-specific behaviour is controlled by env vars; configuration is validated at startup; data never crosses environments; deploys are triggered by git push and mediated by a deploy hook; rollbacks are a button in the Render UI.

What this buys

Deterministic behaviour. If staging works and production breaks, the delta is always one of: env vars, DB state, or email provider state — never "staging has a different build."
