Deployment Strategy

Four ways to ship a CareKosh change to staging and production. Ranked, with the trade-offs the ranking hides.

CareKosh · April 2026 · Current reference for PRs #4–#13

The four strategies, briefly

Every backend change ends up on Render. Every mobile change ends up on a phone. The only question is the path. There are four realistic paths. We picked one as the default; the rest have their moments.

StrategyWhat it meansDefault?
A — Branch-gated mergeFeature branch → PR → CI + preview APK on label → merge → Render auto-deploys both environmentsYes — default
B — Staging-only service per PREach PR gets its own Render preview environmentOnly if needed
C — Manual stepped rolloutMerge to staging branch first, verify, then merge staging → mainRarely
D — Direct push to mainNo PR, no review, push and prayNever, except hotfix

Ranked — with why, not just "recommended"

Rank 1 · Default

A. Branch-gated merge with label-based APK

One artifact, two environments, gates on automation not humans.
feature branch │ ▼ PR to main ──► CI (backend + frontend + security + pr-check) │ │ │ └── label: build-apk ──► EAS preview APK → sideload → staging backend │ ▼ merge main ──► CI re-runs ──► deploy-backend job ──► Render hook fires │ ├── vitaltrack-api (prod, neondb) └── vitaltrack-api-staging (staging branch) │ └── alembic upgrade head on both

Wins

  • One green merge = both environments updated, zero manual Render interaction
  • CI gates every change; no way to ship a type error or trivy critical
  • Preview APK on a PR label — reviewers exercise the real binary on a real device
  • Alembic runs before Gunicorn serves, so schema and code deploy atomically

Costs

  • Prod and staging always on the same commit — no "soak in staging for 48h" window
  • A bad merge takes both environments down until the fix lands
  • Render free-tier cold starts happen on both services independently
Why this is the default

For a two-person project with a disabled production AAB build, the operational cost of per-environment gating exceeds its benefit. The merge-to-main rule is enforceable via branch protection; everything else is review-by-human, which is the bottleneck we want.

Rank 2

B. Per-PR Render preview environment

Each PR gets its own isolated backend + DB branch.

Render supports preview environments where each PR spawns a short-lived service that's torn down on close. Combined with Neon branches (one per PR), every PR has its own end-to-end sandbox.

Wins

  • Reviewers can exercise the PR without sharing staging
  • Destructive migrations can be trialled without touching the shared staging DB
  • Cleanup is automatic on PR close

Costs

  • Requires Render + Neon plans that support branching previews (costs money)
  • Mobile APK still has EXPO_PUBLIC_API_URL baked in — needs either per-PR EAS builds or a runtime switcher
  • Overhead unused while there's one reviewer
Graduate to this when

Team size > 3, or a backend PR involves a destructive migration that shouldn't touch shared staging until reviewed.

Rank 3

C. Stepped rollout via staging branch

A long-lived staging branch that feeds main.

Feature branches merge into staging; staging periodically merges into main. Render staging service watches staging; production service watches main. Adds a "soak period" between staging and production.

Wins

  • True time separation — production lags staging by hours or days
  • Gives regression bugs a window to surface before real users see them
  • Cleaner hotfix story (branch off main, merge back into both)

Costs

  • Two branches to keep in sync — merge-debt accumulates fast on small teams
  • Requires changing both Render services' watched-branch config
  • Doesn't buy much over Strategy A until you have a test suite worth trusting (CareKosh's test suite is currently 40/53 failing)
Prerequisite

Strategy C's value comes from time in staging. Without automated regression tests running hourly, all you get is "there could be a bug and we might find it." Fix the test suite before adopting this.

Rank 4

D. Direct push to main

For hotfixes only. Even then, think twice.

Push commits straight to main, bypassing PR and review. CI still runs (it's a push trigger), so syntax errors are caught. Everything else is on you.

Wins

  • Fastest possible path to a live fix when production is on fire
  • No waiting for reviewer availability at 2 am

Costs

  • No second pair of eyes — mistakes compound
  • Rebuilds history of a protected branch (if branch protection is off)
  • Writes a bad norm — one direct push becomes two becomes ten
Allowed only if

(1) Production is down and users are affected, (2) the fix is <20 lines and surgical, (3) you open a retroactive PR within 24 h with a post-mortem.

Your ideal workflow — end-to-end

1. git checkout -b feature/something-useful 2. work, commit frequently 3. push branch 4. open PR → main (title in Conventional Commits format) 5. CI goes green (3–4 min) 6. if reviewer needs a binary: add label "build-apk" → APK sideloaded 7. reviewer approves 8. merge (squash or merge, your call — both work) 9. CI re-runs on main; deploy-backend POSTs Render hook 10. BOTH Render services redeploy; Alembic runs on both DBs 11. verify deploy: ├── curl https://vitaltrack-api-staging.onrender.com/health ├── curl https://vitaltrack-api.onrender.com/health ├── Render dashboard → Events tab → "Deploy live" └── alembic current (via Render shell) = latest migration filename 12. if change touches Play Store surface: eas build --profile production → eas submit --profile production (manual, for now) 13. smoke test on real device (Pixel 9 Pro XL · Android 16)
Backend-only changes

For PRs like #12 (auth hardening) and #13 (account deletion), Strategy A is essential: both Render services must deploy together because staging needs to receive the same email-verification + cascade-delete semantics before users test on it. Mobile changes are minimal but backend endpoints must be live on staging before the PR is considered "verified."

Rollback — the part people forget

Every strategy needs an exit. Here's what actually works on this stack.

ProblemFix
Bad deploy, production brokenRender dashboard → Deploys tab → "Rollback" on the last known-good deploy. ~1 min.
Bad migration committed + auto-appliedHard. Either alembic downgrade -1 from Render shell (if reversible), or write a forward-fix migration. Never edit an applied migration file.
Bad mobile release (APK in the wild)Ship a fix + rebuild + re-submit. Cannot "rollback" a Play Store release — users on the bad version need the update to land.
Secret leaked in commitRotate the secret first. Then rewrite history if feasible. Consider the secret compromised regardless.
Design for reversibility

Prefer additive migrations (ADD COLUMN NULL) over destructive ones (DROP COLUMN). Prefer feature flags over big-bang cutovers. PR #4 (server-first migration) was one big-bang PR — it worked, but only because sync.ts was already dead code and the risk was mobile-side, not data-side.
