01The big picture

CareKosh is a mobile app (React Native / Expo) talking to a Python backend (FastAPI) on a database (Neon PostgreSQL). Five independent services each own one job. The art is in how they connect.

Cloudflare — owns the domain & DNS, receives support mail Render — runs the backend container Neon — the PostgreSQL database Brevo — sends transactional email GitHub — source of truth + CI/CD

Figure 1 — The whole system. Four boxes across the top are a single user request; the lanes below are deployment and email.

02The custom domain — why we own carekosh.com

Before this work, the app called Render's own hostnames: vitaltrack-api.onrender.com (prod) and vitaltrack-api-staging.onrender.com (staging). That works, but it permanently chains us to one hosting provider: the URL is baked into every shipped app binary, so moving hosts later would force an app update for every user.

The fix is to call a name we own. We registered carekosh.com on Cloudflare Registrar (at-cost pricing, free DNS) and pointed two subdomains at Render:

EnvironmentPublic hostname (what the app calls)Points at (Render service)
Productionapi.carekosh.comvitaltrack-api.onrender.com
Stagingstaging-api.carekosh.comvitaltrack-api-staging.onrender.com

Now portability comes from three things we control: we own the domain, we control DNS, and the app calls our domain instead of a provider URL. If we ever leave Render, we change one CNAME — no app update required.

The exact steps we performed

  1. Registered the domain. Bought carekosh.com on Cloudflare Registrar (at-cost, free DNS, full control of the zone). See Figure 2.
  2. Added the custom domains in Render. For each service, opened Render → the service → Settings → Custom Domains and added api.carekosh.com to vitaltrack-api (production) and staging-api.carekosh.com to vitaltrack-api-staging (staging). Render then showed the CNAME target to point each name at.
  3. Created the CNAME records in Cloudflare. In Cloudflare → carekosh.com → DNS → Records, added two CNAMEs: api → vitaltrack-api.onrender.com and staging-api → vitaltrack-api-staging.onrender.com (left as "DNS only" while Render issued certificates).
  4. Let Render issue TLS. Render verified the DNS and automatically issued and installed TLS certificates for both custom domains, so HTTPS works on the carekosh names.
  5. Set FRONTEND_URL per environment (Render dashboard). Production = https://api.carekosh.com/api/v1/auth, staging = https://staging-api.carekosh.com/api/v1/auth. This is the base used inside email links (see §7). Value only — never type FRONTEND_URL= into the value box.
  6. Updated the codebase on branch config/use-carekosh-custom-domains so future builds and ops checks use the carekosh names — see the table below.
  7. Confirmed the GitHub Actions secrets the pipeline needs were present: EXPO_TOKEN and RENDER_DEPLOY_HOOK (see §4 / Figure 4).
  8. Verified. Ran dig to confirm api / staging-api resolve to Render, and ran the smoke checks against the new hostnames.

03DNS — how a name becomes a server

DNS is the phone book of the internet. When anything needs to reach api.carekosh.com, it asks DNS "what is this name?" and gets back where to go. We use three record types in this project:

  • CNAME — "this name is an alias for that name." api.carekosh.com is a CNAME to vitaltrack-api.onrender.com, so traffic ends up at Render.
  • TXT — free-text records used for proof and email policy (domain ownership, SPF, DMARC, DKIM keys).
  • MX — "mail for this domain goes to these servers." Ours point at Cloudflare so support mail can be received.

Because Cloudflare is both our registrar and DNS host, every record lives in one place: Cloudflare → carekosh.com → DNS. The full list, with the live values we verified, is in section 12.

Figure 3 — Name → address → server. Render uses the incoming Host: api.carekosh.com header to pick the right service and serve its TLS certificate.

04Render + GitHub — how code becomes a running backend

The backend is defined as code in render.yaml (Infrastructure as Code). It declares a Docker web service named vitaltrack-api, region Singapore, health check at /live, and a set of environment variables. Secrets are marked sync:false (set in the dashboard, never in git); SECRET_KEY is auto-generated.

services:
  - type: web
    name: vitaltrack-api
    runtime: docker
    branch: main
    healthCheckPath: /live
    envVars:
      - key: DATABASE_URL      # sync:false — set in dashboard (a secret)
      - key: MAIL_PASSWORD     # sync:false — Brevo API key (a secret)
      - key: MAIL_FROM         # sync:false — noreply@carekosh.com
      - key: FRONTEND_URL
        value: https://api.carekosh.com/api/v1/auth

How GitHub and Render connect

Render is linked to the GitHub repo. When main changes, Render automatically rebuilds and redeploys — no extra step. Our CI pipeline (.github/workflows/ci.yml) also calls a Render deploy hook as an explicit backup trigger. Neither decides which backend the app talks to; that is fixed at app build time (section 5).

The two GitHub secrets

The repository has exactly two Actions secrets, and that is correct and complete — they are the only two referenced anywhere in the workflow.

SecretUsed byWhat it doesRequired?
EXPO_TOKENEAS Build (Expo cloud)Lets GitHub authenticate to Expo non-interactively and queue an APK/AAB buildYes, for any CI build
RENDER_DEPLOY_HOOKRender APIA private URL that triggers a redeploy on push to mainNo — Render also auto-deploys; this is a backup

Backend runtime secrets (DATABASE_URL, MAIL_PASSWORD, SECRET_KEY) correctly live in Render, not GitHub. The two systems each hold only what they need.

05The mobile app — how it knows which backend to call

The app does not ask anyone at runtime which backend to use. The URL is baked into the binary at build time through the variable EXPO_PUBLIC_API_URL. Expo inlines that value into the JavaScript bundle, and services/api.ts reads it:

// services/api.ts
const API_BASE_URL = process.env.EXPO_PUBLIC_API_URL || 'http://localhost:8000';
const API_VERSION  = '/api/v1';   // every API call is `${API_BASE_URL}${API_VERSION}/...`

Which value gets baked depends on the EAS build profile in eas.json:

Build profileEXPO_PUBLIC_API_URL baked inResult
preview (APK)https://staging-api.carekosh.comTesters hit staging + the staging database
production (AAB)https://api.carekosh.comPlay Store users hit production
Critical consequence

Because the URL is baked in, a build pointed at the wrong backend can't be fixed without rebuilding. That's why app.config.js contains throw-guards: if a production build's URL isn't exactly the production domain, the build fails on purpose. After this change, the next production AAB must be rebuilt for users to hit api.carekosh.com.

06Request journey — what happens when a button is tapped

Putting sections 3–5 together, here is the life of a single tap — say, loading the inventory list:

  1. The user taps a button. The app builds a URL: https://api.carekosh.com/api/v1/items (base from the baked EXPO_PUBLIC_API_URL, plus /api/v1).
  2. The phone resolves api.carekosh.com via DNS → Cloudflare answers with the CNAME chain to Render.
  3. The request travels over HTTPS to Render. Render reads the Host: api.carekosh.com header, matches it to the vitaltrack-api custom domain, and terminates TLS with the certificate Render issued for that domain.
  4. FastAPI handles the route, checks the auth token, and queries Neon PostgreSQL (asyncpg, SSL via connect args).
  5. Neon returns rows → FastAPI serializes JSON → the response travels back the same path to the app, which renders the list.

Cross-origin requests are allowed by the backend's CORS_ORIGINS setting. Health is monitored at /live (process-only) and /health (database-backed).

07Email sending — Brevo & the noreply@ address

All transactional email (verification, password reset, deletion) is sent by the backend through Brevo's HTTP API. Brevo is a sending service only — it never receives mail for us.

# app/utils/email.py  (simplified)
url = "https://api.brevo.com/v3/smtp/email"
headers = {"api-key": settings.mail_password_value}      # the Brevo key (a Render secret)
payload = {
    "sender":  {"name": "CareKosh", "email": settings.MAIL_FROM},   # noreply@carekosh.com
    "to":      [{"email": to_email, "name": to_name}],
    "subject": subject,
    "htmlContent": html_content,                           # contains the FRONTEND_URL link
}

The problem we started with: emails came from a Brevo domain

Before authentication, Brevo had to send from its own shared domain, so the from: showed ...@brevosend.com with our Gmail as reply-to. Unbranded, and weaker for deliverability.

Before — unauthenticated
from:CareKosh <you@xxxxxxxx.brevosend.com>
reply-to:CareKosh <you@example.com>
signed by:xxxxxxxx.brevosend.com

The exact steps we performed

  1. Started domain authentication in Brevo. Brevo → Senders, Domains & Dedicated IPs → Domains → Authenticate a domain, entered carekosh.com, and chose "Authenticate the domain automatically (Recommended)" so Brevo writes the records straight into Cloudflare.
  2. Authorized the DNS records (one-time). Brevo connected to Cloudflare and added four records: a verification brevo-code TXT, two DKIM CNAMEs (brevo1._domainkey, brevo2._domainkey), and a DMARC TXT. DKIM is the key one — it lets Brevo cryptographically sign as carekosh.com. (Note: Brevo authenticates via DKIM and did not add an SPF record — important later for §8.)
  3. Waited for propagation and re-verified. Brevo's checker briefly showed some records as "mismatch / not detected" — that is normal DNS lag, not an error. We confirmed the values were live in DNS and clicked verify again until all went green.
  4. Domain showed Authenticated. Once verified, sending from any @carekosh.com address was unlocked.
  5. Set the sender in Render. Set MAIL_FROM=noreply@carekosh.com (value only, no MAIL_FROM= prefix) and redeployed. The code already reads settings.MAIL_FROM and uses the sender name "CareKosh", so no code change was needed.
  6. Tested. Triggered a forgot-password — the email now arrives from noreply@carekosh.com, DKIM-signed by carekosh.com, with no reply-to leaking a personal address (see below). Still to confirm: set the same MAIL_FROM on the production service and re-test, since the first test ran on staging.
After — authenticated
from:CareKosh <noreply@carekosh.com>
signed by:carekosh.com
reply-to:(none — no personal address leaks)
mailed-by:gy.d.sender-sib.com (Brevo's plumbing — normal, harmless)

08Email receiving — Cloudflare & the support@ address

Brevo cannot give us an inbox. To receive mail at support@carekosh.com we use Cloudflare Email Routing, which is forwarding, not a mailbox: it catches mail for the address and forwards it to an existing Gmail. There is no new login — support mail simply arrives in the normal Gmail inbox.

The exact steps we performed

  1. Opened Email Routing. Cloudflare → carekosh.com → Email → Email Routing. It started Disabled with DNS "Not configured" — expected at this point.
  2. Added and verified the destination. Destination Addresses → Add, entered the Gmail, then clicked the verification link Cloudflare emailed to it. (A rule pointing at an unverified address won't deliver.)
  3. Created the routing rule. Routing rules → Create routing rule: pattern support @ carekosh.com, action Send to an email, destination the verified Gmail → Save.
  4. Ignored the "Connect" button. The blue Connect button opens Email Workers (programmatic processing) — it is not how routing is enabled. We closed it.
  5. Enabled routing by adding the MX records. Via Settings → DNS records (the "DNS records: Not configured" link). Cloudflare added the three route1/2/3.mx.cloudflare.net MX records, an SPF TXT, and its own routing DKIM.
  6. Confirmed Enabled and locked the records. The overview flipped to Routing status: Enabled / DNS Configured. We then "Locked" the records so they can't be edited or deleted by accident.
  7. Tested receiving. Emailed support@carekosh.com from a different account; it forwarded to Gmail and the Activity Log showed Forwarded.
Why no SPF conflict

Cloudflare added the only SPF record (v=spf1 include:_spf.mx.cloudflare.net ~all). Because Brevo authenticates via DKIM (it added no SPF of its own), there's nothing to merge — and Brevo's outgoing mail still passes DMARC on DKIM alignment. The two DKIM setups (Brevo's for sending, Cloudflare's for routing) coexist cleanly.

Two addresses, two jobs

noreply@carekosh.com = outgoing, via Brevo. support@carekosh.com = incoming, via Cloudflare → Gmail. They share only the domain name. The app never sends from support@; nothing ever receives at noreply@ (replies to it simply bounce, which is exactly how a no-reply address should behave). There is also no separate "CareKosh inbox" to log into — support mail lands in your normal Gmail.

09Flow: a user emails support@carekosh.com

Figure 12 — Receiving is forwarding. Brevo, Render and our backend are not involved at all.

  1. Someone composes an email to support@carekosh.com and hits send.
  2. Their mail server asks DNS for the MX records of carekosh.com and gets back Cloudflare's three mail servers.
  3. The message is delivered to Cloudflare Email Routing.
  4. Cloudflare matches the active rule for support@ and forwards the message to the configured Gmail.
  5. It lands in the normal Gmail inbox. The Activity Log records it as "Forwarded."
Gmail quirk

If you test by sending from the same Gmail it forwards to, Gmail may hide the looped-back copy as a duplicate. Test from a different account, or trust the Activity Log's "Forwarded" status.

10Flow: forgot password, start to finish

Figure 13 — Forgot-password is a round trip: app → backend → Brevo → user, then user → backend to actually change the password.

  1. In the app, the user requests a reset. The app calls POST https://api.carekosh.com/api/v1/auth/forgot-password with their email.
  2. The backend finds the active user, generates a random token, stores only its SHA-256 hash with a 1-hour expiry (the raw token is never stored), and returns a generic success either way to prevent account-enumeration. Rate limited to 3/hour.
  3. It calls Brevo to send the reset email. The link is {FRONTEND_URL}/reset-password?token=… → https://api.carekosh.com/api/v1/auth/reset-password?token=…. The email arrives from noreply@carekosh.com, DKIM-signed.
  4. The user taps the link. The backend serves an HTML reset form (this is why FRONTEND_URL points at the backend, not a separate website).
  5. The user submits a new password. The backend re-hashes the submitted token, checks it matches and hasn't expired, enforces password strength, stores the new hash, revokes all refresh tokens (other devices are logged out), and sends a "password changed" notice. Rate limited to 5/hour.

11Flow: registration & email verification

Figure 14 — Registration intentionally does not log the user in; access is gated until the email is verified.

  1. The user registers. The app calls POST /api/v1/auth/register. The backend hashes the password (Argon2, bcrypt fallback), creates the user with is_email_verified = false, generates a verification token (stored as a SHA-256 hash, 24-hour expiry), and sends the verification email via Brevo. The link is {FRONTEND_URL}/verify-email?token=….
  2. The app deliberately does not log the user in — it routes to a "verify your email" screen.
  3. The user opens the email (from noreply@carekosh.com) and taps Verify. The backend hashes the token, checks the match and expiry, sets is_email_verified = true, clears the token, and returns an HTML success page.
  4. The user returns to the app and logs in. With REQUIRE_EMAIL_VERIFICATION on, login is blocked until verified; once verified, login succeeds and access + refresh tokens are issued.

On SPF & DKIM

13How to verify everything yourself

Trust, but verify. The dashboard can lag; public DNS is the ground truth. These read-only commands confirm the live state:

# Sending side — Brevo auth
dig +short CNAME brevo1._domainkey.carekosh.com @1.1.1.1
dig +short TXT   _dmarc.carekosh.com            @1.1.1.1

# Receiving side — Cloudflare routing
dig +short MX  carekosh.com  @1.1.1.1        # expect route1/2/3.mx.cloudflare.net
dig +short TXT carekosh.com  @1.1.1.1        # expect the v=spf1 … line

# Routing side — app domains
dig +short api.carekosh.com         @1.1.1.1 # expect the Render chain
dig +short staging-api.carekosh.com @1.1.1.1

End-to-end manual tests:

  • Sending: trigger a forgot-password and confirm the email reads from: noreply@carekosh.com and signed by: carekosh.com.
  • Receiving: email support@carekosh.com from a different account and confirm it lands in Gmail (and shows "Forwarded" in Cloudflare's Activity Log).
Still to confirm

Set MAIL_FROM=noreply@carekosh.com on the production Render service (not only staging) and run one production forgot-password test, so production mail is branded too.

14Glossary

TermPlain meaning
DNSThe internet's phone book — turns names into server addresses.
CNAMEAn alias: "this name is really that name." Used to point our subdomains at Render.
MX"Mail for this domain goes here." Ours point at Cloudflare.
